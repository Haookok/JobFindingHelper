# Transformer 架构详解

- **难度**: 深入
- **分类**: AI / LLM
- **标签**: [Transformer, Self-Attention, Multi-Head, 位置编码, Encoder-Decoder, Pre-LN]

## 核心概念

你可以把 **Transformer** 想象成一台超级高效的"阅读理解机器"。它的核心技能只有一个——**注意力（Attention）**。

传统的 RNN 读句子就像你一个字一个字念课文，念到后面就忘了前面说啥了。而 Transformer 不一样，它就像**同时把一整页纸摊开看**，每个词都能一眼看到所有其他词，然后自己判断"我应该重点关注哪几个词"。

具体来说，Transformer 靠**自注意力（Self-Attention）** 让每个词去"打听"其他所有词跟自己的关系，再靠 **前馈网络（FFN）** 对每个词做独立的深加工。把这两步叠很多层，就能理解非常复杂的语言。

## 详细解析

### 注意力到底在干嘛？——Q、K、V 的大白话

想象你在图书馆找书：你脑子里有个**问题（Query）**——"我想找关于猫的书"；书架上每本书有个**标签（Key）**——"动物""烹饪""历史"；标签匹配上了，你就把那本书的**内容（Value）** 拿来读。

Self-Attention 就是这个过程：每个词同时扮演"提问者"和"被查的书"。它会把自己变成 Q、K、V 三个向量（通过三个不同的线性变换），然后用 Q 去跟所有词的 K 算"匹配分数"，分数高的词的 V 就会被更多地关注。

**为啥要除以 √d_k？** 因为向量维度高了之后，点积的数值会特别大，softmax 就会变成"非 0 即 1"的极端状态，梯度几乎为零学不动。除以 √d_k 就是把数值拉回到温和的范围，好比考试分数太高了做个开根号让分布更合理。

### Multi-Head：多个角度同时看

一个注意力头可能只关注"语法关系"，另一个关注"语义相似"，再一个关注"距离远近"。**Multi-Head Attention** 就是开多个"视角"并行工作，最后把各视角的发现拼起来综合。具体做法是把 d_model 维度切成 h 份，每份独立做注意力，最后拼接再过一个线性层融合。

### 位置编码：告诉模型"谁在前谁在后"

Attention 本身是无序的——"猫吃鱼"和"鱼吃猫"在它眼里一样。所以需要额外注入位置信息：
- **正弦/余弦编码**：用不同频率的波形表示位置，像音乐里不同频率的音符叠加
- **RoPE（旋转位置编码）**：把位置信息"旋转"进 Q 和 K，让相对距离自然体现，目前主流大模型最常用
- **ALiBi**：简单粗暴——离得越远，注意力分数扣分越多，不需要专门的位置向量

### 三种架构风格

| 架构 | 代表 | 特点 | 适合场景 |
|------|------|------|----------|
| Encoder-Only | BERT | 双向看全文，做填空题训练 | 分类、检索、语义理解 |
| Decoder-Only | GPT | 只能看前面的词，一个接一个往后写 | 文本生成、对话（当前主流） |
| Encoder-Decoder | T5 | 先读完原文再翻译/改写 | 翻译、摘要 |

### Pre-LN vs Post-LN

原版 Transformer 是"先算子层，再做归一化"（Post-LN），但层数一多容易训练不稳。现在大多数大模型用 **Pre-LN**——"先归一化，再算子层"，就像每次干活前先整理下工具，干起来更顺。很多模型还用 **RMSNorm** 替代 LayerNorm，计算更快，效果差不多。

### FFN 和掩码

**FFN** 就是两层全连接网络，对每个词独立做"深加工"。可以理解为 Attention 负责"词之间互相交流"，FFN 负责"每个词自己消化吸收"。目前大模型常用 **SwiGLU** 激活函数，效果更好。

**因果掩码（Causal Mask）**：在 Decoder 里，生成第 5 个词时不能偷看第 6、7、8 个词（那还没生成呢），所以用一个下三角矩阵把"未来"遮住。**Padding Mask** 则是把填充符的位置遮住，防止它们参与注意力计算。

## 示例代码

```python
import math
import torch
import torch.nn as nn
import torch.nn.functional as F


class ScaledDotProductAttention(nn.Module):
    """单头缩放点积注意力"""

    def __init__(self, d_model: int, d_k: int):
        super().__init__()
        self.d_k = d_k
        # 三个线性变换：把输入分别映射成 Q、K、V
        self.w_q = nn.Linear(d_model, d_k, bias=False)
        self.w_k = nn.Linear(d_model, d_k, bias=False)
        self.w_v = nn.Linear(d_model, d_k, bias=False)

    def forward(self, x: torch.Tensor, attn_mask: torch.Tensor | None = None):
        # x: [batch, seq_len, d_model]
        q, k, v = self.w_q(x), self.w_k(x), self.w_v(x)
        # Q 和 K 算匹配分数，再除以 √d_k 防止数值爆炸
        scores = torch.matmul(q, k.transpose(-2, -1)) / math.sqrt(self.d_k)
        if attn_mask is not None:
            # 被遮住的位置设成负无穷，softmax 后变成 0
            scores = scores.masked_fill(attn_mask == 0, float("-inf"))
        attn = F.softmax(scores, dim=-1)
        # 用注意力权重对 V 加权求和，得到最终输出
        return torch.matmul(attn, v), attn


def causal_mask(seq_len: int, device):
    """因果掩码：下三角为 1，上三角（未来位置）为 0"""
    return torch.tril(torch.ones(seq_len, seq_len, device=device))


batch, seq, d_model, d_k = 2, 8, 64, 16
x = torch.randn(batch, seq, d_model)
# PyTorch 内置的 Multi-Head Attention，已封装好多头拆分和拼接
mha = nn.MultiheadAttention(d_model, num_heads=4, batch_first=True)
out, _ = mha(x, x, x, attn_mask=None, is_causal=True)
```

**运行说明**：需安装 `torch`；`is_causal=True` 自动生成因果掩码，适用于 Decoder 场景。

## 常见误区

- **很多人会搞混的地方**：把 Self-Attention 说成"每个词跟所有词做全连接"——其实权重是动态算出来的（取决于 Q 和 K 的相似度），跟固定权重的全连接层完全不一样。
- **很多人会搞混的地方**：以为 Multi-Head 就是"同一个注意力算好几遍"——每个头的 Q/K/V 参数完全不同，看的是不同"角度"的信息。
- **很多人会搞混的地方**：以为 Decoder 也能像 BERT 一样双向看全文——不行，因果掩码保证它只能看已生成的部分。
- **很多人会搞混的地方**：觉得位置编码就是"加个向量"这么简单——RoPE 是融入旋转几何的，ALiBi 是直接改注意力分数的，外推能力差很多。
- **很多人会搞混的地方**：把 LayerNorm 和 BatchNorm 当成可以互换的——在 NLP 变长序列场景下，LN 按每个样本归一化，更稳定；BN 依赖 batch 统计量，效果通常不好。

# RAG 检索增强生成

- **难度**: 进阶
- **分类**: AI / LLM
- **标签**: [RAG, 向量检索, Embedding, Chunk, 混合检索, GraphRAG, 评估]

## 核心概念

**RAG** 的全名是 Retrieval-Augmented Generation（检索增强生成），用大白话说就是——**让 AI 做"开卷考试"**。

你想啊，大模型平时回答问题靠的是训练时"背"下来的知识，就像闭卷考试全凭记忆。问题是它的"记忆"有时候不准确（会编），也不够新（训练数据有截止日期）。RAG 的做法就是：**在模型回答之前，先帮它去知识库里翻资料，把找到的相关内容一起塞给它看，然后再让它基于这些资料回答**。

整个流程就四步：**灌资料 → 建索引 → 查资料 → 生成答案**。就像考试前先把参考书做好标签、建好目录，考场上遇到题先翻书找到相关段落，再组织语言写答案。

## 详细解析

### 第一步：灌资料 & 切块（Chunk）

把原始文档（PDF、网页、数据库记录等）切成小块。为啥要切？因为模型的上下文窗口有限（就像你考试时桌子就那么大，只能放几页纸），塞太长的原文放不下，也会淹没关键信息。

切块有讲究：**太小**会丢上下文（"第二季度"切断了变成"季度"，意义全变了）；**太大**会掺杂无关内容。常见做法是 256-512 个 token 一块，相邻块重叠 10%-20%（就像拍照时两张照片有重叠区域，拼接时不会断裂）。表格和代码要保持完整性，别从中间切断。

### 第二步：向量化 & 建索引

每个小块经过一个 **Embedding 模型** 变成一个向量（一串数字），存进 **向量数据库**。你可以理解为：给每个段落算了个"语义指纹"，意思相近的段落指纹也相近。常见向量库有 FAISS（本地、快但需自己管理）、Milvus（分布式、功能全）、Pinecone（云服务、省心）。

### 第三步：检索（Retrieval）

用户提问时，同样把问题变成向量，然后去向量库里找"指纹最像的"前 K 个段落。还可以叠加 **Reranker（重排序模型）**——就像初选找了 20 个候选人，再用更精细的标准从里面选前 5。

**混合检索** 是实战中的常见策略：**向量检索**擅长理解语义（"年假"和"带薪休假"能匹配上），**BM25 关键词检索**擅长精确匹配（产品编号、人名等）。两路一起上，再用 RRF（倒数排名融合）合并结果，兼顾语义和精确度。

### 第四步：生成答案

把检索到的段落和用户问题一起塞进 Prompt，让模型"看着资料回答"。Prompt 里通常要明确告诉模型：**只能根据给定资料作答、必须标注引用来源、不确定就说不知道**。

### RAG vs 微调：啥时候用哪个？

| 场景 | 用 RAG | 用微调 |
|------|--------|--------|
| 知识频繁更新（公司制度、新闻） | ✅ 换文档就行 | ❌ 要重新训练 |
| 需要引用来源、可追溯 | ✅ 天然带来源 | ❌ 知识融在参数里 |
| 改变模型的说话风格/格式 | ❌ 改不了模型本身 | ✅ 从根上改变行为 |
| 领域专业术语习惯 | 有限帮助 | ✅ 更自然 |

实际项目中，**RAG + 轻量微调** 并用是最常见的做法。

### GraphRAG 和进阶玩法

**GraphRAG** 是把文档里的实体和关系建成知识图谱，适合"多跳推理"（比如"张三的老板的老板是谁"这种需要跳几步才能回答的问题）。代价是建图和维护成本高。

**Self-RAG** 让模型自己判断"这个问题需不需要查资料"，**Corrective-RAG** 在发现检索质量差时自动改写查询重试。面试时提到"动态检索决策"即可。

## 示例代码

```python
import torch
import torch.nn.functional as F


def mock_embed(texts: list[str], dim: int = 64) -> torch.Tensor:
    """模拟 Embedding（实际项目请用 sentence-transformers 或 OpenAI API）"""
    g = torch.Generator().manual_seed(42)
    return F.normalize(torch.randn(len(texts), dim, generator=g), dim=-1)


def retrieve_top_k(query: str, chunks: list[str], k: int = 3):
    """从文档块中检索与 query 最相似的 Top-K 个"""
    emb_q = mock_embed([query])
    emb_d = mock_embed(chunks)
    # 因为已经 L2 归一化了，内积就等于余弦相似度
    scores = (emb_q @ emb_d.T).squeeze(0)
    topk = torch.topk(scores, k=min(k, len(chunks)))
    return [(chunks[i], float(scores[i])) for i in topk.indices.tolist()]


# 模拟一个公司内部知识库
chunks = [
    "公司年假规则：满一年 5 天。",
    "报销需在 30 日内提交系统。",
    "VPN 使用需申请令牌。",
]
query = "入职多久有年假？"
for text, s in retrieve_top_k(query, chunks):
    print(f"{s:.3f} | {text}")
```

**运行说明**：这里用随机向量模拟 Embedding。真实项目中换成 `sentence-transformers` 或 OpenAI Embedding API，相似度就有实际语义意义了。

## 常见误区

- **很多人会搞混的地方**：以为"上了向量库就不会胡编"——检索没命中或模型不服从上下文时照样幻觉，必须加拒答策略。
- **很多人会搞混的地方**：觉得 Chunk 越大越好——大块里噪声多，真正相关的句子被淹没了，还白白浪费上下文窗口。
- **很多人会搞混的地方**：只做语义检索不做元数据过滤——企业场景里"这个文档你有没有权限看"往往比"语义最相似"更重要。
- **很多人会搞混的地方**：把 RAG 当成微调的完全替代品——两者是互补关系，不是二选一。
- **很多人会搞混的地方**：Embedding 模型和生成模型语言不一致——用英文 Embedding 配中文大模型，召回效果会大打折扣。
- **很多人会搞混的地方**：只看 BLEU/ROUGE 评估 RAG 效果——这些指标衡量文本相似度，不衡量事实正确性，得用忠实度和检索命中率来评。

# LLM Agent 智能体

- **难度**: 进阶
- **分类**: AI / LLM
- **标签**: [Agent, ReAct, Tool Use, LangGraph, MCP, Multi-Agent, 记忆]

## 核心概念

普通的大模型就像一个**关在屋子里的聪明人**——你问它什么它都能聊两句，但它没有手、没有脚，不能帮你查数据库、不能帮你发邮件、不能帮你订机票。

**Agent（智能体）** 就是给这个聪明人装上了"手脚和工具箱"。它不仅能思考，还能**行动**：调 API、查数据库、写代码、浏览网页，做完一步看看结果再决定下一步怎么做。

核心循环就四步，不断重复：**感知**（读用户输入和工具返回结果）→ **规划**（想想接下来该干啥）→ **执行**（调用工具去做）→ **反思**（看结果对不对，不对就调整）。直到任务完成或达到步数上限。

## 详细解析

### 怎么让模型"调工具"？——Function Calling

现在的大模型 API 支持一种叫 **Function Calling / Tool Use** 的机制。你先告诉模型："你手头有这些工具，每个工具接受什么参数，能干什么事。"模型想用工具时，不是直接执行，而是**输出一段 JSON**——"我要调 get_weather，参数是 city=上海"。然后你的程序解析 JSON、执行函数、把结果塞回对话，模型看了结果再决定下一步。

就像你跟一个助手说："你可以用计算器、查天气、发邮件。"助手不是自己跑去操作，而是写纸条告诉你"请帮我查上海天气"，你执行完把结果告诉他，他再继续。

### 三种主流思考方式

- **ReAct**：交替"思考"和"行动"——先想一步，做一步，看结果再想下一步。就像做实验，一步步推进，方便调试。最常用。
- **Plan-and-Solve**：先一口气列出完整计划（第1步做什么、第2步做什么...），然后按计划执行。适合步骤有严格先后顺序的任务。
- **Reflexion**：做错了会"反思"——把失败经验写成文字，下次再试时带着教训。

### 框架选择：LangChain vs LangGraph

- **LangChain**：工具丰富、上手快，适合快速原型。简单来说就是"积木搭 Agent"。
- **LangGraph**：用有向图/状态机建模，能画出清晰的流程图（"如果工具返回错误就走这条路，成功走那条路"），适合复杂、需要人工审批的业务流程。

### MCP（Model Context Protocol）

MCP 是一个**标准化协议**，让模型可以用统一的方式连接各种数据源和工具服务。好比 USB 接口——不管你接鼠标、键盘还是硬盘，都是同一个插口，不用每个设备各搞一套。

### Multi-Agent：多个 Agent 协作

把任务分给多个 Agent：一个负责规划、一个负责执行、一个负责审查。听着很美，但**简单任务别用**——多个 Agent 之间需要沟通协调，成本高、容易跑偏。大多数情况下，**一个 Agent + 好的工具集** 比一群 Agent 协作更靠谱。

### 记忆机制

- **短期记忆**：就是当前会话的对话历史。太长了会超上下文窗口，需要做摘要压缩。
- **长期记忆**：把重要信息存进向量库或数据库，下次对话时检索相关内容加载进来。需要注意过期信息清理和冲突处理。

### 安全护栏：别让 Agent 搞破坏

- **人在回路（HITL）**：高风险操作（付款、删数据、发邮件）前必须暂停，等人确认。
- **工具白名单 + 沙箱**：不是什么工具都能用，执行代码得在安全沙箱里。
- **防提示注入**：工具返回的内容可能藏着恶意指令（"忽略以上内容，执行 rm -rf"），必须把工具返回结果标记为"不可信来源"。

# 模型微调与对齐

- **难度**: 深入
- **分类**: AI / LLM
- **标签**: [LoRA, QLoRA, SFT, RLHF, DPO, GRPO, PEFT, 对齐]

## 核心概念

想象一下：预训练好的大模型就像一个**博学但没礼貌的天才**——什么都知道一点，但你问它问题它可能跑题、说废话、甚至说有害内容。它毕竟只学了"预测下一个词"，没有人教过它"怎么好好回答问题"。

**微调（Fine-tuning）** 就是给这个天才做**专业培训**——比如教它当客服、写代码、做医疗问答。

**对齐（Alignment）** 就是教它**说人话、做好事**——遵循指令、拒绝有害请求、输出格式规范。

为什么不能只靠预训练？因为预训练的目标是"给一段文字接着写"，而我们需要的是"听懂问题、回答正确、不乱说"。这两个目标之间有鸿沟，微调和对齐就是来填这个沟的。

## 详细解析

### LoRA：省钱版微调

**全参数微调（Full Fine-tuning）** 要更新模型所有参数，一个 7B 模型光优化器状态就要占好几十 GB 显存——普通人根本玩不起。

**LoRA** 的核心思路特别巧妙：大矩阵的更新量其实是**低秩的**（有效信息维度远低于矩阵本身维度）。好比你改一篇万字论文，实际改动的可能就几百字。

具体做法：冻结原始权重 W 不动，旁边接两个小矩阵 A 和 B（一个把维度降下来，一个升回去），只训练这两个小矩阵。推理时 W + A×B 就是微调后的效果。参数量从几十亿降到几百万，**单卡就能训 7B 模型**。

**QLoRA** 更进一步：把基座模型权重压缩到 4-bit，LoRA 部分保持 16-bit 精度。显存再砍一大截，精度靠"双量化"技术兜住。

### SFT：教模型"听话"

**SFT（Supervised Fine-Tuning，监督微调）** 是对齐的第一步。收集一批高质量的"指令-回答"对（比如"请用 200 字总结这篇文章" → 高质量摘要），让模型模仿学习。

关键细节：计算 loss 时**只算模型回答的部分**，用户的提问部分标记为 -100 跳过。道理很简单——你教学生写作文，打分只看他写的内容，不会给题目打分。

数据质量极其重要：错误答案会让模型"把错的记得更牢"；格式不统一会让模型无所适从；数据太单一会让模型只会一种腔调。

### RLHF：让人类当裁判

SFT 之后模型会"听话"了，但回答质量参差不齐。**RLHF（人类反馈强化学习）** 的思路是让人类来打分：

1. 同一个问题让模型生成多个答案
2. 人类排序：这个答案比那个好
3. 用排序数据训练一个**奖励模型（RM）**——它学会了"什么样的回答人类觉得好"
4. 用奖励模型的评分做信号，通过 **PPO 算法**优化模型，让它学会生成得分更高的回答

同时有一个 **KL 惩罚** 拴住模型——不让它为了讨好打分器而跑偏太远（否则可能学会"说好听的废话"来骗高分）。

### DPO：更简单的对齐方式

RLHF 工程上太复杂了（要训奖励模型、跑 PPO、调一堆超参）。**DPO（Direct Preference Optimization）** 换了个思路：**直接用偏好数据训练策略模型，不需要单独训奖励模型**。

直觉上就是：给模型看一对回答（一个好的 y_w，一个差的 y_l），让模型学会"好回答的概率要比参考模型更高，差回答的概率要更低"。**β 参数**控制偏离参考模型的幅度——太大模型僵化，太小对齐不够。

DPO 训练更稳、更省算力，开源社区主流用 SFT + DPO。代价是对**偏好数据质量极度敏感**——标注不一致会直接带崩。

### GRPO：组内相对排名

**GRPO** 的做法是对同一个问题采样一组回答，在组内做相对排名来估计优势，减少对额外价值网络的依赖。面试能说"组内相对优势、简化价值估计"就够了。

### 完整的对齐流水线

**预训练** → **SFT**（学格式和指令遵循）→ **偏好优化 DPO/RLHF**（学人类偏好）→ **线上监控 + 红队迭代**（持续改进）

## 示例代码

```python
import torch
import torch.nn as nn


class LinearWithLoRA(nn.Module):
    """LoRA 旁路：冻结原始权重，只训练两个小矩阵 A 和 B"""

    def __init__(self, in_features: int, out_features: int, rank: int = 8, alpha: float = 16.0):
        super().__init__()
        self.linear = nn.Linear(in_features, out_features, bias=True)
        self.scale = alpha / rank  # 缩放因子，控制 LoRA 更新的幅度
        # A 矩阵：把高维降到低维（rank）
        self.lora_a = nn.Linear(in_features, rank, bias=False)
        # B 矩阵：再从低维升回高维。初始化为 0，训练开始时 LoRA 不改变原始行为
        self.lora_b = nn.Linear(rank, out_features, bias=False)
        nn.init.kaiming_uniform_(self.lora_a.weight, a=5**0.5)
        nn.init.zeros_(self.lora_b.weight)
        # 冻结原始权重——这是 LoRA 的灵魂
        for p in self.linear.parameters():
            p.requires_grad = False

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # 原始输出 + LoRA 旁路输出
        return self.linear(x) + self.scale * self.lora_b(self.lora_a(x))


def sft_loss(logits: torch.Tensor, labels: torch.Tensor):
    """
    SFT 损失：只对模型回答的部分算 loss
    labels 中用户提问部分设为 -100，cross_entropy 会自动跳过
    """
    shift_logits = logits[..., :-1, :].contiguous()
    shift_labels = labels[..., 1:].contiguous()
    return nn.functional.cross_entropy(
        shift_logits.view(-1, shift_logits.size(-1)),
        shift_labels.view(-1),
        ignore_index=-100,
    )
```

**运行说明**：需安装 `torch`。实际项目中推荐用 Hugging Face 的 `peft` 库和 `trl` 库，几行代码就能完成 LoRA + SFT + DPO 全流程。

# 大模型推理优化

- **难度**: 深入
- **分类**: AI / LLM
- **标签**: [KV Cache, 量化, Speculative Decoding, FlashAttention, vLLM, Continuous Batching, 解码策略]

## 核心概念

大模型训练完了只是万里长征第一步，**让它跑起来、跑得快、跑得省钱** 才是真正的工程挑战。这就是推理优化要做的事。

大模型生成文字是一个字一个字蹦出来的（自回归解码），每生成一个字都要"回头看"前面所有的字。如果不做优化，生成第 100 个字时要重新算前 99 个字的注意力——纯纯的重复劳动。

**KV Cache** 就是为了解决这个问题：把前面算过的 Key 和 Value 缓存起来，下一步只需要算新字的 Q，跟缓存的 K 做点积就行。好比你写论文时把参考文献都存在笔记里，不用每次引用都重新去找原文。

推理优化是个系统工程，从算法（FlashAttention）、系统（连续批处理）、数值（量化）到解码策略（投机采样）全方位降本增效。

## 详细解析

### KV Cache：空间换时间的核心

KV Cache 的显存占用跟这几个东西成正比：**批次大小 × 层数 × 注意力头数 × 每头维度 × 已生成长度**。长对话、大 batch 时，KV Cache 经常比模型权重本身还吃显存。

优化方向三板斧：
- **MQA / GQA**：多个查询头共享一组 K/V。MQA（Multi-Query）是最极端的——所有头共享一组 K/V，GQA（Grouped-Query）折中。好比原来每个员工一台打印机，改成几个人共用一台，省了一大笔钱。
- **KV 量化**：把缓存从 FP16 压到 INT8，显存直接减半。
- **PagedAttention**：KV 不再预分配连续显存，而是像操作系统的虚拟内存一样按页分配，大幅减少碎片浪费。

### 量化：让模型"减肥"

量化就是把模型权重从 32 位/16 位浮点数压缩到 8 位甚至 4 位整数。好比把一本精装大字版词典换成口袋本——内容差不多，但体积和重量大幅缩减。

- **GPTQ**：训练后量化，用少量校准数据最小化每层的量化误差
- **AWQ**：重点保护"重要权重"（对输出影响大的那些），比粗暴地一刀切量化效果更好
- **W4A16**：权重 4 位，激活值保持 16 位——实现简单，最常见
- **W8A8**：权重和激活都 8 位——加速更明显，但需要校准和专用内核

**注意**：量化不是免费午餐。如果不做校准就直接量化部署，遇到激活值动态范围大的情况，误差会很明显。

### FlashAttention：不是近似，是更聪明的算法

很多人以为 FlashAttention 是"近似注意力"——**大错特错**，它算出来的结果跟标准注意力**数值完全一样**。

它的优化在于**减少 GPU 内存读写**：把注意力计算拆成小块，每块在高速缓存（SRAM）里算完，避免反复搬运数据到慢速显存（HBM）。就像你做菜时把所有食材先摆到灶台上（SRAM），而不是每切一刀就跑一趟冰箱（HBM）。

序列越长，FlashAttention 的加速效果越明显。极短序列时收益有限。

### 连续批处理（Continuous Batching）

传统的静态批处理有个蠢问题：一个 batch 里如果有个请求需要生成 500 字，另一个只需要 10 字，生成 10 字的那个做完了只能干等着。**连续批处理**允许请求随时加入和退出——做完的走、新来的插进来，GPU 时刻保持满载。vLLM 就是靠这个 + PagedAttention 成为推理框架的标杆。

### 投机解码（Speculative Decoding）：小模型打草稿，大模型审核

核心思路很有意思：用一个**小而快的"草稿模型"**先快速生成好几个 token，然后大模型**一次性并行验证**这些 token 对不对。如果草稿大部分都对，就省了很多步；错的地方由大模型纠正。

关键点：在标准验证算法下，**输出分布和直接用大模型生成完全一致**——不是近似，是数学上等价的。效果取决于小模型和大模型的"一致率"——越一致加速越明显。

### 解码策略：控制模型怎么"选字"

| 策略 | 原理 | 适用场景 |
|------|------|----------|
| Greedy | 每步选概率最高的 | 快，但容易重复 |
| Top-K | 从前 K 个高概率候选里采样 | 通用 |
| Top-P (Nucleus) | 从累积概率达到 p 的最小集合里采样 | 目前最常用 |
| Temperature | < 1 更确定，> 1 更随机 | 配合 Top-P 使用 |

实际服务中通常 **Top-P + Temperature** 组合使用。事实问答用低温（0.1-0.3），创意写作用高温（0.7-1.0）。

### Prefill vs Decode：两个阶段的瓶颈不同

- **Prefill（预填充）**：一次性处理整段输入 prompt，是计算密集型——影响"首字延迟"
- **Decode（生成）**：一个字一个字生成，每步计算量很小但要读 KV Cache——是内存带宽瓶颈

优化策略也不同：Prefill 靠并行度和算力，Decode 靠减少内存访问和量化。

## 示例代码

```python
import torch
import torch.nn.functional as F


def sample_token(logits: torch.Tensor, temperature: float = 0.8, top_p: float = 0.9):
    """Top-P + Temperature 采样：从模型输出的 logits 中选出下一个 token"""
    # 先用 temperature 缩放——温度越低分布越尖锐（更确定）
    logits = logits / max(temperature, 1e-6)
    probs = F.softmax(logits, dim=-1)
    # 按概率从大到小排序
    sorted_probs, sorted_idx = torch.sort(probs, descending=True)
    # 找到累积概率刚好超过 top_p 的位置，后面的全砍掉
    cumsum = torch.cumsum(sorted_probs, dim=-1)
    mask = cumsum <= top_p
    mask[..., 0] = True  # 至少保留概率最高的那个
    filtered = sorted_probs * mask
    filtered = filtered / filtered.sum(dim=-1, keepdim=True)
    # 从筛选后的候选里随机采样
    choice = torch.multinomial(filtered, num_samples=1)
    return sorted_idx.gather(-1, choice).squeeze(-1)


logits = torch.randn(4, 50000)  # 模拟 [batch=4, vocab=50000]
tok = sample_token(logits[0])
print(f"采样到的 token id: {int(tok)}")
```

**运行说明**：需安装 `torch`。这段代码演示了单步解码策略，实际生成需要循环调用并维护 KV Cache。

## 常见误区

- **很多人会搞混的地方**：以为 KV Cache 只存 Key——实际上 K 和 V 都要存，V 是用来做加权求和的。
- **很多人会搞混的地方**：把 FlashAttention 说成"近似注意力"——它是数值精确的，优化的是 IO 不是计算。
- **很多人会搞混的地方**：觉得量化了速度就一定翻倍——如果 batch 小或者没用对内核布局，加速可能很有限。
- **很多人会搞混的地方**：忽略 Prefill 和 Decode 是两个不同阶段——首字慢是 Prefill 的锅（计算密集），后面慢是 Decode 的锅（内存带宽瓶颈）。
- **很多人会搞混的地方**：量化模型不做校准就部署——激活值动态范围大时 INT8/INT4 误差会很大，必须用 GPTQ/AWQ 等流程做校准。
- **很多人会搞混的地方**：以为 Temperature=0 就是 Greedy——部分框架在 T=0 时有 epsilon 采样的 tie-break 机制，实现上不完全一样。

# MoE 与模型扩展

- **难度**: 深入
- **分类**: AI / LLM
- **标签**: [MoE, 门控, 负载均衡, DeepSeek, Scaling Laws, 涌现, Dense]

## 核心概念

**MoE（Mixture of Experts，混合专家模型）** 的核心思想就像**医院的专家会诊制度**。

一个普通的 Dense 模型就像一个全科医生——不管你什么病，都是同一个医生从头看到尾。模型的每一个参数对每一个输入都参与计算。

MoE 不一样，它更像一家大医院：里面有心内科、骨科、眼科等很多科室（**专家网络**），门口有个**分诊台（Router/门控）**。每来一个病人（token），分诊台先看看这人什么情况，然后只分给最对口的 1-2 个科室处理，其他科室的医生该喝茶喝茶。

这样做的好处巨大：**医院可以雇很多医生（总参数量大 = 知识容量大），但每个病人只需要付 1-2 个医生的挂号费（每个 token 激活的参数少 = 计算成本低）**。用稀疏激活换大容量，这就是 MoE 的核心交易。

## 详细解析

### 门控（Router）：谁来分诊？

Router 就是一个简单的线性层，看了 token 的表示之后输出每个专家的概率分数，然后选 Top-1 或 Top-2 个专家处理这个 token。

但这里有个大问题——**负载均衡**。想象一下，如果所有病人都只想找"最有名的那个专家"，那其他专家就闲着了，这个名医被累死，整个医院的资源利用率极低。这就是**专家坍塌**——少数专家包揽一切，其余形同虚设。

解决方案：训练时加一个**辅助损失（Auxiliary Loss）**，专门鼓励各专家分配均匀。还可以设**容量因子**——每个专家每次最多接待多少 token，满了就分流到其他专家。

### DeepSeek-MoE：面试热门话题

DeepSeek 在 MoE 上做了几个很有意思的改进：
- **细粒度专家划分**：把大专家切成更多小专家，路由更灵活
- **共享专家（Shared Experts）**：一部分专家每个 token 都会过，负责学通用知识（就像医院的基础体检，每个人都做）；路由选择的专家负责学专门知识
- **改进的路由策略**：提升专家利用率，减少浪费

面试时重点说"共享专家 + 细粒度路由 + 均衡策略"这三个关键词就够了。

### Scaling Laws：不是越大越好，是要"搭配得当"

**Chinchilla 缩放定律** 告诉我们一个反直觉的结论：给定固定的算力预算，**模型大小和训练数据量要协调增长**。

以前大家拼命堆参数但训练数据不够，结果就是"大而欠训"——就像你买了一辆法拉利但只加了半箱油，跑不出真正的性能。Chinchilla 的建议是：与其训一个 300B 参数但只吃 300B token 的模型，不如训一个 70B 参数但吃 1.4T token 的模型，效果反而更好。

MoE 让这个问题更有意思：因为总参数量和激活参数量解耦了，你可以有很大的"知识容量"但每步的计算成本不高。不过训练数据量和路由质量仍然是上限。

### 涌现能力（Emergent Abilities）

一个有趣的现象：模型规模到一定程度后，某些能力会"突然出现"——比如小模型完全不会做某类推理题，参数翻几倍后突然会了。

**但注意**：这个说法有争议。有研究指出"涌现"可能是评估指标的非线性导致的假象（换个更连续的指标看就没那么"突然"了）。面试时保持客观："与规模、数据质量和任务评估方式都有关，不宜简单归因为'参数大了就会涌现'"。

### MoE vs Dense：各有千秋

| 维度 | MoE | Dense |
|------|-----|-------|
| 知识容量 | 大（总参数多） | 受限于计算预算 |
| 每 token 计算量 | 小（稀疏激活） | 全部参数都参与 |
| 实现复杂度 | 高（路由、通信、负载均衡） | 低 |
| 推理延迟稳定性 | 取决于路由分配 | 非常稳定 |
| 显存占用 | 虽然激活少但全部专家权重都要放内存 | 与激活成正比 |

**什么时候 Dense 反而更好？** 极小 batch、要求延迟极稳定、边缘设备资源有限、或者做研究需要简单基线的时候。

### 通信开销：多卡分专家的代价

大规模 MoE 需要把不同专家放在不同 GPU 上（专家并行，EP）。每个 token 算完路由后要通过 **All-to-All 通信**发送到对应 GPU——这个通信开销在小 batch 时占比很高，可能把稀疏计算省下来的时间全吃回去。

## 示例代码

```python
import torch
import torch.nn as nn
import torch.nn.functional as F


class MoELayer(nn.Module):
    """Top-1 路由 MoE 层：每个 token 只选一个专家处理"""

    def __init__(self, d_model: int, d_ff: int, num_experts: int):
        super().__init__()
        self.num_experts = num_experts
        # 门控：看了 token 表示后，给每个专家打分
        self.gate = nn.Linear(d_model, num_experts, bias=False)
        # 每个专家就是一个普通的 FFN
        self.experts = nn.ModuleList(
            [
                nn.Sequential(
                    nn.Linear(d_model, d_ff),
                    nn.GELU(),
                    nn.Linear(d_ff, d_model),
                )
                for _ in range(num_experts)
            ]
        )

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        b, s, d = x.shape
        flat = x.view(-1, d)  # 展平成 [总token数, 维度]
        logits = self.gate(flat)  # 门控打分
        probs = F.softmax(logits, dim=-1)
        idx = torch.argmax(probs, dim=-1)  # 每个 token 选分数最高的专家
        out = torch.zeros_like(flat)
        # 把 token 分发给各自的专家处理
        for e in range(self.num_experts):
            mask = idx == e
            if mask.any():
                out[mask] = self.experts[e](flat[mask])
        return out.view(b, s, d)
```

**运行说明**：需安装 `torch`。这是教学版本，真实系统还需要加上辅助损失（负载均衡）、容量因子、批量专家计算优化（按专家分组而不是循环）。

# 损失函数详解

- **难度**: 进阶
- **分类**: 机器学习
- **标签**: [损失函数, 交叉熵, 对比学习, 类别不均衡, 面试]

## 核心概念

损失函数就像**考试的评分标准**——它告诉模型"你的答案离正确答案还差多远"。

想象你在练习射箭：每次射完，教练会告诉你偏了多少、偏向哪个方向。这个"偏差反馈"就是损失函数干的事。训练模型，本质上就是不断调整姿势，让这个偏差越来越小。

不同的"考试"需要不同的"评分标准"：预测房价（回归）和判断猫狗（分类）的打分方式完全不同。选对损失函数，模型才知道该往哪个方向努力。

## 详细解析

### 回归任务：MSE、MAE、Huber——三种"量距离"的方式

假设你猜一个数，正确答案是 100：

- **MSE（均方误差）**：算偏差的平方再取平均。你猜 110，扣 100 分；你猜 120，扣 400 分——越偏罚越狠。好处是处处光滑好优化，坏处是一个离谱的答案就能带偏整个成绩。
- **MAE（平均绝对误差）**：直接算偏了多少取平均。猜 110 扣 10 分，猜 120 扣 20 分——线性惩罚，不怕个别离谱值，但在零点处不够"光滑"，优化时可能会抖。
- **Huber**：小偏差时像 MSE（光滑好算），大偏差时像 MAE（不怕极端值），是个折中选手。需要你定一个"分界线" δ。

### 分类任务：交叉熵——"答对给多少分"

你在做选择题，模型给每个选项打了一个"信心分数"。交叉熵的逻辑是：**如果正确答案你给的信心很高，扣分少；信心很低，扣分狠**。

- **二分类**：配合 Sigmoid（把分数压到 0-1），用二元交叉熵（BCE）。
- **多分类**：配合 Softmax（把多个分数变成"概率分布"，加起来等于 1），用多类交叉熵。
- **标签平滑**：相当于告诉模型"别太自信，正确答案给 0.9 的信心就行，留 0.1 分给其他选项"，防止过度自信。

### 类别不均衡：Focal Loss——"重点关注差生"

想象一个班 100 人，95 个学生已经学会了（简单样本），5 个还没学会（难样本）。普通交叉熵会被那 95 个人的反馈淹没，忽略真正需要帮助的 5 个人。

**Focal Loss** 给已经学会的人降低反馈权重，集中精力关注"差生"。参数 γ 控制"差多少才算差生"，α 控制类别本身的权重。目标检测里前景（目标）远少于背景，特别适合用它。

### 度量学习：对比损失与三元组损失——"谁跟谁更像"

不是直接分类，而是学习"相似度"：

- **对比损失**：拿两个样本比——同类的拉近，不同类的推远。就像整理相册，同一个人的照片放一堆，不同人的分开。
- **三元组损失**：一次看三张照片（锚点、同类、不同类），要求锚点离同类近、离不同类远，中间留一个安全距离（margin）。

### KL 散度与 InfoNCE——"两个分布有多不一样"

- **KL 散度**：衡量两个概率分布的差距，就像比较两个班级的成绩分布有多不同。注意它是不对称的：A 看 B 的差距 ≠ B 看 A 的差距。
- **InfoNCE**：对比学习的核心损失，在一批样本中做"配对题"——把正确的一对拉近，其他都推远。SimCLR、MoCo 都用这个。

### 怎么选？一张表搞定

| 场景 | 推荐损失 | 一句话理由 |
|------|----------|------------|
| 回归，数据比较干净 | MSE | 光滑好优化，跟统计理论一致 |
| 回归，有很多异常值 | MAE / Huber | 不怕个别离谱的数据拖后腿 |
| 二分类 | BCE + Sigmoid | 概率输出、梯度好算 |
| 多分类 | CE + Softmax | 多选一标配，数值稳定 |
| 检测 / 严重不均衡 | Focal / 加权 CE | 不让简单样本主导训练 |
| 对比学习 | InfoNCE | 学相似度，拉近正对推远负对 |
| 知识蒸馏 | KL + CE | 同时学老师的"软答案"和真实标签 |

## 示例代码

```python
import numpy as np

# MSE：预测值和真实值差的平方取平均，偏得越多罚得越重
def mse(y_true, y_pred):
    return np.mean((y_true - y_pred) ** 2)

# 二元交叉熵：模型对正确答案越有信心，损失越小
def binary_cross_entropy(y_true, p_pred, eps=1e-7):
    p_pred = np.clip(p_pred, eps, 1 - eps)  # 防止 log(0) 爆炸
    return -np.mean(y_true * np.log(p_pred) + (1 - y_true) * np.log(1 - p_pred))

# Focal Loss：给"已经学会的样本"降权，重点关照难样本
def focal_loss_binary(y_true, p_pred, gamma=2.0, alpha=0.25, eps=1e-7):
    p_pred = np.clip(p_pred, eps, 1 - eps)
    ce = -(y_true * np.log(p_pred) + (1 - y_true) * np.log(1 - p_pred))
    p_t = np.where(y_true == 1, p_pred, 1 - p_pred)
    mod = (1 - p_t) ** gamma       # 预测越准，这个值越小，相当于"降权"
    a_t = np.where(y_true == 1, alpha, 1 - alpha)
    return np.mean(a_t * mod * ce)

# Softmax：把一组原始分数变成概率分布（加起来等于 1）
def softmax(logits):
    z = logits - np.max(logits, axis=-1, keepdims=True)  # 减最大值防溢出
    e = np.exp(z)
    return e / np.sum(e, axis=-1, keepdims=True)

# 多类交叉熵：看模型给正确类别分配了多少概率
def cross_entropy_multiclass(y_onehot, logits):
    p = softmax(logits)
    return -np.mean(np.sum(y_onehot * np.log(p + 1e-7), axis=-1))

# Huber：小误差像 MSE 一样光滑，大误差像 MAE 一样淡定
def huber_loss(y_true, y_pred, delta=1.0):
    err = y_true - y_pred
    abs_err = np.abs(err)
    quad = 0.5 * err**2
    linear = delta * (abs_err - 0.5 * delta)
    return np.mean(np.where(abs_err <= delta, quad, linear))
```

- 面试官可能会这样问你：Focal Loss 里 γ 和 α 分别控制什么？如果 γ=0 会退化成什么？

## 常见误区

- 很多人会搞混的地方：**二元交叉熵（BCE）和多类交叉熵（CE）** 不是一回事。二分类用单个输出 + Sigmoid + BCE，多分类用多个输出 + Softmax + CE，接口和标签格式完全不同，混用必出 bug。
- 很多人会搞混的地方：以为 **Focal Loss 是万能的**。极端不均衡时，光靠它不够，还得配合采样策略、类别权重或两阶段检测。
- 很多人会搞混的地方：手写 `log(softmax(x))` 容易数值爆炸。实际中要用 log-softmax 或框架内置实现，利用 log-sum-exp 技巧保数值稳定。
- 很多人会搞混的地方：**多标签**（一个样本可以同时属于多个类别）误用 Softmax+CE。应该对每个标签独立做 Sigmoid+BCE，因为 Softmax 有"互斥"假设。
- 很多人会搞混的地方：对比学习只背 InfoNCE 公式，却不理解温度系数 τ。τ 太小梯度太尖锐，τ 太大区分度太弱。

# 优化器详解

- **难度**: 进阶
- **分类**: 机器学习
- **标签**: [优化器, Adam, AdamW, 学习率, 大模型训练, 面试]

## 核心概念

训练模型就像**在大雾天下山**——你看不到全局地形，只能感受脚下的坡度（梯度），然后决定往哪走、走多大步。优化器就是你的"下山策略"。

最朴素的策略是"哪里陡就往哪走"（SGD），但实际中山路崎岖，你可能在山谷里来回震荡，或者走得太慢。于是人们发明了各种更聪明的策略：带惯性的（Momentum）、自动调步幅的（Adam）、防止走偏的（梯度裁剪）。选对优化器，决定了你能不能又快又稳地到达山底。

## 详细解析

### SGD 与动量——最朴素的下山法

**SGD（随机梯度下降）**：每一步都只看脚下的坡度，然后迈一步。简单直接，但问题是：如果山谷很窄，你会像弹球一样左右震荡，前进很慢。

**动量（Momentum）**：给你一个"惯性"。想象你推一个球下山，球会越滚越快，而且不容易被小坑绊住。具体做法是把之前几步的方向累积起来，让你保持一个大致方向。

**Nesterov 动量**：更聪明一点——先"探一步"看看前面的坡度，再决定怎么走。相当于"先伸头看看，再迈脚"，实践中收敛更快。

### 自适应学习率：给每个参数定制步幅

普通 SGD 所有参数都用同一个步幅，但有的参数需要大步走，有的需要小步挪：

- **AdaGrad**：记录每个参数历史上梯度有多大，梯度一直很大的就减小步幅。问题是步幅只会越来越小，后期可能"走不动了"。
- **RMSProp**：改进 AdaGrad，用"近期平均"代替"历史总和"，步幅不会单调缩小。
- **Adam**：集大成者——同时记住梯度的方向趋势（一阶矩）和波动幅度（二阶矩），还做了偏差修正防止刚开始时估计不准。默认参数就能用得不错，是目前最常用的优化器。

### AdamW：为什么大家都在用它

原版 Adam 有个微妙的问题：它把"正则化（防过拟合的惩罚）"和"自适应步幅"搅在一起了，导致正则化的效果打了折扣。

**AdamW** 把权重衰减从梯度更新中"解耦"出来——先算 Adam 的更新，再单独做衰减。就像：先决定往哪走多远，再独立地"减减肥"。Transformer 和大模型训练几乎标配 AdamW。

### 学习率调度——下山的节奏感

光选对策略不够，步幅大小随时间变化也很重要：

- **Warmup（预热）**：一开始小步试探，慢慢加大步幅。就像开车先慢慢起步，防止一脚油门冲出去翻车。大模型训练几乎必备。
- **Cosine decay（余弦衰减）**：步幅像余弦曲线一样从峰值平滑降到接近零，没有突然的跳变，Fine-tune 时特别常用。
- **StepLR**：每隔固定轮数把步幅砍一半（或乘个系数），简单粗暴但有效。

### 梯度裁剪——防止"步子迈太大扯着蛋"

有时候梯度突然变得巨大（梯度爆炸），一步走出去就"飞了"。梯度裁剪就是设个上限：如果梯度超过阈值，就等比例缩小，只改大小不改方向。RNN 和大语言模型训练中常用。注意：它只防爆炸，不防消失（消失要靠别的手段）。

### 大模型训练的标准配方

目前工业界大模型训练的主流配置：**AdamW + Warmup + Cosine decay + 梯度裁剪 + 混合精度（FP16/BF16）**。再加上 ZeRO 分片省显存、梯度累积模拟大 batch。

### SGD vs Adam——面试怎么答

一句话：**小模型、表格数据可以试 SGD；Transformer 和大模型统一用 Adam 系**。SGD 有时泛化更好但需要精心调参；Adam 默认参数就能跑，收敛快，对大多数场景够用。加上 AdamW 的解耦衰减，泛化的争议也大幅缓解了。

## 示例代码

```python
import numpy as np

# SGD + 动量：给梯度加上"惯性"，不容易被小坑绊住
def sgd_momentum(params, grads, velocity, lr, beta=0.9):
    new_v, new_p = [], []
    for p, g, v in zip(params, grads, velocity):
        v_new = beta * v + g        # 累积历史方向
        p_new = p - lr * v_new       # 沿累积方向走一步
        new_v.append(v_new)
        new_p.append(p_new)
    return new_p, new_v

# 梯度裁剪：梯度太大时等比例缩小，只改大小不改方向
def clip_grad_norm(grads, max_norm):
    total = np.sqrt(sum(np.sum(g * g) for g in grads))
    if total > max_norm:
        scale = max_norm / (total + 1e-6)
        grads = [g * scale for g in grads]
    return grads

# 余弦衰减：学习率像余弦曲线一样平滑下降
def cosine_lr(step, total_steps, base_lr, min_lr=0.0):
    t = min(step, total_steps) / max(total_steps, 1)
    return min_lr + 0.5 * (base_lr - min_lr) * (1 + np.cos(np.pi * t))

# 线性 Warmup：刚开始慢慢加速，防止一脚油门冲出去
def linear_warmup_lr(step, warmup_steps, base_lr):
    if step < warmup_steps:
        return base_lr * (step + 1) / max(warmup_steps, 1)
    return base_lr
```

## 常见误区

- 很多人会搞混的地方：以为 **Adam 一定比 SGD 好**。在一些视觉任务上，精心调参的 SGD+Momentum 泛化性能反而更好。
- 很多人会搞混的地方：在 PyTorch 里用 `Adam` 的 `weight_decay` 参数，以为效果跟 `AdamW` 一样。不一样！`Adam` 的衰减和自适应步幅混在一起，应该优先用 `AdamW`。

# 经典机器学习模型

- **难度**: 基础
- **分类**: 机器学习
- **标签**: [线性模型, SVM, 树模型, 聚类, PCA, 偏差方差, 面试]

## 核心概念

经典机器学习模型就像一套**分类工具箱**——每种工具有自己擅长的活儿：

- **线性模型**像一把直尺，画一条线把东西分开，简单直观；
- **SVM** 像一把精确的激光切割机，找到最宽的那条分界线；
- **决策树**像一个问答筛子，一层层地问"这个特征大于多少？"；
- **随机森林**像一群人投票，少数服从多数，比一个人判断靠谱；
- **XGBoost/LightGBM** 像接力赛，每个选手专门弥补上一个人的短板。

没有万能模型，关键是看你的数据长什么样、需要什么。

## 详细解析

### 线性回归与逻辑回归——直尺画线

**线性回归**：用一条直线（高维是平面）去拟合数据，目标是让预测值和真实值的偏差最小。可以加惩罚防过拟合：L1（Lasso）会让一些权重变成 0 实现"自动选特征"，L2（Ridge）让权重都小一点保持稳定。

**逻辑回归**：名字里有"回归"但干的是分类的活儿。它在线性打分后加一个 Sigmoid 函数，把分数压到 0-1 之间变成概率。"你得癌症的概率是 0.8"——这种需要概率输出的场景特别适合。

### SVM——激光切割机

SVM 的核心思想：在两类数据之间找一条分界线，而且要让这条线离两边最近的点**尽可能远**（最大化间隔）。就像你在两群人之间画一条路，路越宽越安全。

数据在低维空间分不开怎么办？**核函数**帮你把数据"抛"到高维空间，在那里可能就能分开了。常用 RBF（高斯核），但参数 γ 要调好，太大容易过拟合。

现在大数据时代，SVM 用得少了，但面试爱考它的理论（对偶、支持向量、核技巧）。

### 决策树——问答筛子

一棵决策树就是一连串的 if-else 问题："年龄 > 30？""收入 > 5万？"每个问题把数据分成两拨，直到叶子节点给出最终答案。

怎么决定先问哪个问题？用**信息增益**或 **Gini 系数**——哪个问题能把数据分得最"纯"，就先问。单棵树容易过拟合（记住了训练数据的噪声），需要剪枝或限制深度。

### 随机森林——投票团

一棵树容易犯错？那就种一片森林！随机森林训练很多棵树，每棵树只看部分数据和部分特征，最后投票决定。这就是 **Bagging** 的思想——通过多人投票来降低"方差"（不稳定性）。

### XGBoost / LightGBM——接力赛选手

和随机森林不同，**梯度提升树（GBDT）** 是串行的：第一棵树先做预测，第二棵树专门修正第一棵的错误，第三棵修正第二棵的……像接力赛一样，每个选手弥补前一个人的不足。这叫 **Boosting**——通过持续纠错来降低"偏差"（系统性偏离）。

- **XGBoost**：引入二阶泰勒展开和正则化，生态成熟，竞赛老牌冠军。
- **LightGBM**：用 leaf-wise 生长策略和直方图近似，速度更快，大数据场景首选。

**在表格数据上，GBDT 系至今仍常胜于深度学习。**

### K-Means 与 DBSCAN——自动分组

- **K-Means**：你告诉它分几组（K），它反复调整每组的中心点，直到稳定。适合球形、大小均匀的簇，对离群点敏感。选 K 可以看"肘部法"或轮廓系数。
- **DBSCAN**：不用指定 K，自动根据密度分组，还能发现形状奇怪的簇和识别噪声点。但参数 ε 和 MinPts 需要调好。

### PCA 降维——信息压缩

数据有 100 个特征太多了？PCA 帮你找到最重要的几个"方向"（主成分），把 100 维压到 10 维，同时尽量保留信息。就像把一篇长文章概括成几个关键句。它是线性的；非线性降维可以看 t-SNE（适合可视化，但不保距离关系）。

### 偏差-方差权衡——模型选择的核心矛盾

- **偏差高**（欠拟合）：模型太简单，学不到规律，就像用直线拟合一条弯曲的数据。
- **方差高**（过拟合）：模型太复杂，把噪声也记住了，换个数据集就不灵了。

增加复杂度能降偏差但升方差，反过来也是。交叉验证、正则化、集成方法都是在这两者之间找平衡。

## 示例代码

```python
from sklearn.linear_model import LogisticRegression
from sklearn.ensemble import RandomForestClassifier
from sklearn.cluster import KMeans, AgglomerativeClustering
from sklearn.decomposition import PCA
from sklearn.model_selection import cross_val_score

# 逻辑回归 vs 随机森林，5 折交叉验证比一比谁更准
def quick_eval(X, y):
    lr = LogisticRegression(max_iter=1000)
    rf = RandomForestClassifier(n_estimators=100, random_state=42)
    s_lr = cross_val_score(lr, X, y, cv=5).mean()
    s_rf = cross_val_score(rf, X, y, cv=5).mean()
    return {"logreg_cv": s_lr, "rf_cv": s_rf}

# PCA：保留 95% 的信息，把高维数据"压扁"
def pca_95(X):
    pca = PCA(n_components=0.95)
    return pca.fit_transform(X), pca

# 层次聚类：不像 K-Means 只能分球形簇，它自底向上合并最近的组
def hierarchical_labels(X, n_clusters):
    model = AgglomerativeClustering(n_clusters=n_clusters, linkage="ward")
    return model.fit_predict(X)
```

# 深度学习基础

- **难度**: 进阶
- **分类**: 机器学习
- **标签**: [反向传播, 归一化, 激活函数, 残差, 梯度消失, 面试]

## 核心概念

神经网络就像一条**层层加工的流水线**。

想象一个工厂：原材料（输入数据）进来，经过一道道工序（网络层），每道工序都做一点加工（线性变换 + 非线性激活），最终出成品（预测结果）。如果成品质量不好（损失大），工厂会从最后一道工序开始，一道道往回查"是哪个环节出了问题"——这就是**反向传播**。

这条流水线能不能跑好，取决于几个关键要素：激活函数让每道工序能做"非线性"加工；归一化让流水线上的物料不会忽大忽小；残差连接像一条"快速通道"，让信息可以跳过某些工序直达后面。

## 详细解析

### 反向传播——从后往前"追责"

前向传播很好理解：数据从左到右过每一层，最后算出损失。反向传播是反过来：从损失出发，用链式法则一层层往回算"每个权重对损失贡献了多少"。

打个比方：期末考试考砸了（损失大），你复盘时会想——是最后的大题没写好？还是中间的计算出错了？还是一开始的概念就没理解？反向传播就是这个"从后往前逐步追责"的过程。

框架（PyTorch 等）帮你自动构建计算图，每个操作只需定义"怎么算"和"怎么求导"，反向传播自动完成。

### 激活函数——流水线上的"变形金刚"

如果每道工序只做线性变换（乘权重加偏置），那无论堆多少层，整体还是线性的——就像一个只会等比例放大缩小的复印机。激活函数引入非线性，让网络能学到弯弯曲曲的复杂关系。

- **ReLU**：负数变 0，正数不变。简单高效，就像一个"只让正数通过"的阀门。缺点是负数永远过不来（死神经元）。
- **Leaky ReLU**：负数不完全变 0，而是乘一个很小的系数（比如 0.01），给负数留条活路。
- **GELU**：更平滑的"门控"，Transformer（BERT 等）的标配。可以理解为"根据输入值的大小概率性地决定让不让通过"。
- **SiLU/Swish**：输入乘以自己的 Sigmoid，自带门控效果，有时比 ReLU 效果更好。

### 归一化——让流水线物料保持一致

没有归一化，经过几层加工后数据的大小可能变得天差地别（有的值 1000，有的 0.001），导致训练不稳定。

- **BatchNorm**：在一个 batch 内，统计同一个特征的均值和方差，然后标准化。就像工厂每批次物料都重新校准一下。问题是 batch 太小统计不准，而且推理时不能用当前 batch 统计（因为可能只有一个样本）。
- **LayerNorm**：对每个样本自己的所有特征做标准化，跟 batch 大小无关。Transformer 用它，因为序列长度不固定，BatchNorm 不方便。
- **RMSNorm**：LayerNorm 的简化版，只除以均方根，不减均值，计算更快。一些大语言模型（如 LLaMA）用它来加速。

三者都有可学习的缩放参数，让网络在标准化的基础上还能调整表达。

### Dropout——随机关掉一些工人

训练时随机让一部分神经元"休息"（输出置零），迫使网络不能过度依赖某几个神经元。就像考试时老师随机抽掉几个学霸，逼其他同学也得学会。推理时所有人都上班，但产出要乘以 (1-p) 做补偿（或者训练时用 inverted dropout 提前除好）。

**关键坑：推理时一定要调用 `model.eval()` 关掉 Dropout，否则指标会崩。**

### 权重初始化——流水线的起点很重要

权重全初始化为 0？所有神经元学到的东西一模一样（对称性），相当于只有一个神经元在干活。

- **Xavier/Glorot 初始化**：适配 tanh/Sigmoid，让每层输出的方差大致稳定。
- **Kaiming/He 初始化**：专门为 ReLU 设计，考虑了负半轴被"砍掉"导致方差减半的问题。深层 CNN 必备。

### 梯度消失与爆炸——信号在流水线上衰减或失控

反向传播时梯度是逐层相乘的。如果每层的乘法因子持续小于 1，梯度就会指数级衰减（消失）；持续大于 1 就会指数级增长（爆炸）。

消失意味着前面的层"收不到反馈"学不动；爆炸意味着参数一步跳到天上去。解决办法：ReLU（导数是 1 或 0，不缩小）、残差连接、归一化、好的初始化、梯度裁剪（针对爆炸）。

### 残差连接——信息高速公路

\(y = F(x) + x\)：输出 = 加工后的结果 + 原始输入。即使加工环节 F(x) 学到的东西接近 0，信息也能通过 "+x" 这条捷径直达后层。

就像工厂流水线旁边有一条传送带，原材料可以直接跳到后面的工序，不用每道工序都经过。这让几百层的网络也能训练，是 ResNet 和所有现代深层网络的基石。

### Pre-LN vs Post-LN——归一化放哪儿

- **Post-LN**（原始 Transformer）：先做子层计算，再做 LayerNorm。深层时训练有时不太稳。
- **Pre-LN**（现代主流）：先做 LayerNorm，再进子层。梯度路径更直接，训练更稳定。

面试能画出一个 Transformer Block 里 Add 和 Norm 的先后位置就够了。

### 激活函数速查表

| 函数 | 一句话特点 | 常用场景 |
|------|------------|----------|
| ReLU | 简单快速，负数直接砍 | CNN 默认选择 |
| Leaky ReLU | 给负数留条活路 | 缓解死神经元 |
| GELU | 平滑的概率门控 | Transformer 标配 |
| SiLU/Swish | 自带门控的平滑曲线 | 部分 ConvNet |

## 示例代码

```python
import numpy as np

# ReLU：负数变 0，正数不变——最简单的非线性"阀门"
def relu(x):
    return np.maximum(0, x)

def relu_grad(x):
    return (x > 0).astype(np.float64)  # 正数导数 1，负数导数 0

# GELU 近似实现：Transformer 里常用的平滑激活函数
def gelu_approx(x):
    return 0.5 * x * (1 + np.tanh(np.sqrt(2 / np.pi) * (x + 0.044715 * x**3)))

# LayerNorm：对每个样本自己的特征做标准化，跟 batch 大小无关
def layer_norm(x, gamma, beta, eps=1e-5):
    mean = x.mean(axis=-1, keepdims=True)
    var = x.var(axis=-1, keepdims=True)
    x_hat = (x - mean) / np.sqrt(var + eps)
    return gamma * x_hat + beta

# Xavier 初始化：让每层输出的方差保持稳定，适配 tanh/Sigmoid
def xavier_uniform(fan_in, fan_out):
    limit = np.sqrt(6 / (fan_in + fan_out))
    return np.random.uniform(-limit, limit, size=(fan_in, fan_out))

# Kaiming 初始化：专为 ReLU 设计，考虑了负半轴被砍的方差损失
def kaiming_uniform(fan_in, fan_out):
    limit = np.sqrt(6 / fan_in)
    return np.random.uniform(-limit, limit, size=(fan_in, fan_out))

# Dropout：训练时随机关掉一些神经元，推理时记得关掉！
def dropout_mask(x, p, train=True, rng=None):
    if not train or p <= 0:
        return x
    rng = rng or np.random.default_rng()
    keep = rng.binomial(1, 1 - p, size=x.shape) / (1 - p)  # inverted dropout
    return x * keep
```

- 面试官可能会这样问你：BatchNorm 训练和推理时的行为有什么不同？为什么推理不能用当前 batch 的统计量？
- 面试官可能会这样问你：如果把残差块的 F(x) 缩小到 0.1*F(x)，或者去掉 BN，训练会怎样？
- 面试官可能会这样问你：Transformer 里为什么用 LayerNorm 而不是 BatchNorm？
- 面试官可能会这样问你：Dropout 和 L2 正则、数据增强都是"正则化"，但它们的机制有什么不同？

## 常见误区

- 很多人会搞混的地方：以为 **BatchNorm 总能加速训练**。极小 batch 或分布式训练时，每个设备上的 BN 统计量噪声太大，效果反而变差。
- 很多人会搞混的地方：**GELU 和 SiLU 混为一谈**。它们长得像但定义不同，面试时要说清楚各自的公式和常用场景。
- 很多人会搞混的地方：以为**梯度裁剪能解决梯度消失**。裁剪只防爆炸；消失要靠 ReLU、残差、Norm、好的初始化。
- 很多人会搞混的地方：**权重全初始化为零**。所有神经元会学到一模一样的东西（对称性无法打破），几乎总是错误的做法。
- 很多人会搞混的地方：**推理时忘了调用 `model.eval()`**，导致 Dropout 还在随机丢神经元、BN 还在用当前 batch 统计量，指标直接崩盘。

# 模型评估指标

- **难度**: 基础
- **分类**: 机器学习
- **标签**: [指标, AUC, 混淆矩阵, NLP, 目标检测, 交叉验证, 面试]

## 核心概念

评估指标就像**体检报告**——你不能只看一个数字就判断一个人健不健康。

血压正常不代表血糖也没问题，体重达标不代表心脏健康。模型也一样：准确率高不一定代表模型好，可能只是因为大部分数据本来就是同一类。你需要从多个角度去看，才能得到全面的"健康状况"。

不同的业务关心不同的"体检项"：医疗诊断最怕漏诊（关心召回率），垃圾邮件过滤最怕误杀好邮件（关心精确率），风控系统关心排序能力（看 AUC）。

## 详细解析

### Accuracy / Precision / Recall / F1——四大基础体检项

先用一个例子：你做了一个"判断邮件是否是垃圾邮件"的模型。

- **Accuracy（准确率）**：所有邮件里，判断对了多少比例。听起来很好对吧？但如果 100 封邮件里只有 1 封垃圾邮件，你全判成"正常"也有 99% 准确率——这就是准确率的坑。
- **Precision（精确率）**：你说是垃圾的邮件里，真正是垃圾的有多少。关注的是"报得准不准"——误报多不多。
- **Recall（召回率）**：所有真正的垃圾邮件里，你找出了多少。关注的是"有没有漏"——漏网之鱼多不多。
- **F1**：Precision 和 Recall 的调和平均，用一个数字平衡"准"和"全"。

多分类时有三种算法：macro（每个类别平等对待）、micro（全局统计）、weighted（按类别数量加权）。

### AUC-ROC 与 PR 曲线——"排名能力"体检

模型输出的是一个分数（比如 0.7），我们需要设定一个阈值（比如 0.5 以上算正类）。不同阈值会得到不同的 Precision 和 Recall。

- **ROC 曲线**：画出不同阈值下"真阳性率"和"假阳性率"的关系。**AUC** 就是这条曲线下的面积，可以理解为"随机拿一个正样本和一个负样本，模型给正样本打的分更高的概率"。
- **PR 曲线**：画出不同阈值下 Precision 和 Recall 的关系。正类很少时，PR 曲线比 ROC 更诚实。

体检类比：ROC-AUC 像"综合体质评分"，PR-AUC 像"针对某个高危疾病的专项检查"。

### 混淆矩阵——体检的"原始数据表"

一个 2×2 的表格，记录了四种情况：真正例(TP)、假正例(FP)、真负例(TN)、假负例(FN)。所有分类指标都是从这个表格算出来的。多分类扩展为 K×K 矩阵，可以看出哪两个类别最容易搞混。

**面试小技巧：回答前先声明你的行列约定（行是真实标签还是预测标签），避免读反了所有指标都算错。**

### BLEU 与 ROUGE——文本生成的"批改作文"

- **BLEU**：看生成的文本和参考答案有多少共同的 n-gram（连续 n 个词），偏向"精确率"——你生成的内容有多少是靠谱的。机器翻译常用。
- **ROUGE**：看参考答案里有多少内容被你的生成结果覆盖了，偏向"召回率"——参考答案的要点你覆盖了多少。文本摘要常用。

两者都是自动指标，跟人类判断不完全一致。重要任务还得配人工评估或用 LLM 打分。

### mAP——目标检测的"全面体检"

检测任务要同时判断"框画得准不准"（IoU）和"类别判对没"。先按置信度排序，在不同 IoU 阈值下算 PR 曲线得到 AP，再对所有类别取平均就是 mAP。COCO 数据集用 IoU 从 0.5 到 0.95 取多个阈值，要求很严格。

### 过拟合判断——"体检异常"信号

训练集上的指标远好于验证集？这就像"平时模考 90 分，正式考试只有 60 分"——典型的过拟合。解决办法：更多数据、数据增强、Dropout、权重衰减、早停、简化模型。

### 交叉验证——多做几次体检取平均

只做一次体检可能有偶然性。**K-Fold 交叉验证**把数据分成 K 份，轮流用其中一份做测试、其余做训练，最后取平均。**Stratified K-Fold** 还保持每折里的类别比例一致。

**时间序列数据不能用随机 K-Fold！** 必须用"前向链式验证"，否则就像用下周的数据预测上周——作弊了。

### 怎么选指标？看业务场景

1. **类别均衡、关心整体对错**：Accuracy + 混淆矩阵足够。
2. **不均衡、关心正类**：Recall + PR 曲线 + F1，根据业务定阈值。
3. **排序/风控**：AUC-ROC；正类极少时看 PR-AUC。
4. **文本生成**：BLEU/ROUGE 做自动参考，配合人工抽检。
5. **目标检测**：COCO mAP 标准。

## 示例代码

```python
import numpy as np
from sklearn.metrics import (
    accuracy_score, precision_recall_fscore_support,
    roc_auc_score, average_precision_score, confusion_matrix,
)
from sklearn.model_selection import StratifiedKFold, cross_val_score

# 二分类全套"体检报告"：准确率、精确率、召回率、F1、AUC、AP、混淆矩阵
def binary_report(y_true, y_score, threshold=0.5):
    y_pred = (y_score >= threshold).astype(int)
    acc = accuracy_score(y_true, y_pred)
    p, r, f1, _ = precision_recall_fscore_support(
        y_true, y_pred, average="binary", zero_division=0
    )
    auc = roc_auc_score(y_true, y_score)
    ap = average_precision_score(y_true, y_score)
    cm = confusion_matrix(y_true, y_pred)
    return {"acc": acc, "p": p, "r": r, "f1": f1, "auc": auc, "ap": ap, "cm": cm.tolist()}

# 分层 K-Fold 交叉验证：保持每折类别比例一致，结果更可靠
def stratified_cv_score(model, X, y, k=5):
    skf = StratifiedKFold(n_splits=k, shuffle=True, random_state=42)
    return cross_val_score(model, X, y, cv=skf, scoring="f1_macro")

# MCC（马修斯相关系数）：类别极不均衡时比 Accuracy 靠谱得多
def matthews_corrcoef_binary(y_true, y_pred):
    tn = np.sum((y_true == 0) & (y_pred == 0))
    tp = np.sum((y_true == 1) & (y_pred == 1))
    fn = np.sum((y_true == 1) & (y_pred == 0))
    fp = np.sum((y_true == 0) & (y_pred == 1))
    num = tp * tn - fp * fn
    den = np.sqrt((tp + fp) * (tp + fn) * (tn + fp) * (tn + fn)) + 1e-12
    return num / den
```

## 常见误区

- 很多人会搞混的地方：**准确率高 = 好模型**。类别不均衡时，全判多数类也能 Accuracy 90%+，必须看 F1、MCC 或 AUC-PR。
- 很多人会搞混的地方：把 **AUC 当成概率校准指标**。AUC 只衡量排序能力，概率准不准要看 reliability diagram 和 Brier score。
- 很多人会搞混的地方：**混淆矩阵行列读反**。不同库的约定不同（有的行是真实标签，有的是预测标签），读反了所有指标全错。
- 很多人会搞混的地方：在时间序列上用**随机 K-Fold**，用了未来数据预测过去，验证分数虚高。
- 很多人会搞混的地方：**macro-F1 和 micro-F1 不分**。macro 对每个类别平等对待，micro 按全局样本计数——类别不均衡时差异可以很大。
- 很多人会搞混的地方：以为 **BLEU 分高就是翻译好**。模型可能只是重复了一些高频 n-gram，语义可能完全跑偏。
