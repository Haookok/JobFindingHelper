# 分布式一致性

- **难度**: 深入
- **分类**: 系统设计 / 分布式
- **标签**: [CAP, BASE, Paxos, Raft, 2PC, 3PC, 最终一致性]

## 核心概念

想象你和三个朋友**同时编辑同一份在线文档**。你在北京改了标题，朋友在上海改了正文，网断了五秒——这五秒里大家看到的文档不一样，这就是"不一致"。**分布式一致性**要解决的就是：多台机器存了同一份数据的副本，怎么让大家对"数据现在长什么样"达成共识。

**强一致**就像所有人实时看到同一个屏幕，体验最好但代价最高（要不停互相确认）。**最终一致**就像微信群消息有人晚收到几秒，但最终都能收齐。工程上要根据业务选择"多强的一致性才够用"。

## 详细解析

**CAP 定理——三选二的无奈**：网络分区（P）是必然会发生的（网线会断、机房会抖），所以在出问题时你只能在"一致性 C"和"可用性 A"之间选一个。打比方：快递站系统断网了，是选择"谁都查不了快递"（保一致性，拒绝服务）还是"能查但可能显示旧状态"（保可用性，容忍不一致）？

**BASE——退一步海阔天空**：很多互联网业务选择"基本可用 + 软状态 + 最终一致"。你下单后余额可能晚几百毫秒才扣，但最终一定会扣对——这就够了。跟银行转账那种"必须立刻一致"的 ACID 相比，BASE 换来了更高的可用性。

**Raft——选个班长来记笔记**：一群服务器选出一个 Leader（班长），所有写操作都交给 Leader，Leader 把"操作日志"复制给大多数 Follower 后才算成功。Leader 挂了，大家重新选举。Raft 比 Paxos 好懂得多，etcd、部分数据库都用它。

**Paxos——学术界的经典**：和 Raft 解决同一个问题（大家对一个值达成共识），但角色划分不同，教科书里必提。面试能说清"跟 Raft 思想相近但实现更复杂"就够了。

**两阶段提交（2PC）——先举手再干活**：协调者问所有参与者"你们准备好了吗？"全说 OK 就提交，有一个说不行就回滚。问题是：如果协调者问完就挂了，参与者就卡在那里不知道该提交还是回滚——这就是"阻塞问题"。

**最终一致性的工程手段**：消息队列 + 幂等处理、定期对账、版本号冲突解决、Outbox 模式（业务写 DB 的同时写一张"发件箱"表，后台异步发消息，避免"DB 成功但消息丢了"）。

**Saga 和 TCC——长流程的拆分**：微服务里跨多个服务的"大事务"很难用 2PC，改成一连串小事务 + 补偿操作。比如订机票和订酒店，酒店订失败了就取消机票——这就是 Saga 的补偿思路。

**时钟问题**：分布式系统里不能信任"墙上的钟"（机器时间会漂移），所以用逻辑时钟（Lamport Clock）来判断事件的先后因果。面试提一句"墙上时钟不可靠"就能加分。

## 示例代码

```python
# 2PC 简化伪代码：先问所有人准备好没，再统一提交或回滚
def two_pc(participants):
    # 第一阶段：所有参与者表态
    if not all(p.prepare() for p in participants):
        for p in participants:
            p.abort()       # 有人说不行，全体回滚
        return False
    # 第二阶段：所有人都 OK，统一提交
    for p in participants:
        p.commit()
    return True
```

- **很多人会搞混的地方**：混淆"线性化"和"可串行化"——前者是单个数据的实时顺序，后者是数据库事务的隔离级别，不是一回事。

# 缓存架构设计

- **难度**: 进阶
- **分类**: 系统设计 / 缓存
- **标签**: [缓存穿透, 缓存击穿, 缓存雪崩, CacheAside, 热点Key, 一致性]

## 核心概念

缓存的思路特别朴素：**把常翻的书放在桌上，而不是每次都跑去图书馆借**。数据库是图书馆（靠谱但慢），缓存是你桌上（快但容量有限）。访问量大的时候，让大部分请求在"桌上"就能找到答案，图书馆就不会被挤爆。

但"桌上放书"这件事有三个经典的坑：**穿透**（有人老问你一本根本不存在的书，你每次都得跑图书馆确认）、**击穿**（全班最热门的那本书到期了，所有人同时冲去图书馆借）、**雪崩**（桌上的书同一时间全到期了，全班集体冲向图书馆）。

## 详细解析

**缓存穿透——查不存在的数据**：恶意请求或参数错误，查的 key 在数据库里压根不存在，缓存永远不命中。解决办法：对不存在的 key 也缓存一个短期的"空值"（"这本书确实没有"）；或者用**布隆过滤器**先拦一道——它就像一个快速名单，不在名单上的直接拒绝。

**缓存击穿——热点 key 过期的那一瞬间**：明星同款商品的缓存过期了，几千个请求同时扑向数据库。解决办法：**加锁/单飞**（只让一个请求去数据库取，其他人等着）；**逻辑过期**（缓存不真的过期，后台异步更新）；热点 key 提前预热。

**缓存雪崩——大量 key 同时失效**：比如凌晨统一刷了缓存，早高峰一到全部过期。解决办法：给 TTL **加随机抖动**（每个 key 的过期时间错开）；设置多级缓存（本地 + Redis）；准备熔断降级。

**Cache Aside——最常用的更新策略**：读的时候先查缓存，没有就查数据库再回填缓存。写的时候**先更新数据库，再删除缓存**。为什么是"删"而不是"更新"缓存？因为删更简单，下次读的时候自然会从数据库拉最新的。

**热点 Key——一个 key 被万人追捧**：单个 key QPS 极高，Redis 某个节点扛不住。解决办法：在应用本地也加一层缓存；或者把一个 key 拆成多个副本（加随机后缀），分散到不同节点读。

**缓存和数据库的一致性**：没有完美方案。常用"TTL + 删除缓存"兜底，业务能容忍短暂不一致就行。更严格的场景可以用 Canal 监听数据库 binlog 来主动清缓存。

## 示例代码

```python
import random

def cache_ttl_jitter(base_sec: int, spread: float = 0.2) -> int:
    """防雪崩：给基础 TTL 加一个随机偏移，让 key 不要同时过期"""
    return int(base_sec * (1 + random.uniform(-spread, spread)))

- **面试官可能会这样问你**："先删缓存再写数据库"和"先写数据库再删缓存"各有什么并发问题？哪种更常用？

## 常见误区

- **很多人会搞混的地方**：用布隆过滤器解决"击穿"——布隆过滤器主要防"穿透"（拦截不存在的 key），击穿要用加锁或逻辑过期。
- **很多人会搞混的地方**：空值缓存 TTL 设太长——本来不存在的数据后来写入了，结果缓存里还是"空"，造成长期脏读。
- **很多人会搞混的地方**：以为 Cache Aside 删缓存就能保证强一致——并发场景下还是有短暂不一致的窗口，业务要能容忍。
- **很多人会搞混的地方**：把本地缓存（Caffeine/Guava）当成分布式一致的——多个服务实例之间本地缓存是各自为政的，只适合能容忍短暂不一致的读多写少场景。
- **很多人会搞混的地方**：互斥锁实现不当导致死锁——一定要设短超时，否则抢到锁的进程挂了，其他人全卡住。

# 消息队列

- **难度**: 进阶
- **分类**: 系统设计 / 消息中间件
- **标签**: [Kafka, RabbitMQ, RocketMQ, 消息丢失, 重复消费, 顺序消息, 积压]

## 核心概念

消息队列就像**快递驿站**：你（生产者）寄了一个包裹放到驿站（Broker），收件人（消费者）什么时候方便什么时候去取。你不用等收件人在家，收件人也不用盯着你什么时候发货——这就是**解耦**和**异步**。

更重要的是**削峰填谷**：双十一零点涌来百万订单，如果直接打到后端数据库，立马崩盘。但如果先把订单消息扔进驿站，后端按自己的处理能力慢慢取，就像驿站帮你缓冲了快递洪峰。

代价是什么？引入了中间层就多了**延迟**（不是实时送达）、**运维复杂度**（驿站也会故障）以及**一致性难题**（包裹丢了怎么办？送重了怎么办？）。

## 详细解析

**三大主流 MQ 对比（面试简版）**：

| | Kafka | RabbitMQ | RocketMQ |
|---|---|---|---|
| 擅长 | 大数据日志流、高吞吐 | 灵活路由、低延迟任务 | 阿里系场景、事务消息 |
| 模型 | Topic + 分区，日志追加 | Exchange + Queue，AMQP 协议 | 借鉴 Kafka，加了队列级有序 |
| 消费 | 消费者组内分区独占 | 多种模式（工作队列、广播等） | 集群消费/广播消费 |

**消息丢失——三个环节都可能丢**：生产者发送时网络抖了（→ 开启发送确认）；Broker 还没落盘就宕机了（→ 开启持久化和副本同步）；消费者拿到消息还没处理就确认了（→ 先处理业务再提交 offset）。

**重复消费——比丢消息更常见**：网络重试、消费者重平衡都可能导致同一条消息被处理两次。解决核心是**业务幂等**——同一个操作执行多次效果跟执行一次一样。比如用订单号做唯一键，重复写入直接忽略。

**顺序消费——没你想的那么简单**：全局有序代价极高。通常的做法是把同一个业务 ID（比如同一个用户的操作）哈希到同一个分区，保证**分区内有序**就够了。

**消息积压——驿站包裹堆满了**：消费者处理太慢或者挂了，消息越堆越多。应急手段：扩容消费者、降级非核心逻辑、临时转发到备用队列用更多实例消费。但根因通常是下游慢或消费者有 bug。

**精确一次（Exactly Once）——理想很丰满**：端到端的精确一次在分布式场景下非常难做到。Kafka 的事务只保证"Kafka 系统内"的语义，跨数据库和 MQ 还是需要幂等 + 去重 + Outbox 模式来兜底。

**延迟队列和死信队列**：想让消息"30 分钟后再处理"？用延迟队列。消息消费失败重试多次还不行？扔进死信队列（DLQ），避免一条"毒消息"卡住整个队列。

# Docker 核心原理

- **难度**: 进阶
- **分类**: 云原生 / 容器
- **标签**: [Docker, Namespace, Cgroups, UnionFS, 镜像, Dockerfile, 网络, 安全]

## 核心概念

Docker 容器就像**打包好的外卖盒**：把你的应用代码、依赖库、运行环境全部封在一个盒子里。不管是在你的笔记本上、同事的电脑上还是云服务器上，打开就能跑，不用再操心"我这里能跑你那里跑不了"。

但容器**不是虚拟机**。虚拟机像是在一栋楼里重新盖了一间完整的房子（带自己的地基和水电），而容器更像是同一栋楼里的隔间——共用地基和水电（共享操作系统内核），只是用隔板把各个租户分开。所以容器启动快、占用资源少，但隔离性没虚拟机强。

## 详细解析

**隔间怎么做的？三大核心技术**：

**Namespace（隔板）**：让容器看到的"世界"是独立的——PID namespace 让容器觉得自己的进程是从 1 号开始的，NET namespace 给容器独立的网卡和 IP，MNT namespace 给它独立的文件目录。各个隔间互相看不见对方。

**Cgroups（水电表）**：给每个容器分配 CPU、内存、磁盘 IO 的配额。超了就限制（CPU 限流）或者杀进程（内存 OOM）。就像给每个隔间装了独立水电表，用多少付多少，用超了就断。

**UnionFS（千层蛋糕式文件系统）**：Docker 镜像像千层蛋糕——底层是操作系统（比如 Alpine Linux），上面一层是你装的 Python，再上面是你的代码。每层是只读的，容器运行时在最上面加一层可写层。多个容器可以共用底下的只读层，省空间。

**Dockerfile——外卖的配方**：用文本描述"怎么做这个外卖盒"。选基础镜像（FROM）、装依赖（RUN）、拷贝代码（COPY）、设启动命令（ENTRYPOINT）。**多阶段构建**是个好习惯：先在一个大锅（编译环境）里煮好菜，然后只把成品装进小盒子（精简运行镜像），编译器和源码都不带走。

**网络模式**：**bridge**（默认）给每个容器分配独立 IP，通过 NAT 访问外网；**host** 直接用宿主机的网络，性能好但端口会冲突；**none** 完全没网。

**数据卷**：容器是"用完即扔"的，数据不能存在容器里。把数据放到**卷（Volume）**里，就像把重要文件存在移动硬盘上，容器删了数据还在。

**安全加固**：不要用 root 跑服务（万一被入侵影响更大）、用最小镜像减少攻击面、不要把密钥写进镜像、定期扫描镜像漏洞。

RUN CGO_ENABLED=0 go build -o /app -trimpath -ldflags="-s -w" ./cmd/server

# 运行阶段：只拷贝编译好的二进制文件，镜像非常小
FROM gcr.io/distroless/static-debian12:nonroot
COPY --from=build /app /app  # 只拿成品，源码和编译器都不带
USER nonroot:nonroot          # 不用 root 跑，更安全
ENTRYPOINT ["/app"]
```

# Kubernetes 架构

- **难度**: 进阶
- **分类**: 云原生 / Kubernetes
- **标签**: [K8s, API Server, etcd, Scheduler, kubelet, Service, 工作负载, HPA, RBAC]

## 核心概念

如果 Docker 是"打包好的外卖盒"，那 Kubernetes（K8s）就是**管理整个外卖平台的调度中心**。它负责：这个外卖派给哪个骑手（调度到哪台服务器）？骑手翻车了怎么重新派单（容器挂了自动重启）？高峰期怎么多叫几个骑手（自动扩缩容）？新菜品怎么平滑上线（滚动更新）？

K8s 分两部分：**控制平面**（总部大脑）和**数据平面**（干活的节点）。总部制定策略和决策，节点执行命令和跑容器。

## 详细解析

**控制平面——总部有四个关键角色**：

**API Server（前台接待）**：所有请求的唯一入口。你想创建服务、查看状态、改配置，都得先过它。它负责身份验证（你是谁）、权限检查（你能干嘛）、数据校验。

**etcd（档案室）**：一个分布式键值数据库，存着整个集群"应该是什么样"的期望状态。API Server 收到请求后把数据存到 etcd。备份 etcd 几乎等于备份了整个集群。

**Controller Manager（各种管理员）**：一堆"管理员"不断对比"期望状态"和"实际状态"。比如 Deployment Controller 发现"我期望有 3 个副本但只有 2 个在跑"，就会创建一个新的。这种"不断检查并修正"的思路叫**调谐（Reconcile）**。

**Scheduler（派单员）**：新创建的 Pod 还没被分配到节点时，Scheduler 根据资源余量、亲和性规则、污点容忍等条件，选出最合适的节点把 Pod "派"过去。

**数据平面——每个节点上两个关键角色**：

**kubelet（骑手）**：每个节点上的代理，收到"派单"后负责拉取镜像、创建容器、做健康检查（liveness/readiness 探针）、上报状态。

**kube-proxy（路由表）**：维护 Service 到 Pod 的转发规则，让流量能找到正确的容器。

**Pod——最小的"外卖订单"**：一个 Pod 里可以有一个或多个容器，它们共享网络和存储。Pod 重建后 IP 会变，所以需要 Service 提供稳定的访问入口。

**常见工作负载**：**Deployment** 管无状态服务（比如 Web API），支持滚动更新和回滚；**StatefulSet** 管有状态服务（比如数据库），每个 Pod 有固定的名字和存储；**DaemonSet** 在每个节点上跑一份（比如日志收集器）。

**HPA（自动扩缩容）**：根据 CPU、内存或自定义指标自动调整 Pod 数量。高峰期多开几个，低谷期收回去省钱。

**RBAC（权限控制）**：谁能看什么、改什么，通过 Role 和 RoleBinding 精细控制。原则是**最小权限**——只给需要的权限。

## 示例代码

```yaml
# 一个最基本的 Deployment：跑 2 个副本，限制了资源
apiVersion: apps/v1
kind: Deployment
metadata:
  name: demo
spec:
  replicas: 2                # 期望跑 2 个 Pod
  selector:
    matchLabels:
      app: demo
  template:
    metadata:
      labels:
        app: demo
    spec:
      serviceAccountName: demo-sa
      containers:
        - name: app
          image: demo:1.0.0
          resources:
            requests:         # "我至少需要这么多"——调度器参考
              cpu: 100m
              memory: 128Mi
            limits:           # "最多给我这么多"——超了就限流/杀进程
              memory: 256Mi
```

- **面试官可能会这样问你**：kube-proxy 的 iptables 模式和 ipvs 模式有什么区别？
- **面试官可能会这样问你**：Pod 一直 Pending 怎么排查？常见原因有哪些？

## 常见误区

- **很多人会搞混的地方**：以为 Pod IP 是稳定的——Pod 一重建 IP 就变了，要用 Service 或 StatefulSet 的 DNS 来访问。
- **很多人会搞混的地方**：把 readiness 和 liveness 探针配反——readiness 是"我准备好接客了没"（没准备好就不给流量），liveness 是"我还活着没"（死了就重启）。配反了要么未就绪就接了流量，要么活着却被不断重启。
- **很多人会搞混的地方**：给所有容器设超大 limits——看起来调度成功了，但运行时节点超卖严重，大家互相抢资源频繁 OOM。
- **很多人会搞混的地方**：把 Secret 当加密存储——Secret 默认只是 base64 编码，能解码看到明文，需要配合 etcd 加密和 RBAC 保护。
- **很多人会搞混的地方**：以为 NetworkPolicy 默认就是"全部拒绝"——大多数 CNI 插件默认全通，你不写 NetworkPolicy 就等于没有网络隔离。
- **很多人会搞混的地方**：忽略 etcd 备份——etcd 存了整个集群的灵魂，它挂了又没备份，集群等于从零开始。

# Web 安全基础

- **难度**: 进阶
- **分类**: 安全 / Web
- **标签**: [XSS, CSRF, SQL注入, SSRF, 文件上传, CSP, CORS, OWASP]

## 核心概念

Web 安全就像**给房子装锁和监控**：你不能假设所有来访的人都是善意的。核心原则三条：**输入不可信**（所有用户输入都可能有猫腻）、**最小权限**（只给必要的权限）、**纵深防御**（不能只靠一道锁，要层层设防）。

最常见的几种"入室方式"：**XSS** 是在你的网页里塞了坏人的脚本，就像有人在你家墙上贴了一张假公告，来访的客人信以为真；**CSRF** 是趁你还登着某个网站的时候，骗你点一个链接自动帮坏人操作，就像有人模仿你的签名去银行取钱；**SQL 注入**是在输入框里写 SQL 代码，让数据库把家底都翻出来。

## 详细解析

**XSS（跨站脚本）——往你的页面塞坏代码**：分三种类型。**反射型**：坏人把恶意代码放在 URL 里，你点了链接就中招，像一次性的陷阱。**存储型**：恶意代码存进了数据库（比如评论区），所有看到这条评论的人都中招，杀伤力最大。**DOM 型**：纯前端问题，用 innerHTML 之类的方法直接拼接了用户输入。

防御核心：**对输出做转义**（把 `<script>` 变成无害的文本）、Cookie 设 **HttpOnly**（JS 读不到）、用 **CSP** 策略限制页面能执行什么脚本。

**CSRF（跨站请求伪造）——借你的身份干坏事**：你登着银行网站，点了一个恶意链接，浏览器自动带上你的 Cookie 发了一个转账请求。防御：Cookie 设 **SameSite**（Lax 或 Strict）、关键操作带 **CSRF Token**、敏感操作要求输密码或验证码。

**SQL 注入——在输入框里写代码**：如果后端把用户输入直接拼进 SQL 语句，攻击者可以输入 `' OR 1=1 --` 之类的东西把数据全部捞出来。防御很简单：**用参数化查询/预编译**，永远不要拼接 SQL 字符串。

**SSRF（服务端请求伪造）——骗服务器帮你访问内网**：如果你的服务接受用户传一个 URL 然后去请求，攻击者可能让你请求内网地址或云服务器的元数据接口。防御：URL 白名单、禁止访问内网 IP 段、禁用危险协议。

**文件上传——上传一个"炸弹"**：如果只检查文件后缀不检查内容，攻击者可能上传一个伪装成图片的可执行脚本。防御：白名单检查文件类型和魔数、重命名文件、存到没有执行权限的地方。

**CORS 不是安全机制**：很多人误解 CORS。它只是浏览器的"跨域读限制"，放宽的是"别的网站能不能读你的响应"。攻击者用 curl 等工具不受 CORS 限制，所以**CORS 不能替代鉴权**。

**访问控制漏洞——最容易被忽略的大坑**：改个 URL 里的 ID 就能看到别人的数据（水平越权），普通用户直接调管理接口（垂直越权）。防御：服务端**每次请求**都校验当前用户有没有权限访问这个资源，别只靠前端隐藏按钮。

## 示例代码

```http
Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none'; base-uri 'self'
Set-Cookie: sid=...; HttpOnly; Secure; SameSite=Lax; Path=/
```

说明：CSP 限制只能加载自己域名的脚本；Cookie 设了 HttpOnly（JS 读不到）、Secure（只走 HTTPS）、SameSite=Lax（跨站不自动带）。

## 常见误区

- **很多人会搞混的地方**：以为关掉 CORS 就安全了——攻击者可以不用浏览器直接发请求，CORS 只约束浏览器行为。
- **很多人会搞混的地方**：用黑名单过滤 XSS（比如替换 `<script>`）——各种编码和变形可以绕过，应该用**上下文编码 + CSP**。
- **很多人会搞混的地方**：以为"用了 JWT 就不用防 CSRF"——JWT 放在 localStorage 虽然不受 CSRF 影响，但更容易被 XSS 偷走。
- **很多人会搞混的地方**：文件上传只检查后缀名——把 .php 文件改名成 .jpg 照样能上传，要检查文件内容（魔数）。
- **很多人会搞混的地方**：只靠前端路由隐藏管理页面——直接请求 API 照样能越权，授权必须在服务端做。
- **很多人会搞混的地方**：CSP 配了 `unsafe-inline` 和 `unsafe-eval`——这基本把 XSS 防御全抵消了。

# 认证授权与加密

- **难度**: 进阶
- **分类**: 安全 / 身份与密码学
- **标签**: [Session, JWT, OAuth2, TLS, 对称加密, 非对称加密, 哈希, RBAC, 零信任]

## 核心概念

**认证**就像**进小区要刷门禁卡**——回答"你是谁"。**授权**就像刷了卡之后，物业系统判断你能去哪栋楼、能不能进地下车库——回答"你能做什么"。

**Session**：你刷卡进小区后，保安给你发了一个"通行牌号"（session ID），存在保安室的本子上（服务端）。你每次经过都亮牌号，保安查本子确认。好处是保安能随时把你的牌号划掉（即时登出），坏处是保安室要维护一大本记录。

**JWT**：换一种方式——你的通行牌上**直接写了你的身份信息和有效期**，用防伪水印签了名。保安不用查本子，扫一下水印就知道是不是真的。好处是不用维护本子（无状态），坏处是一旦发出去，在过期前很难作废（除非另建一个"黑名单"）。

**加密**分两种：**对称加密**像你和朋友共用一把钥匙——开锁快但钥匙怎么安全地给朋友是个问题。**非对称加密**像信箱——人人都能往里塞信（公钥加密），但只有你有钥匙打开（私钥解密）。实际中常常两种结合用：先用信箱方式安全交换一把共享钥匙，再用共享钥匙快速加解密大量数据。

## 详细解析

**Session vs JWT 怎么选**：Session 适合"保安室方便管"的场景——需要立即踢人下线、控制同时在线数。JWT 适合"到处跑"的场景——跨多个服务、移动端。但 JWT 有坑：别放在 localStorage（容易被 XSS 偷），Token 里别存敏感信息（Base64 不是加密，谁都能解码看），一定要设过期时间。

**OAuth2——不是登录协议，是授权框架**：你用微信登录某个 App，其实是你**授权**那个 App 去微信那里拿你的头像和昵称，不是微信帮你"登录"。真正的登录要用 OpenID Connect（在 OAuth2 上加了身份层）。

**TLS/HTTPS——快递包装纸箱**：你和服务器之间的数据像快递一样，TLS 就是给快递加了个防拆纸箱。握手过程（简化版）：双方先确认用什么暗号方式（密码套件），服务器出示"身份证"（证书），客户端验明正身后双方商定一把临时共享钥匙，之后所有数据都用这把钥匙加密传输。

**密码存储——绝对不能明文**：正确做法是用 **bcrypt/Argon2** 这类"故意很慢的哈希"加上随机的盐（salt）来存储密码。查询时不是"解密"密码，而是把用户输入的密码同样哈希一遍，看结果是否匹配。用 MD5 或 SHA256 直接哈希密码是**不安全的**——太快了，暴力破解很容易。

**RBAC（基于角色的权限控制）**：给人分组——管理员能干所有事，普通用户只能看自己的数据，访客只能浏览。简单清晰，大部分系统够用了。

**零信任——别因为在公司内网就放松警惕**：传统思维是"内网安全、外网危险"，零信任认为**谁都不可信**，每次访问都要验证身份，哪怕你坐在办公室里。就像小区不光看门禁卡，每栋楼每层楼都有独立的门锁。

## 常见误区

- **很多人会搞混的地方**：把 OAuth2 当登录协议——它是授权框架，登录要用 OpenID Connect。
- **很多人会搞混的地方**：JWT 不设过期时间或者密钥太弱——泄露了就相当于永久通行证，谁拿到都能用。
- **很多人会搞混的地方**：用 MD5/SHA256 直接哈希密码——太快了，GPU 暴力破解分分钟搞定。要用 bcrypt/Argon2 这类故意慢的算法。
- **很多人会搞混的地方**：TLS 只在网关终止，内网全走明文——内网被攻破后数据裸奔，敏感服务之间也应该加密。
- **很多人会搞混的地方**：以为零信任就是买个产品装上——不改造身份管理和网络分段，买再贵的产品也没用。
- **很多人会搞混的地方**：自签名证书然后关掉验证（InsecureSkipVerify）——中间人可以随意拦截和篡改流量，等于没加密。
- **很多人会搞混的地方**：把 JWT payload 当加密的——Base64 只是编码不是加密，任何人都能解码看内容。敏感信息不能放在里面。
- **很多人会搞混的地方**：Refresh Token 没有轮换检测——被盗了就能一直续期，应该用 rotation + reuse 检测。
