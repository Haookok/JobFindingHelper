# 排序算法

- **难度**: 基础
- **分类**: 算法 / 排序
- **标签**: [冒泡, 快排, 归并, 堆排序, 稳定性, 复杂度]

## 核心概念

排序将序列按关键字非降序（或非增序）重排。**时间复杂度**描述随 n 增长的趋势；**空间复杂度**指额外辅助空间。**稳定性**：相等元素排序后相对顺序不变——对多关键字排序或需要稳定语义时重要。

## 详细解析

| 算法 | 平均时间 | 最坏时间 | 空间 | 稳定性 |
|------|----------|----------|------|--------|
| 冒泡 | O(n²) | O(n²) | O(1) | 稳定 |
| 选择 | O(n²) | O(n²) | O(1) | 不稳定 |
| 插入 | O(n²) | O(n²) | O(1) | 稳定 |
| 归并 | O(n log n) | O(n log n) | O(n) | 稳定 |
| 快排 | O(n log n) | O(n²) | O(log n) 栈均摊 | 不稳定 |
| 堆排序 | O(n log n) | O(n log n) | O(1) | 不稳定 |

**冒泡**：相邻比较交换，每轮最大「沉底」；可提前无交换则结束。

**选择**：每轮选最小放到前面；交换可能跨过相等元素 → 不稳定。

**插入**：维护有序前缀，逐个插入；近乎有序时接近 O(n)。

**归并**：分治合并两个有序数组；需 O(n) 辅助数组，适合**链表**或**外排序**多路归并。

**快排**：选 pivot 划分；随机 pivot 或三数取中缓解最坏；小区间可换插入排序。**TimSort**（Python、Java `Arrays.sort` 对象）是归并+插入的混合工业实现。

**堆排序**：建最大堆，反复弹出堆顶；原地、最坏 O(n log n)，但常数因子与缓存友好性通常不如快排。

**实际应用场景**：

- **标准库**：C++ `std::sort` 多为内省排序（快排 + 堆排序防退化）；Go `sort.Slice` 为快排；Python `list.sort` 为 Timsort。
- **稳定性需求**：电商订单按金额排序后再按下单时间稳定排序，需稳定算法或一次复合键排序。
- **链表**：归并可 O(1) 辅助实现原地归并思路（面试常考）；快排链表需小心 pivot 选择。
- **海量数据**：外排序 = 分块排序 + k 路归并，归并思想是核心。

**计数排序 / 桶排序 / 基数排序**：在关键字范围有限或可分解为位/桶时可达 O(n) 级别，需额外空间或数据分布假设，与比较排序下界 O(n log n) 不矛盾。

## 示例代码

```python
def quick_sort(arr):
    if len(arr) <= 1:
        return arr
    pivot = arr[len(arr) // 2]
    left = [x for x in arr if x < pivot]
    mid = [x for x in arr if x == pivot]
    right = [x for x in arr if x > pivot]
    return quick_sort(left) + mid + quick_sort(right)


def merge_sort(arr):
    if len(arr) <= 1:
        return arr
    m = len(arr) // 2
    left, right = merge_sort(arr[:m]), merge_sort(arr[m:])
    return merge(left, right)


def merge(a, b):
    i = j = 0
    out = []
    while i < len(a) and j < len(b):
        if a[i] <= b[j]:
            out.append(a[i])
            i += 1
        else:
            out.append(b[j])
            j += 1
    return out + a[i:] + b[j:]


# 插入排序：近乎有序时性能好，可作快排小区间 fallback
def insertion_sort(arr):
    for i in range(1, len(arr)):
        key = arr[i]
        j = i - 1
        while j >= 0 and arr[j] > key:
            arr[j + 1] = arr[j]
            j -= 1
        arr[j + 1] = key
    return arr
```

- **追问 2**：归并排序链表为什么可以做到 O(1) 辅助空间思路？（拆半递归合并）
- **追问 3**：哪些排序是稳定的？不稳定排序举例说明「相等元素顺序改变」的场景危害。

## 常见误区

- 背复杂度却说不清**递归栈**算不算空间（快排平均 O(log n) 栈深）。
- 认为「快排一定比堆排序快」——与数据分布、实现、常数有关；堆排序最坏更稳。
- 忽略 **n 很小** 时 O(n²) 插入/希尔可能更优（小区间优化）。
- 认为 **原地排序** 空间一定是 O(1)——递归实现快排仍占调用栈空间。
- 混淆 **比较次数与赋值次数**——面试手写时要能说清主要代价来自比较还是移动元素。

# 动态规划

- **难度**: 进阶
- **分类**: 算法 / 动态规划
- **标签**: [DP, 背包, LCS, 记忆化, 状态转移]

## 核心概念

动态规划用于多阶段决策问题：大问题可拆成**子问题**，子问题**重叠**（重复出现），且问题具有**最优子结构**（最优解包含子问题的最优解）。核心是**状态定义**与**状态转移方程**，用表或变量记录已算结果，避免指数级重复递归。

实现两条路：**自顶向下记忆化（DFS + memo）** 直观、只算可达状态；**自底向上递推** 常更省栈、便于空间优化（滚动数组）。

## 详细解析

**识别 DP**：问「最优」「方案数」「能否」「最少步数」且 n 较大；暴力递归发现重复子问题。

**步骤**：定义 `dp[i]` 或 `dp[i][j]` 的含义；写转移；定初值与边界；定遍历顺序（保证依赖已算）。

**0-1 背包**：n 个物品，重量 `w[i]`、价值 `v[i]`，容量 `W`，每件最多选一次。`dp[i][c]` 表示前 i 件、容量 c 的最大价值。转移：`dp[i][c] = max(dp[i-1][c], dp[i-1][c-w[i]]+v[i])`（若 `c>=w[i]`）。可滚成一维并从大到小遍历 c 防重复使用。

**完全背包**：每件无限件，一维时从小到大遍历容量。

**最长公共子序列（LCS）**：`dp[i][j]` 为 `s1[0..i)` 与 `s2[0..j)` 的 LCS 长度。字符相等则 `dp[i][j]=dp[i-1][j-1]+1`；否则 `max(dp[i-1][j], dp[i][j-1])`。

**经典例题**：爬楼梯（斐波那契）、硬币兑换、编辑距离、最长递增子序列（LIS）、区间 DP（石子合并）等。

**最长递增子序列（LIS）**：`dp[i]` 表示以 `nums[i]` 结尾的 LIS 长度，转移为对 `j<i` 且 `nums[j]<nums[i]` 取 `max(dp[j])+1`，复杂度 O(n²)。另有 **耐心排序 + 二分** 的 O(n log n) 做法，面试常追问两种思路对比。

**编辑距离**：`dp[i][j]` 表示 `s1` 前 i 与 `s2` 前 j 的最少操作数；字符相等则继承 `dp[i-1][j-1]`，否则在三类操作（删/插/换）中取最小 +1。

**区间 DP**：如矩阵链乘、石子合并，`dp[l][r]` 表示闭区间 `[l,r]` 最优值，枚举分割点 k，注意长度从小到大填表。

## 示例代码

```python
# 0-1 背包一维优化
def knapsack(weights, values, W):
    dp = [0] * (W + 1)
    for w, v in zip(weights, values):
        for c in range(W, w - 1, -1):
            dp[c] = max(dp[c], dp[c - w] + v)
    return dp[W]


# LCS 长度
def lcs_len(s1, s2):
    m, n = len(s1), len(s2)
    dp = [[0] * (n + 1) for _ in range(m + 1)]
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if s1[i - 1] == s2[j - 1]:
                dp[i][j] = dp[i - 1][j - 1] + 1
            else:
                dp[i][j] = max(dp[i - 1][j], dp[i][j - 1])
    return dp[m][n]


# 硬币兑换：最少硬币数（无限每种），不可凑成返回 -1
def coin_change(coins, amount):
    dp = [float('inf')] * (amount + 1)
    dp[0] = 0
    for x in range(1, amount + 1):
        for c in coins:
            if c <= x:
                dp[x] = min(dp[x], dp[x - c] + 1)
    return dp[amount] if dp[amount] != float('inf') else -1
```

## 面试追问

- **追问 1**：记忆化与递推在时间复杂度上是否总一致？空间如何优化到 O(W) 或 O(min(m,n))？
- **追问 2**：如何区分 DP 与贪心？（贪心需正确性证明，如活动选择；DP 子问题不独立时常需 DP）
- **追问 3**：二维 DP 打印具体 LCS 字符串怎么做？（从 `dp[m][n]` 回溯）
- **追问 4**：状态压缩 DP 适用什么场景？ bitmask 表示集合时注意什么？

## 常见误区

- 状态定义含糊导致转移写错（尤其区间、字符串下标是「前缀」还是「以 i 结尾」）。
- 0-1 背包一维循环从小到大，导致同一物品用多次。
- 把每个 NP 难问题都硬套 DP；需先验证最优子结构与重叠子问题。
- **遍历顺序**错误：如依赖 `dp[i+1]` 却从小到大 i，导致用到未更新状态。
- 记忆化用可变对象（如 `list`）作 key 未转 `tuple`，导致缓存失效或哈希错误。

# 进程与线程

- **难度**: 基础
- **分类**: 操作系统
- **标签**: [进程, 线程, 协程, IPC, 死锁]

## 核心概念

**进程** 是资源分配的基本单位，拥有独立虚拟地址空间、文件描述符表等；**线程** 是 CPU 调度的基本单位，共享所属进程的地址空间与大部分资源，自带栈、寄存器、程序计数器等执行上下文。

**协程** 是用户态调度的轻量执行单元，由运行时或程序员协作让出执行权，切换成本低，适合高并发 I/O，但多核利用需结合线程池等模型。

**PCB（进程控制块）** 是内核描述进程的数据结构，保存 PID、状态、寄存器现场、内存指针、打开文件、调度信息等，是进程存在与切换的依据。

## 详细解析

### 进程 vs 线程

- **创建与切换**：进程更重（资源多）；线程共享地址空间，创建与上下文切换通常更轻（仍涉及内核态调度）。
- **通信**：进程间需 IPC；线程间可直接读写共享内存，但要同步。
- **健壮性**：一线程崩溃可能影响整进程；进程隔离性更好。

### 进程间通信（IPC）

- **管道（pipe）**：单向字节流，亲缘进程常用；命名管道（FIFO）可有路径名。
- **消息队列**：结构化消息，内核维护队列，适合离散消息。
- **共享内存**：最快路径之一，数据在用户空间映射同一块物理区域，需 **信号量/锁** 等协调生产者与消费者。
- **信号（signal）**：异步通知，语义简单，适合事件通知而非大块数据传输。
- **套接字**：可跨主机；本地也可用 Unix domain socket。

（不同 OS 还有 mmap、eventfd 等扩展，面试答清分类与典型场景即可。）

### 线程同步

- **互斥锁（mutex）**：临界区互斥。
- **读写锁**：读多写少时提高并发度。
- **条件变量**：等待某条件成立，需与互斥锁配合避免丢失唤醒。
- **信号量**：计数资源，可用于限流与同步。
- **自旋锁**：短期等待不休眠，适合极短临界区且在多核上避免频繁切换（误用会浪费 CPU）。

### 死锁的四个必要条件

1. **互斥**：资源独占使用。  
2. **占有且等待**：持有资源同时等待其他资源。  
3. **不可抢占**：已分配资源只能由持有者释放。  
4. **循环等待**：存在进程/线程资源的等待环。

**预防/缓解**：破坏任一条件，例如 **按序加锁**、**超时放弃**、**银行家算法**（经典教材）、减少嵌套锁、使用 **try-lock**、资源分级分配等。

## 示例代码

```c
// POSIX 线程互斥 + 条件变量（示意）
pthread_mutex_t mu = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t cv = PTHREAD_COND_INITIALIZER;
int ready = 0;

void wait_ready(void) {
    pthread_mutex_lock(&mu);
    while (!ready)
        pthread_cond_wait(&cv, &mu);  // 原子释放锁并等待
    pthread_mutex_unlock(&mu);
}

void signal_ready(void) {
    pthread_mutex_lock(&mu);
    ready = 1;
    pthread_cond_broadcast(&cv);
    pthread_mutex_unlock(&mu);
}
```

## 面试追问

- **追问 1**：为什么 `pthread_cond_wait` 必须在循环里检查条件？与虚假唤醒、状态变更有什么关系？
- **追问 2**：协程切换发生在用户态，与线程的「内核栈/用户栈」有何区别？Go 的 G-M-P 模型解决了什么问题？
- **追问 3**：共享内存 + 信号量 与 消息队列 在延迟、拷贝次数、适用场景上如何对比？

## 常见误区

- 认为「多线程一定比多进程快」：竞争锁、伪共享、调度开销可能抵消收益。
- 把 **协程** 当成免费并行：CPU 密集任务仍需多线程/多进程占满核心。
- 死锁只背四条件不举 **按序加锁、避免持有多锁** 等工程手段。
- 忽略 **文件描述符、SIGPIPE、僵尸进程** 等与进程生命周期相关的常见坑（与 IPC/子进程创建相关题）。

# IO 模型

- **难度**: 进阶
- **分类**: 操作系统 / 网络编程
- **标签**: [阻塞IO, epoll, IO多路复用, Reactor]

## 核心概念

Unix 下 **IO 分两段**：数据从设备到达内核缓冲区、再从内核缓冲区拷贝到用户空间。所谓「同步/异步」常指 **用户进程在发起 IO 后，是否需亲自等待数据就绪或拷贝完成**；「阻塞/非阻塞」指 **调用在数据未就绪时是否立刻返回**。

**IO 多路复用** 用一个阻塞点同时监视多个描述符的就绪状态，避免「每连接一线程」的阻塞等待浪费。

## 详细解析

### 五种模型（POSIX 经典划分）

1. **阻塞 IO**：调用一直阻塞到数据就绪并完成拷贝（或出错）。
2. **非阻塞 IO**：轮询 `read`，未就绪立即返回 `EAGAIN` 等，CPU 空转需节制。
3. **IO 多路复用**：`select` / `poll` / `epoll` 等阻塞在「多路等待」，返回后再对就绪 fd 读写。
4. **信号驱动 IO**：数据就绪时内核发信号，用户再读；实际工程较少作为主模型。
5. **异步 IO（真异步）**：提交请求后内核完成拷贝再通知用户（如 aio）；与「异步」一词在框架层的混用需分清。

Linux 上高并发网络服务最常见组合：**非阻塞 socket + epoll + 线程池**。

### select / poll / epoll

| 能力 | select | poll | epoll |
|------|--------|------|-------|
| fd 上限 | 受 `FD_SETSIZE` 限制 | 链表，理论上受系统限制 | 高并发友好 |
| 就绪通知 | 返回后需遍历全部 fd | 同上 | `epoll_wait` 只返回就绪集合 |
| 内核机制 | 位图扫描 | 链表扫描 | 红黑树 + 就绪链表，边缘/水平触发 |

**epoll LT（水平触发）**：只要可读/可写未处理完会持续通知；**ET（边缘触发）**：状态变化时通知一次，需 **一次性读写到 `EAGAIN`**，编程要求高但减少唤醒次数。

### Reactor 与 Proactor（对比记忆）

- **Reactor**：「就绪通知」——多路复用告知可读/可写，**应用线程** 执行 `read/write`。Nginx、Netty（NIO）、Redis 网络部分等属此类思路。
- **Proactor**：「完成通知」——由系统或线程完成 IO 拷贝后回调；Windows IOCP 更接近此形态。

单线程 Reactor 适合 I/O 密集；CPU 密集或 SSL 等可配合 **多 Reactor** 或 worker 池。

### 与语言运行时常见对应关系

- **Java NIO / Netty**：`Selector` + 非阻塞 Channel，典型主从 Reactor 或单 Reactor 多线程。
- **Go**：`netpoller` 把 epoll/kqueue 与 goroutine 结合，阻塞 `Read` 时挂起 G 而非占 OS 线程。
- **Node.js**：libuv 统一封装多路复用；单线程事件循环 + 线程池做部分阻塞/CPU 任务。

理解「**内核多路复用 + 用户态调度**」的分工，有助于回答「为什么高并发服务不采用纯阻塞多线程」类问题。

## 示例代码

```c
// epoll 极简骨架（边缘触发需循环读到 EAGAIN，此处略）
int epfd = epoll_create1(0);
struct epoll_event ev = { .events = EPOLLIN, .data.fd = listen_fd };
epoll_ctl(epfd, EPOLL_CTL_ADD, listen_fd, &ev);

struct epoll_event events[64];
for (;;) {
    int n = epoll_wait(epfd, events, 64, -1);
    for (int i = 0; i < n; i++) {
        int fd = events[i].data.fd;
        if (fd == listen_fd) {
            // accept 新连接并 epoll_ctl ADD, 设非阻塞
        } else if (events[i].events & EPOLLIN) {
            // read(fd, ...) 业务处理
        }
    }
}
```

## 常见误区

- 把 **`epoll` 等同异步 IO**：`epoll` 仍是 **就绪通知**，读数据常仍是同步拷贝。
- **惊群**：多进程 `accept` 同一 listen fd 时旧版内核可能唤醒过多进程；现代内核与 `SO_REUSEPORT` 等可缓解，面试需提「曾经问题与演进」。
- 认为 **多路复用一定优于阻塞 IO**：连接数极少、逻辑极简单时，模型复杂度可能得不偿失。
- 忽略 **写事件**：高流量下需关注 **发送缓冲区满** 时的 `EPOLLOUT` 与可写再续写，否则可能丢发送或阻塞。

# TCP 三次握手与四次挥手

- **难度**: 基础
- **分类**: 计算机网络 / 传输层
- **标签**: [TCP, 三次握手, 四次挥手, TIME_WAIT, SYN Flood]

## 核心概念

TCP 是面向连接、可靠的字节流协议。**三次握手**在数据传输前同步双方的初始序列号（ISN）并确认收发能力；**四次挥手**在关闭时让双方各自把数据发完并独立释放连接。连接状态由内核维护，与套接字 API 的 `connect`/`accept`/`close` 对应。

**序列号与确认号**：序列号标识**本端发送**的字节位置，确认号表示**期望收到的下一字节**（累积确认）。握手阶段的 SYN/FIN 各消耗一个序列号，因此 ACK 常为对端 seq+1。

## 详细解析

**三次握手流程**（客户端主动）：

1. **SYN**：客户端发 SYN=1，携带客户端 ISN（seq=x）。
2. **SYN-ACK**：服务端 SYN=1、ACK=1，ack=x+1，携带服务端 ISN（seq=y）。
3. **ACK**：客户端 ACK=1，ack=y+1，seq=x+1。此后进入 **ESTABLISHED**。

**为什么是三次而不是两次？**  
两次只能保证「客户端知道服务端能收」，无法保证「服务端知道客户端能收」。若仅两次：旧连接的延迟 SYN 可能被服务端误认为新连接并分配资源；或客户端已放弃而服务端仍认为连接已建立（**半开连接**）。第三次 ACK 让双方对序列号与可达性达成一致，并消耗掉错误的重复 SYN。

**四次挥手流程**（全双工需两端分别关闭）：

1. **FIN**：主动关闭方 FIN=1，进入 **FIN_WAIT_1**。
2. **ACK**：对端确认，主动方 **FIN_WAIT_2**，对端 **CLOSE_WAIT**（仍可向主动方发数据）。
3. **FIN**：对端也发 FIN，主动方 **TIME_WAIT**，对端 **LAST_ACK**。
4. **ACK**：主动方最后 ACK，对端关闭；主动方 **TIME_WAIT** 等待 2MSL 后彻底关闭。

**TIME_WAIT**：主动关闭连接的一方在发完最后 ACK 后进入，持续约 **2MSL**（报文在网络中最大生存时间的两倍）。作用：若最后 ACK 丢失，对端会重传 FIN，本端仍能正确响应；并避免同一四元组上旧连接的延迟报文被新连接误收。

**半连接队列（SYN 队列）与全连接队列（accept 队列）**：  
`SYN_RCVD` 状态的连接挂在 **半连接队列**（syn queue）；完成三次握手后进入 **全连接队列**，等待 `accept()` 取走。队列满时内核可能丢弃 SYN 或发 RST（行为与 OS/配置有关）。

**SYN Flood 攻击**：攻击者大量发 SYN 但不完成第三次握手，耗尽半连接队列或连接表项。**缓解**：SYN Cookie（不分配资源直到验证合法握手）、调大队列、防火墙/限速、连接代理。

**ISN 与安全性**：现代实现中 ISN 常**随机化**，降低**序列号预测**与劫持风险；与握手目的「区分新旧连接」一致。

**同时打开（simultaneous open）**：双方几乎同时发 SYN，会交换 SYN 并进入 SYN-SENT/SYN-RECEIVED，最终仍通过 ACK 完成同步——仍是「交换三个带控制位的段」这一思想，面试可顺带提**非常见边界**。

**保活与异常关闭**：`SO_KEEPALIVE` 可探测对端死机或中间设备丢弃空闲连接；与挥手不同，保活是**探测**而非优雅关闭。`RST` 用于异常复位，不经历完整四次挥手。

## 面试答题骨架（口述）

1. 先画 **客户端/服务端** 与时间线，标出 **SYN、SYN-ACK、ACK** 及 seq/ack 变化。  
2. 解释 **ISN 随机化** 与 **防止旧连接报文干扰**。  
3. 挥手强调 **半关闭**：一端 FIN 后另一端仍可发送数据。  
4. 收尾点 **TIME_WAIT 意义** 与 **SYN 队列** 在压测中的表现。

**与 UDP 对比（一句话）**：TCP 用连接状态机换**可靠有序**；UDP 无握手，适合可容忍丢包或应用层自补的场景（如 QUIC 在 UDP 上重建可靠性）。

**窗口与吞吐**：握手后还有 **慢启动、拥塞避免** 等阶段影响实际吞吐；面试若追问「为何高带宽延迟积下首包慢」，可引向 **BBR/CUBIC** 与初始窗口演进（点到为止）。

## 常见误区

- 把「三次握手」说成「交换三次报文」即可，却说不清 **序列号同步** 与 **防止历史连接** 的作用。

- 误以为 **TIME_WAIT** 只存在于服务端；**主动关闭**的一方才会经历 TIME_WAIT。
- 混淆 **MSL** 与 **RTO**：前者是报文寿命上界（用于 2MSL），后者是重传超时（由 RTT 估计驱动）。
- 认为调大 **半连接队列** 即可抗住一切 SYN 攻击——在超高发包速率下仍需 **SYN Cookie** 与网络层清洗配合。

# HTTP 与 HTTPS

- **难度**: 进阶
- **分类**: 计算机网络 / 应用层
- **标签**: [HTTP, HTTPS, TLS, HTTP2, HTTP3, Keep-Alive, 队头阻塞]

## 核心概念

**HTTP** 是无状态的应用层协议，基于请求/响应模型；**HTTPS** 在 HTTP 之下增加 **TLS**（原 SSL），提供加密、完整性校验与（可选）身份认证。版本演进主要围绕**连接复用**、**多路复用**与**传输路径**（TCP → QUIC/UDP）的优化。

**REST 与动词**：虽与版本无强绑定，面试常问 **GET 幂等**、**PUT 与 PATCH**、**Idempotency-Key** 与网关超时重试的关系。

## 详细解析

**HTTP 1.0 / 1.1 / 2 / 3 对比**：

| 维度 | 1.0 | 1.1 | 2 | 3 |
|------|-----|-----|---|---|
| 连接 | 默认短连接 | **Keep-Alive** 默认更普及 | 单连接多路复用 | 基于 **QUIC**（UDP） |
| 特性 | 简单 | 管道化（受限）、Host、分块 | **二进制分帧**、HPACK 压缩、服务端推送（实践中少用） | 0-RTT/1-RTT、连接迁移、内置加密 |
| 队头阻塞 | 每请求可占连接 | 同连接上响应仍可能互相等待 | **应用层**多流并行；**TCP 层**仍可能 HOL | QUIC 流级，缓解 TCP HOL |

**HTTPS 握手（简化）**：TCP 三次握手后进入 **TLS 握手**（以 TLS 1.2 为例）：ClientHello（支持的算法、随机数）→ ServerHello + 证书 + ServerKeyExchange 等 → 客户端验证证书并生成预主密钥 → 双方算出会话密钥 → ChangeCipherSpec，之后 HTTP 密文传输。**TLS 1.3** 常减少往返（1-RTT / 0-RTT 恢复）。

**证书验证**：客户端校验证书链是否由受信 CA 签发、域名是否匹配（SAN/CN）、有效期、是否吊销（OCSP/CRL）、证书透明度等。

**HTTP 状态码（常见）**：2xx 成功（200、201、204）；3xx 重定向（301 永久、302/307/308 临时与语义差异）；4xx 客户端错误（400、401、403、404）；5xx 服务端错误（500、502、503、504）。面试需能区分 **301 vs 302**、**401 vs 403**、**502 vs 504**。

**Keep-Alive**：同一 TCP 连接上发送多个请求/响应，减少握手开销；1.1 默认 `Connection: keep-alive`（除非显式 close）。注意与**超时**、**最大请求数**等服务器限制。

**队头阻塞**：**HTTP/1.1** 同连接上若前一个响应慢，后续响应可能被拖住；**HTTP/2** 在单连接上多流交错，但底层 **TCP 丢包重排** 仍可能造成整连接阻塞；**HTTP/3 + QUIC** 在传输层以流为单位，改善该问题。

**ALPN**：TLS 握手中通过 **Application-Layer Protocol Negotiation** 协商上层是 `h2` 还是 `http/1.1` 等，避免额外往返。

**HPACK（HTTP/2）**：头部用**静态表 + 动态表 + Huffman** 压缩；动态表在连接级维护，需注意**安全面**（CRIME 类攻击历史上有 HTTP 压缩相关讨论，HPACK 设计有所规避）。

**QUIC 要点**：基于 UDP，**连接 ID** 利于**网络切换**（Wi-Fi ↔ 蜂窝）；TLS 1.3 集成；用户态协议栈更易迭代，也带来**中间盒**与审计上的新话题。

**Cookie / Session**：HTTP 无状态，身份常靠 **Cookie** 或 **Authorization**；与 **SameSite**、**CSRF** 防护常一起考。

**缓存与条件请求（顺带）**：**Cache-Control** 控制浏览器/CDN 缓存；**ETag/If-None-Match**、**Last-Modified** 实现 **304** 节省带宽；与 **HTTPS** 结合时注意**敏感页**禁用缓存。

**安全响应头（实践）**：**CSP**、**X-Content-Type-Options**、**X-Frame-Options/Frame-Ancestors** 等与 HTTP 版本独立，面试可展示「懂 Web 全链路」。

**Upgrade 与协议切换**：**Connection: Upgrade** 用于 **WebSocket** 等；与 **HTTP/2** 的 ALPN 协商不同路径，勿混用术语。

**413/414/431**：分别对应 **Payload 过大**、**URI 过长**、**请求头过大**（常见于 Cookie 膨胀），排障时比笼统 400 更有指向性。

**Range 与断点续传**：`Accept-Ranges` / `206 Partial Content` 用于大文件下载与视频拖拽；CDN 源站与缓存策略需一致，否则易出现**异常 Range** 或缓存碎片。

- **追问 2**：HTTP/2 的**流优先级**与**流量控制**如何工作？为何仍有开发者关闭服务端推送？

## 常见误区

- 把 HTTPS 等同于「只加密」——忽略 **完整性** 与 **身份认证**（中间人若未校验证书仍可被攻击）。
- 认为 HTTP/2 一定比 1.1 快：小站点、错误配置（如未合并与压缩策略不当）可能收益有限。
- 混淆 **302** 与 **307**：后者在重定向时更严格保留方法与 body（实践中仍要看客户端实现）。
- 把 **204 No Content** 与 **200 + 空 body** 混用导致缓存与中间件行为不一致。
- 误以为 **HTTP/3** 不需要可靠传输——QUIC 在 UDP 之上实现了**用户态可靠性与拥塞控制**。

**面试串联**：可从「一次浏览器输入 URL」串起 **DNS → TCP/TLS → HTTP 版本协商 → 缓存头（Cache-Control/ETag）**。

# DNS 解析原理

- **难度**: 基础
- **分类**: 计算机网络 / 应用层
- **标签**: [DNS, 递归, 迭代, 缓存, DNS劫持, HTTPDNS]

## 核心概念

**DNS** 将域名映射为 IP（及 MX、TXT 等记录）。解析过程涉及**多级域名服务器**与**多级缓存**；查询方式分为**递归**（代客户端查到底）与**迭代**（只返下一跳，由请求方继续问）。

**记录类型速览**：**A/AAAA**（IPv4/v6）、**CNAME**（别名）、**NS**（委派权威）、**MX**（邮件交换）、**TXT**（验证与策略如 SPF）。面试能说清 **CNAME 与 A 同名的限制**（规范上避免冲突）加分。

## 详细解析

**递归查询 vs 迭代查询**：

- **递归**：客户端问本地解析器（如运营商 DNS、公共 DNS），解析器**代替**客户端一路查到权威结果并返回。客户端只发一次请求。
- **迭代**：上级服务器不代查，只返回「去问谁」的**引荐**（referral），由发起方继续查询。根/顶级域服务器对解析器通常是**迭代**应答。

实际场景中：**stub resolver → recursive resolver** 常为递归；**recursive resolver ↔ 根/TLD/权威** 多为迭代。

**DNS 缓存层级**（由近到远）：浏览器/OS **hosts** → **操作系统 stub 缓存** → **本地递归解析器**（路由器/ISP/公司 DNS）→ 可能还有**转发器**。TTL 决定记录在各层保留多久；TTL 过短增加负载，过长则变更生效慢。

**域名解析完整流程（典型）**：

1. 查浏览器/OS 缓存；未命中则向**配置的递归解析器**发查询。
2. 递归解析器查自身缓存；未命中则从**根服务器**开始（迭代）。
3. 根返回负责该 TLD（如 `.com`）的**顶级域服务器**列表。
4. 解析器问 TLD，得到**权威 DNS**（或进一步子域委派）。
5. 问权威得到最终 **A/AAAA** 等记录；递归解析器缓存并按 TTL 返回客户端。

**CNAME** 可能触发额外查询；**EDNS** 可传更大 UDP 载荷；大型站点常用**地理就近**（智能 DNS）返回不同 IP。

**DNS 劫持**：恶意或错误配置导致用户得到**错误 IP**（路由器篡改、ISP 劫持、本地 malware、伪造响应的**中间人**）。防护：DNSSEC（签名防篡改，部署率有限）、DoH/DoT（加密传输防窃听与部分篡改）、校验 HTTPS 证书。

**HTTPDNS**：客户端通过 **HTTPS API** 向服务商查询域名→IP，绕过传统 UDP DNS 路径，减轻运营商 Local DNS 劫持/调度不准问题；需 App 集成，与 **SNI**、证书校验策略要结合设计。

**Glue 记录**：在父域权威返回子域 **NS** 时，同时附带子域权威服务器的 **A/AAAA**，避免「查 NS 又要先解析 NS 主机名」的**循环依赖**。

**权威 vs 递归**：**权威**只对自己区段负责；**递归**为终端聚合答案。配置错误（如把内网递归暴露公网）会造成**开放解析器**被用于放大攻击。

**DoH / DoT**：DNS over HTTPS / TLS，解决明文 DNS 的**隐私与篡改**问题；与操作系统/浏览器默认解析器策略、企业审计需求之间存在张力。

**解析失败排查顺序**：本机 `hosts` → 换公共 DNS 对比 → `dig +trace` 看委派链 → 查权威 SOA/TTL → 是否只有部分地域异常（**智能 DNS** 配置）。

**IPv6**：**AAAA** 记录与 **双栈** 环境下，部分客户端 **Happy Eyeballs** 会并行尝试 v4/v6，影响**首连时延**与排障观感。

**SRV 记录**：在 **服务发现**（部分游戏、SIP、K8s 外围）中出现；通用 Web 仍以 **A/AAAA + 端口** 为主。

- **追问 2**：**TTL 为 0** 与**负缓存（NXDOMAIN）** 对故障切换与爬虫的影响？

## 常见误区

# MySQL 索引原理

- **难度**: 进阶
- **分类**: 数据库 / MySQL
- **标签**: [索引, B+树, 执行计划, InnoDB]

## 核心概念

InnoDB 默认使用 **B+ 树** 作为索引的物理结构：所有数据（或索引键）存放在叶子节点，叶子之间用指针串联成有序链表，便于范围扫描；非叶子节点只存键与指向子页的指针，使树更矮、减少磁盘 I/O 次。

**聚簇索引** 的叶子节点存整行数据（表通常以主键为聚簇索引）；**二级索引（非聚簇）** 叶子存「索引列 + 主键」，回表时再按主键查聚簇索引。**覆盖索引** 指查询所需列都能从某棵索引树上取得，无需回表。

## 详细解析

### B+ 树为何适合磁盘

- 多路平衡树，层高低，一次页读取可比较多个键。
- 叶子有序且链表相连，`ORDER BY`、范围条件可走索引顺序扫描。
- 与 B 树相比，B+ 非叶子不存数据页指针以外的行数据，同样节点大小下扇出更大。

### 聚簇 vs 非聚簇

- 一张 InnoDB 表 **只有一个聚簇索引**（无主键则用唯一非空列，再否则隐式 6 字节 row_id）。
- 二级索引查找：先定位叶子得到主键，再 **回表** 读聚簇叶子；若 `SELECT` 列全在二级索引里，则构成覆盖，避免回表。

### 最左前缀

联合索引 `(a,b,c)` 等价于先按 `a`、再 `a,b`、再 `a,b,c` 有序。条件必须从 **最左列** 起连续匹配才能充分用上索引；中间断档则后面列无法用于 **索引范围**，可能只剩前缀或完全不走该联合索引。

### 索引失效的常见场景（典型）

- 对索引列做函数/隐式类型转换/运算，导致无法使用索引列比较。
- 前导 `%` 的 `LIKE`（如 `'%abc'`）通常无法走 B+ 树前缀匹配。
- `OR` 两侧条件无法合并为同一索引路径时，优化器可能放弃索引。
- 优化器认为 **全表扫描更便宜**（小表、选择性差、统计信息偏差）。

### EXPLAIN 关注点（面试常问列）

- `type`：`system/const/eq_ref/ref/range/index/ALL`，从左到右一般越来越差；`ALL` 全表扫描需警惕。
- `key`：实际使用的索引；`possible_keys` 为候选。
- `rows`：预估扫描行数（估算值）。
- `filtered`：按条件过滤后剩余比例估计，可与 `rows` 一起看是否「扫很多、留很少」。
- `Extra`：`Using index` 常表示覆盖；`Using filesort`、`Using temporary` 可能带来额外成本；`Using where` 表示在存储引擎返回后再过滤。
- `ref` / `rows` 与 **索引选择性** 相关：重复值多的列单独建索引收益小，常需联合索引或调整查询。

### 索引设计小结（面试加分）

- 高频 **等值 + 范围** 条件：等值列放联合索引左侧，范围列放后（其右侧列往往无法再走范围）。
- 排序字段若与 `WHERE` 可共用同一索引，有机会 **避免 filesort**。
- 长字符串可用 **前缀索引**，代价是无法覆盖完整列做 `ORDER BY`/`GROUP BY` 的精确优化，需权衡。

## 示例代码

```sql
-- 联合索引：注意条件顺序与覆盖
CREATE INDEX idx_user_time ON orders (user_id, created_at);

-- 可走索引：最左列 + 范围
EXPLAIN SELECT * FROM orders
WHERE user_id = 1 AND created_at >= '2024-01-01';

-- 覆盖索引示例（无需回表）
EXPLAIN SELECT user_id, created_at FROM orders
WHERE user_id = 1 AND created_at >= '2024-01-01';

-- 查看执行计划（MySQL 8.0+ 也可用 EXPLAIN ANALYZE 看实际耗时）
EXPLAIN FORMAT=TREE SELECT * FROM orders WHERE user_id = 1;
```

## 面试追问

- **追问 1**：为什么说 InnoDB 的二级索引必然依赖主键？若主键很长，对二级索引有什么影响？
- **追问 2**：`count(*)`、`count(主键)`、`count(二级索引列)` 在 InnoDB 下优化器可能如何选择？和覆盖索引有什么关系？
- **追问 3**：`index merge` 是什么场景下会出现？与单独建联合索引相比各有什么利弊？

## 常见误区

- 把「建了索引」等同于「查询一定快」：写入维护索引、选择性差、回表次数多都会抵消收益。
- 认为 `WHERE a=1 OR b=2` 一定会走索引：两列若无合适合并策略，可能全表扫描。
- 忽略 **统计信息与直方图** 过期导致优化器选错计划；过度依赖单次 `EXPLAIN` 而不结合数据量与数据分布。
- 联合索引列顺序随意：应把 **高选择性、常做等值条件** 的列靠前，兼顾排序与分组需求。

# Redis 数据结构与应用

- **难度**: 进阶
- **分类**: 数据库 / Redis
- **标签**: [Redis, 底层结构, 缓存, 淘汰策略]

## 核心概念

Redis 对外提供 String、List、Hash、Set、Sorted Set 等类型，底层通过 **编码（encoding）** 在 **空间与性能** 间折中：小对象常用紧凑结构（如压缩列表、listpack），数据变大或操作需求变化时再 **编码升级** 为哈希表、跳表、双向链表等。

**缓存穿透** 指查询不存在的数据，流量直达存储；**击穿** 指热点 key 过期瞬间大量请求压垮 DB；**雪崩** 指大量 key 同时过期或 Redis 宕机导致 DB 压力陡增。**淘汰策略** 在内存达 `maxmemory` 时决定逐出哪些 key。

## 详细解析

### String

- 底层可为 **SDS（简单动态字符串）**：记录长度，`O(1)` 取长度；预分配与惰性释放减少频繁重分配；二进制安全。
- 小整型/短字符串可能走 **embstr** 等优化（实现随版本演进，面试答「对象封装 + SDS」即可）。

### List

- Redis 3.2 后常用 **quicklist**：双向链表 + 压缩列表/listpack 片段，兼顾插入与内存。
- 典型场景：消息队列（需知与专业 MQ 的可靠性差异）、最新 N 条、栈/队列。

### Hash

- 字段少且短时可用 listpack 等紧凑编码；字段多则为 **哈希表**。
- 适合对象属性存储，减少 key 数量与网络往返（对比多个 String key）。

### Set

- 整数集合 **intset** 或 **哈希表**；无序唯一成员。
- 场景：标签、共同关注（与 Sorted Set 选型看是否需要排序/分值）。

### ZSet（Sorted Set）

- 需 **按 score 排序** 的成员集合；经典底层为 **跳表 + 哈希表**：哈希保证 `O(1)` 查成员分值，跳表支持范围与排序 `O(log N)`。
- **跳表**：多层索引链表，概率平衡，实现比红黑树更简单，范围遍历友好。

### 压缩列表 / listpack（演进）

- 连续内存、紧凑存储小元素；缺点是修改可能连锁更新（历史 ziplist 更明显）。新版本倾向 listpack 降低边界风险。面试强调「小对象省内存、大对象升级」逻辑即可。

### 穿透 / 击穿 / 雪崩（应对思路）

- **穿透**：布隆过滤器、缓存空值（短 TTL）、接口校验。
- **击穿**：互斥锁重建、逻辑过期、热点永不过期 + 异步刷新。
- **雪崩**：过期时间加随机抖动、多级缓存、熔断限流、高可用集群。

### 淘汰策略（maxmemory-policy 常见值）

- `volatile-lru` / `allkeys-lru`、`volatile-lfu` / `allkeys-lfu`、`volatile-ttl`、`volatile-random`、`allkeys-random`、`noeviction`（写满报错）。根据访问模式选择 LRU 或 LFU。

## 示例代码

```bash
# String：计数、短缓存
SET page:view:1001 1 EX 3600 NX
INCR page:view:1001

# Hash：对象字段
HSET user:1 name alice age 20
HGETALL user:1

# ZSet：排行榜
ZADD rank:game 100.5 user:alice 99.2 user:bob
ZREVRANGE rank:game 0 9 WITHSCORES

# 空值防穿透（业务允许时）
SET user:notexist:999 "" EX 60 NX
```

## 面试追问

- **追问 1**：跳表的时间复杂度与红黑树对比，Redis 为何选跳表实现 ZSet 的有序部分？
- **追问 2**：`DEL` 大 key 或 `HGETALL` 大 Hash 会有什么问题？如何拆分与监控？
- **追问 3**：LRU 在 Redis 中的实现是严格 LRU 吗？LFU 解决了什么问题？

## 常见误区

- 把 Redis 当可靠消息队列默认方案：无消费组语义与强持久化保证时需自行权衡。
- 认为 ZSet 只有跳表一种实现：需说明 **编码可切换**、小集合时可能更省内存的实现。
- 雪崩只想到「加随机 TTL」：忽略 **集群故障、冷启动、热点 key** 等综合因素。
- 淘汰策略选错：例如希望保留热点却使用 `volatile-*` 且 key 无 TTL，导致无法淘汰或行为不符合预期。

# 数据库事务与隔离级别

- **难度**: 进阶
- **分类**: 数据库 / 事务
- **标签**: [ACID, 隔离级别, MVCC, 锁, InnoDB]

## 核心概念

**事务** 是一组读写操作的逻辑单元。**ACID**：原子性（全成或全败）、一致性（约束与业务不变量）、隔离性（并发事务互不干扰的语义）、持久性（提交后崩溃可恢复）。

**隔离级别** 在标准 SQL 中分为四档，从弱到强：读未提交、读已提交、可重复读、串行化。不同级别对 **脏读、不可重复读、幻读** 的容忍度不同。InnoDB 默认 **可重复读**，并通过 **MVCC + 锁** 在很大程度上抑制幻读（快照读场景）。

## 详细解析

### 三种典型异常

- **脏读**：读到别的事务未提交的数据；对方回滚则读到的数据无效。
- **不可重复读**：同一事务内两次读同一行，中间别的事务提交修改，结果不一致（侧重 **行内容变化**）。
- **幻读**：同一条件两次范围查询，后一次多出「新出现的行」（侧重 **行数/新行**）。实现上常与间隙锁讨论。

### 四种隔离级别（ANSI）

| 级别 | 脏读 | 不可重复读 | 幻读 |
|------|------|------------|------|
| 读未提交 | 可能 | 可能 | 可能 |
| 读已提交 | 否 | 可能 | 可能 |
| 可重复读 | 否 | 否 | 标准允许；InnoDB 下多数场景抑制 |
| 串行化 | 否 | 否 | 否 |

实际数据库行为以具体引擎为准；MySQL InnoDB 的 **RR** 与 Oracle 的 **RC** 是常见默认差异。

### MVCC（多版本并发控制）

- 每行有隐藏的事务 ID、回滚指针等（与版本链相关），结合 **Read View** 判断某版本对当前事务是否可见。
- **快照读**（普通 `SELECT`）通常走 MVCC，不加锁读历史版本；**当前读**（`SELECT ... FOR UPDATE`、`UPDATE`）需加锁并读最新已提交版本。
- **读已提交**：语句级快照；**可重复读**：事务级快照，故同一事务内重复读一致。

### undo log

- 用于 **事务回滚** 与 **构造旧版本**：更新时旧值入 undo，形成版本链；MVCC 顺着链找可见版本。
- undo 的清理依赖 **purge**，与不再被活跃事务需要的版本相关。

### 锁机制（InnoDB 面试常考点）

- **行锁**、**间隙锁（Gap）**、**临键锁（Next-Key = 记录锁 + 间隙锁）**：在 RR 下对范围条件加锁时可阻塞插入，缓解幻读（当前读路径）。
- **意向锁**：表级意向与行锁兼容矩阵，提高加表锁效率。
- 死锁检测与 `innodb_lock_wait_timeout` 等属于工程落地细节。
- **插入意向锁**：插入前在间隙上协调，与别的事务的间隙锁可能冲突，是并发插入下死锁日志里常见角色之一。

### 日志与崩溃恢复（与 ACID 串联）

- **redo log（WAL）**：保证 **持久性**，崩溃后可重放已提交事务的页修改。
- **binlog**：主从复制与时间点恢复；与 redo 的 **两阶段提交** 保证二者逻辑一致。
- 面试常把「隔离性」讲清后，用一两句把 **redo/binlog** 与 **原子性、持久性** 串起来，体现体系化理解。

## 示例代码

```sql
-- 查看隔离级别（会话级可改）
SELECT @@transaction_isolation;

SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1;
-- 另一会话提交修改后，本语句在 RC 下可能看到新值，RR 下快照读仍见旧值
COMMIT;

-- 当前读（会加锁，与 MVCC 快照读区分）
START TRANSACTION;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

## 面试追问

- **追问 1**：为什么很多 OLTP 系统推荐 **读已提交** 或把 RR 与业务结合？RR 下哪些语句仍可能遇到「幻读」语义争议？
- **追问 2**：`UPDATE` 未命中索引时，锁范围会如何变化？与死锁有什么关系？
- **追问 3**：binlog 与 redo log 在两阶段提交里各自解决什么问题？和事务持久性如何对应？

## 常见误区

- 把「可重复读」理解成任何读都不会看到别的事务插入：需区分 **快照读** 与 **当前读**，以及范围锁与插入阻塞。
- 认为 **undo 只有回滚用**：忽略其在 **一致性读** 中的版本链作用。
- 混淆 **不可重复读** 与 **幻读**：前者强调同一行被改，后者强调出现新行或范围结果集变化。
- 隔离级别只背表格不结合引擎：例如 InnoDB 的 RR 与 SQL Server、PostgreSQL 的实现细节并不相同。
