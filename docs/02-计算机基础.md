# 排序算法

- **难度**: 基础
- **分类**: 算法 / 排序
- **标签**: [冒泡, 快排, 归并, 堆排序, 稳定性, 复杂度]

## 核心概念

想象你手里有一把乱序的扑克牌，要把它们从小到大排好。不同的排序算法，就是不同的整理策略。有的笨但简单（一张张比），有的聪明但需要额外桌面空间。评价一个排序算法，主要看三点：**多快**（时间复杂度）、**多费空间**（空间复杂度）、**稳不稳定**（两张一样大的牌，排完后谁在前谁在后会不会变）。

## 详细解析

**冒泡排序**——最直觉的方式：从头到尾，相邻两张比大小，大的往后挪。每过一轮，最大的那张就"浮"到最后面了。O(n²)，慢但好写，稳定。

**插入排序**——像你摸牌时的习惯：左手已经排好的牌，右手每次摸一张，插到左手牌里合适的位置。如果牌本来就差不多有序，插入排序特别快，接近 O(n)。

**快排**——最常用的"聪明排法"：随便挑一张牌当"基准"，比它小的放左边，大的放右边，然后左右两堆各自重复这个过程。平均 O(n log n)，但如果每次都挑到最大或最小的牌当基准，就退化成 O(n²)了。工程上用"随机选基准"或"三数取中"来避免。

**归并排序**——"先拆后合"：把牌分成两半，各自排好，再像拉拉链一样把两堆有序的牌合并成一堆。稳定、一定是 O(n log n)，但需要额外 O(n) 的桌面空间。**链表排序和海量数据外排序**的核心思想就是归并。

**堆排序**——想象把牌堆成一个"金字塔"（最大堆），塔顶永远是最大的牌。每次把塔顶拿走放到最后，再调整剩下的牌维持金字塔结构。O(n log n) 且不需要额外空间，但实际速度通常不如快排（缓存不友好）。

| 算法 | 平均时间 | 最坏时间 | 额外空间 | 稳定 |
|------|----------|----------|----------|------|
| 冒泡 | O(n²) | O(n²) | O(1) | 是 |
| 插入 | O(n²) | O(n²) | O(1) | 是 |
| 快排 | O(n log n) | O(n²) | O(log n) | 否 |
| 归并 | O(n log n) | O(n log n) | O(n) | 是 |
| 堆排 | O(n log n) | O(n log n) | O(1) | 否 |

**实际应用**：Python 的 `list.sort` 用 Timsort（归并+插入的混合体），C++ 的 `std::sort` 用内省排序（快排+堆排序防退化）。面试时能说出"标准库用的是什么"会加分。

## 示例代码

```python
# 快排：选中间的牌当基准，小的放左边、大的放右边，递归处理
def quick_sort(arr):
    if len(arr) <= 1:
        return arr
    pivot = arr[len(arr) // 2]          # 选一张"基准牌"
    left = [x for x in arr if x < pivot]   # 比基准小的
    mid = [x for x in arr if x == pivot]   # 和基准一样大的
    right = [x for x in arr if x > pivot]  # 比基准大的
    return quick_sort(left) + mid + quick_sort(right)

# 归并排序：先拆成两半各自排好，再像拉链一样合并
def merge_sort(arr):
    if len(arr) <= 1:
        return arr
    m = len(arr) // 2
    left, right = merge_sort(arr[:m]), merge_sort(arr[m:])
    # 合并两个有序数组
    i = j = 0
    out = []
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:  # <= 保证稳定性
            out.append(left[i]); i += 1
        else:
            out.append(right[j]); j += 1
    return out + left[i:] + right[j:]
```

## 面试追问

- **面试官可能会这样问你**：快排什么时候会变成 O(n²)？工程上怎么避免？（每次选到最大/最小值当基准，用随机选择或三数取中）
- **面试官可能会这样问你**：归并排序在链表上为什么可以不用额外空间？（链表可以直接改指针拼接，不需要辅助数组）
- **面试官可能会这样问你**：哪些排序是稳定的？不稳定会有什么实际危害？（电商订单先按金额排、再按时间排，不稳定的话第二次排序会打乱第一次的结果）
- **面试官可能会这样问你**：数据量很小的时候，O(n²) 的插入排序反而可能比快排快，为什么？（常数因子小、缓存友好，所以标准库的快排在小区间会切换成插入排序）

## 常见误区

- **很多人会搞混的地方**：以为"快排一定比堆排序快"——数据分布不好时快排可能退化，堆排序反而更稳。
- **很多人会搞混的地方**：忘了快排的递归调用栈也占空间，不是 O(1) 的。
- **很多人会搞混的地方**：把"原地排序"等同于"空间 O(1)"——递归版快排虽然不用辅助数组，但调用栈占 O(log n)。
- **很多人会搞混的地方**：只背复杂度表格，却说不清每个算法的核心思路，面试时一追问就露馅。

# 动态规划

- **难度**: 进阶
- **分类**: 算法 / 动态规划
- **标签**: [DP, 背包, LCS, 记忆化, 状态转移]

## 核心概念

动态规划（DP）的核心思想就一句话：**把算过的结果记下来，下次直接查笔记，不要重复算**。想象你在算斐波那契数列，算 f(5) 需要 f(4) 和 f(3)，算 f(4) 又需要 f(3) 和 f(2)——f(3) 被算了两次！如果你拿个小本本记下来，每个数只算一次，从指数级暴力直接变成 O(n)。

DP 能用的前提：大问题能拆成**重叠的子问题**（同一个子问题反复出现），并且具备**最优子结构**（大问题的最优解包含子问题的最优解）。

## 详细解析

**做 DP 题就像填表格**，四步走：
1. **定义状态**：`dp[i]` 或 `dp[i][j]` 代表什么含义——这步最关键，定义错了后面全白搭
2. **写转移方程**：当前格子怎么从之前的格子算出来
3. **定初始值**：表格的第一行第一列填什么
4. **定遍历顺序**：保证算当前格子时，它依赖的格子已经算过了

**经典例题——0-1 背包**：你有个书包能装 W 公斤，有 n 件物品各有重量和价值，每件只能拿或不拿，怎么装价值最大？`dp[c]` 表示容量为 c 时的最大价值，每件物品要么"拿"（`dp[c-w]+v`）要么"不拿"（`dp[c]`），取较大的。**关键技巧**：一维数组从大到小遍历容量，防止同一件物品被拿多次。

**最长公共子序列（LCS）**：比如 "abc" 和 "adc" 的 LCS 是 "ac"。`dp[i][j]` 表示第一个字符串前 i 个字符和第二个前 j 个字符的 LCS 长度。字符相同就从左上角 +1，不同就取上方和左方的较大值。

**硬币兑换**：凑出金额 amount 最少需要几枚硬币？`dp[x]` 表示凑出 x 元最少需要多少枚。对每种硬币面值 c，`dp[x] = min(dp[x], dp[x-c]+1)`。

**怎么判断一道题该用 DP？** 题目问"最优"、"最少"、"方案数"、"能不能"，而且暴力递归会重复算同一个子问题——大概率就是 DP。

## 示例代码

```python
# 0-1 背包：一维优化版，从大到小遍历防止重复拿
def knapsack(weights, values, W):
    dp = [0] * (W + 1)
    for w, v in zip(weights, values):   # 逐个考虑每件物品
        for c in range(W, w - 1, -1):   # 从大到小！防止同一件拿两次
            dp[c] = max(dp[c], dp[c - w] + v)  # 拿 vs 不拿，取大的
    return dp[W]

# 硬币兑换：凑出 amount 最少要几枚硬币
def coin_change(coins, amount):
    dp = [float('inf')] * (amount + 1)  # 初始化为"不可能"
    dp[0] = 0                            # 凑出 0 元需要 0 枚
    for x in range(1, amount + 1):
        for c in coins:                  # 尝试每种硬币
            if c <= x:
                dp[x] = min(dp[x], dp[x - c] + 1)
    return dp[amount] if dp[amount] != float('inf') else -1

# LCS：两个字符串的最长公共子序列长度
def lcs_len(s1, s2):
    m, n = len(s1), len(s2)
    dp = [[0] * (n + 1) for _ in range(m + 1)]
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if s1[i - 1] == s2[j - 1]:   # 最后一个字符相同
                dp[i][j] = dp[i - 1][j - 1] + 1
            else:                          # 不同，取上或左的较大值
                dp[i][j] = max(dp[i - 1][j], dp[i][j - 1])
    return dp[m][n]
```

## 面试追问

- **面试官可能会这样问你**：记忆化搜索和递推表格有什么区别？哪个更好？（记忆化直观、只算可达状态；递推省栈、方便空间优化，各有优势）
- **面试官可能会这样问你**：DP 和贪心有什么区别？（贪心每步选局部最优就行，DP 需要考虑所有子问题；贪心需要证明正确性）
- **面试官可能会这样问你**：二维 DP 怎么输出具体的 LCS 字符串？（从 `dp[m][n]` 开始往回追踪，相等就记录字符）
- **面试官可能会这样问你**：0-1 背包的一维数组为什么必须从大到小遍历？（从小到大的话，同一件物品会被重复使用，变成"完全背包"了）

## 常见误区

- **很多人会搞混的地方**：状态定义含糊——"以 i 结尾"和"前 i 个"是完全不同的定义，搞混了转移方程就会错。
- **很多人会搞混的地方**：0-1 背包一维版本从小到大遍历容量，结果同一件物品被拿了好几次。
- **很多人会搞混的地方**：遍历顺序搞错——比如 `dp[i]` 依赖 `dp[i+1]`，却从小到大遍历 i，用到的是还没更新的值。
- **很多人会搞混的地方**：看到题就硬套 DP，但没验证是否满足"最优子结构"和"重叠子问题"两个前提。

# 进程与线程

- **难度**: 基础
- **分类**: 操作系统
- **标签**: [进程, 线程, 协程, IPC, 死锁]

## 核心概念

把电脑想象成一个工业园区。**进程就是一个个独立的工厂**——每个工厂有自己的厂房（内存空间）、设备（文件资源）、工人。工厂之间互相隔离，一个工厂着火了不会影响隔壁。

**线程就是工厂里的工人**——同一个工厂里的工人共享厂房和设备，沟通方便（共享内存），但得协调好谁用哪台机器，不然打架（竞争条件）。

**协程就是工人的"分身术"**——一个工人可以同时处理好几件事，做 A 等材料的时候切去做 B，材料到了再切回来。成本极低，但只是"假并行"（一个人干多件事，不是真的多个人）。

## 详细解析

### 进程 vs 线程——选哪个？

- **隔离性**：进程之间互不影响（一个崩了不影响另一个），线程共享地址空间，一个线程挂了整个进程都得完蛋
- **通信**：进程之间要"写信"（IPC），线程之间直接"喊话"（共享内存，但要加锁）
- **创建成本**：开一个新工厂（进程）比招一个新工人（线程）贵得多

### 进程之间怎么通信（IPC）？

就像两个工厂之间传消息，有这几种方式：
- **管道（Pipe）**：像传话筒，单向传，只能在有亲缘关系的进程间用
- **消息队列**：像寄快递，发一条一条的消息，内核帮你管理
- **共享内存**：两个工厂共用一块仓库，速度最快，但得加锁协调
- **信号（Signal）**：像拍肩膀提醒一下，只能传简单的通知
- **Socket**：最通用，甚至可以跨机器通信

### 线程之间怎么协调？

同一个工厂里多个工人共用设备，得有规矩：
- **互斥锁（Mutex）**：一次只能一个人用这台机器
- **读写锁**：读的人可以同时来，写的人必须独占
- **信号量**：最多允许 N 个人同时进入这个区域

### 死锁——谁也别想动

死锁就像两个人面对面过独木桥，谁也不肯退让。发生死锁必须同时满足四个条件：互斥、占有且等待、不可抢占、循环等待。**破解方法**：打破任一条件，比如"按固定顺序加锁"、"加锁超时主动放弃"。

## 示例代码

```c
// 用互斥锁 + 条件变量协调"生产者-消费者"
pthread_mutex_t mu = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t cv = PTHREAD_COND_INITIALIZER;
int ready = 0;

void wait_ready(void) {
    pthread_mutex_lock(&mu);
    while (!ready)                        // 用 while 而不是 if，防止"虚假唤醒"
        pthread_cond_wait(&cv, &mu);      // 等的时候自动释放锁，被唤醒后重新加锁
    pthread_mutex_unlock(&mu);
}

void signal_ready(void) {
    pthread_mutex_lock(&mu);
    ready = 1;
    pthread_cond_broadcast(&cv);          // 叫醒所有等着的人
    pthread_mutex_unlock(&mu);
}
```

## 面试追问

- **面试官可能会这样问你**：`pthread_cond_wait` 为什么必须在 while 循环里？（因为有"虚假唤醒"，被叫醒不代表条件真的满足了）
- **面试官可能会这样问你**：协程和线程的本质区别是什么？Go 的 GMP 模型解决了什么问题？（协程在用户态切换，比内核调度轻得多；GMP 让协程能利用多核）
- **面试官可能会这样问你**：共享内存和消息队列怎么选？（共享内存快但要自己加锁，消息队列用起来简单但有拷贝开销）
- **面试官可能会这样问你**：多线程一定比多进程快吗？（不一定！锁竞争激烈时线程反而更慢，进程隔离性更好也更安全）

## 常见误区

- **很多人会搞混的地方**：以为"多线程一定比多进程快"——竞争锁、伪共享等开销可能让线程方案更慢。
- **很多人会搞混的地方**：把协程当成"免费并行"——协程擅长 I/O 等待，CPU 密集任务还是得靠多线程/多进程。
- **很多人会搞混的地方**：死锁只背四个条件，面试官追问"工程上怎么防"就答不上来了。要记住"按序加锁、超时放弃、try-lock"这几招。
- **很多人会搞混的地方**：忽略僵尸进程——子进程退出但父进程没回收，会一直占着进程表项。

# IO 模型

- **难度**: 进阶
- **分类**: 操作系统 / 网络编程
- **标签**: [阻塞IO, epoll, IO多路复用, Reactor]

## 核心概念

理解 IO 模型，想象你去餐厅吃饭。IO 就是"点菜→等厨房做→菜端上来"这个过程。数据从网卡/磁盘到达内核是"厨房做菜"，从内核拷贝到你的程序是"服务员端菜上桌"。不同的 IO 模型，就是你在餐厅里不同的等菜方式。

## 详细解析

### 五种"等菜"方式

1. **阻塞 IO**——"坐在座位上干等，菜没上就不动"。最简单，但你什么都干不了。
2. **非阻塞 IO**——"每隔一分钟去厨房门口问一次：好了没？好了没？"CPU 空转，效率低。
3. **IO 多路复用**——"给服务员留个呼叫器，哪桌菜好了呼叫器就响，我再去端"。一个服务员（线程）可以同时盯着很多桌（连接），这就是 `select/poll/epoll` 的思路。
4. **信号驱动 IO**——"菜好了厨房直接喊我名字"。工程上用得少。
5. **异步 IO**——"菜做好了服务员直接端到我桌上，我啥也不用管"。真正的异步，但 Linux 上支持有限。

**高并发服务器的标配组合**：非阻塞 Socket + epoll + 线程池。

### select / poll / epoll 的区别

打个比方：你是一个餐厅经理，要知道哪些桌的客人需要服务。

- **select/poll**：你必须**挨桌走一遍问**"需要服务吗？"——桌子越多越累，O(n)
- **epoll**：你给每张桌装了**呼叫铃**，只有按了铃的桌你才去——O(1) 定位到就绪的桌子

epoll 还有两种模式：**LT（水平触发）**——铃一直响直到你处理完；**ET（边缘触发）**——铃只响一次，你必须一次性处理完所有事情，没处理完就没了。ET 效率更高但编程更难。

### Reactor 与 Proactor

- **Reactor**（就绪通知）：呼叫铃告诉你"菜好了"，但你得自己去端。Nginx、Redis、Netty 都是这个模式。
- **Proactor**（完成通知）：服务员把菜直接端到你桌上才通知你"可以吃了"。Windows IOCP 是这种。

## 示例代码

```c
// epoll 的极简骨架——一个"呼叫铃系统"
int epfd = epoll_create1(0);                    // 创建呼叫铃管理器
struct epoll_event ev = { .events = EPOLLIN, .data.fd = listen_fd };
epoll_ctl(epfd, EPOLL_CTL_ADD, listen_fd, &ev); // 给监听的桌子装上铃

struct epoll_event events[64];
for (;;) {
    int n = epoll_wait(epfd, events, 64, -1);   // 等铃响，返回响了的桌子列表
    for (int i = 0; i < n; i++) {
        int fd = events[i].data.fd;
        if (fd == listen_fd) {
            // 有新客人来了（新连接），给新桌子也装上铃
        } else if (events[i].events & EPOLLIN) {
            // 某桌的菜好了（数据可读），去处理业务
        }
    }
}
```

## 面试追问

- **面试官可能会这样问你**：epoll 是异步 IO 吗？（不是！epoll 是"告诉你数据准备好了"，你还得自己去读，读的过程是同步的）
- **面试官可能会这样问你**：ET 模式下为什么必须用非阻塞 fd？（因为你必须一次读完所有数据直到返回 EAGAIN，如果 fd 是阻塞的，最后一次 read 会卡住）
- **面试官可能会这样问你**：Go/Node.js/Java NIO 底层用的什么 IO 模型？（Go 用 netpoller 封装 epoll，Node.js 用 libuv，Java NIO 用 Selector）
- **面试官可能会这样问你**：连接数很少的时候，还有必要用 epoll 吗？（不一定，简单的阻塞 IO 可能更合适，epoll 的复杂度反而是负担）

## 常见误区

- **很多人会搞混的地方**：把 epoll 等同于异步 IO——epoll 只是"就绪通知"，读数据仍然是同步拷贝。
- **很多人会搞混的地方**：以为 IO 多路复用一定比阻塞 IO 好——连接数极少时，阻塞 IO 反而更简单高效。
- **很多人会搞混的地方**：忽略"写事件"——高流量下发送缓冲区可能满了，必须关注 EPOLLOUT 才能正确续发数据。
- **很多人会搞混的地方**：惊群问题——多个进程同时 accept 同一个 listen fd 时，旧版内核会唤醒所有进程，造成浪费。

# TCP 三次握手与四次挥手

- **难度**: 基础
- **分类**: 计算机网络 / 传输层
- **标签**: [TCP, 三次握手, 四次挥手, TIME_WAIT, SYN Flood]

## 核心概念

三次握手就像**打电话确认对方在不在**：你说"喂，听得到吗？"（SYN），对方说"听得到，你能听到我吗？"（SYN-ACK），你说"能听到！"（ACK）——这才开始正式聊天。目的是让双方确认：**我能发、你能收；你能发、我能收**，同时交换各自的"起始编号"（序列号），后面传数据就按编号排队。

四次挥手就像**挂电话**：因为电话是双向的，你说完了不代表对方也说完了。你先说"我说完了"（FIN），对方说"好的我知道了"（ACK），但对方可能还有话没说完，说完后对方也来一句"我也说完了"（FIN），你回"好的拜拜"（ACK）。

## 详细解析

### 三次握手流程

1. **客户端 → 服务端**：发 SYN，带上自己的起始编号 x（"我想和你建立连接，我从编号 x 开始"）
2. **服务端 → 客户端**：回 SYN+ACK，确认收到 x（ack=x+1），同时带上自己的起始编号 y
3. **客户端 → 服务端**：回 ACK，确认收到 y（ack=y+1）。连接建立！

**为什么不能只握两次？** 两次只能证明"客户端能发、服务端能收"，但服务端不知道客户端能不能收到自己的消息。更重要的是：如果网络上有个很久以前发的旧 SYN 延迟到达了，服务端会误以为是新连接请求，白白分配资源——第三次握手就是为了让客户端告诉服务端"这是我真的要建立的连接"。

### 四次挥手流程

1. 主动方发 FIN："我说完了"
2. 被动方回 ACK："收到了"（但我可能还有话说）
3. 被动方发 FIN："我也说完了"
4. 主动方回 ACK："好的拜拜"，然后等 **2MSL** 再彻底关闭

**TIME_WAIT 是什么？** 主动关闭方发完最后一个 ACK 后，会等待约 2 个 MSL（报文最大存活时间）。为什么不直接关？第一，万一最后那个 ACK 丢了，对方会重发 FIN，你得还能回应；第二，防止旧连接的"幽灵数据包"跑到新连接里捣乱。

### SYN Flood 攻击

坏人疯狂发 SYN 但永远不回第三次 ACK，服务端的"半连接队列"被塞满。**解决方案**：SYN Cookie（不预先分配资源，收到第三次 ACK 时再验证并创建连接）、调大队列、防火墙限速。

## 示例代码

```python
import socket

# 用户代码只需调用 connect/close，三次握手和四次挥手由内核自动完成
s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.connect(("example.com", 80))   # 这一行触发三次握手
s.sendall(b"GET / HTTP/1.1\r\nHost: example.com\r\n\r\n")
s.close()                         # 这一行触发四次挥手
```

## 面试追问

- **面试官可能会这样问你**：第三次握手可以顺便带数据吗？TFO（TCP Fast Open）了解吗？（可以，TFO 允许在 SYN 中携带数据，减少一个 RTT）
- **面试官可能会这样问你**：大量短连接导致 TIME_WAIT 堆积怎么办？（`SO_REUSEADDR`/`SO_REUSEPORT`、连接池复用、调内核参数）
- **面试官可能会这样问你**：怎么区分 SYN Flood 和正常的半连接？（`ss -ant` 看大量 SYN_RECV 状态；正常情况下半连接队列不会长期堆积）
- **面试官可能会这样问你**：Nagle 算法和延迟 ACK 凑在一起会怎样？（可能导致小包延迟变大，交互式应用通常开 `TCP_NODELAY`）

## 常见误区

- **很多人会搞混的地方**：把三次握手说成"交换三次报文就完事了"，却说不清核心目的是**同步序列号**和**防止历史连接干扰**。
- **很多人会搞混的地方**：以为四次挥手"多出来的一次"只是客气——实际是 TCP 全双工，两个方向的数据流要分别关闭。
- **很多人会搞混的地方**：以为 TIME_WAIT 只出现在服务端——**主动关闭**的那一方才有 TIME_WAIT，客户端主动断开也会有。
- **很多人会搞混的地方**：混淆 MSL 和 RTO——MSL 是"报文在网络中最多活多久"，RTO 是"多久没收到 ACK 就重传"，是两个完全不同的概念。

# HTTP 与 HTTPS

- **难度**: 进阶
- **分类**: 计算机网络 / 应用层
- **标签**: [HTTP, HTTPS, TLS, HTTP2, HTTP3, Keep-Alive, 队头阻塞]

## 核心概念

**HTTP 就像寄明信片**——内容写在外面，谁都能看到（明文传输），简单直接。**HTTPS 就像寄加密信**——信放在锁着的盒子里，只有收件人有钥匙能打开。HTTPS = HTTP + TLS，不仅加密内容，还能验证"这封信真的是那个人寄的"（身份认证）和"路上没被人偷改过"（完整性）。

HTTP 的版本升级，核心都在解决一个问题：**怎么让一根网线上同时传更多东西、更快**。

## 详细解析

### HTTP 版本演进——一句话总结

- **HTTP/1.0**：每次请求都要重新建连接，像每次寄信都要重新去邮局排队
- **HTTP/1.1**：引入 **Keep-Alive**，一个连接可以连续发多封信，不用每次排队。但信必须按顺序处理——前一封卡住了后面都得等（**队头阻塞**）
- **HTTP/2**：一个连接上可以同时发多封信且互不干扰（**多路复用**），还压缩了信封（HPACK 头部压缩）。但底层用的 TCP 丢一个包整条路都会卡
- **HTTP/3**：换了条新路——基于 **QUIC**（UDP 上实现的可靠传输），每封信走自己的"小车道"，一封信丢了不影响其他信

### HTTPS 握手——怎么交换那把"锁的钥匙"？

TCP 连接建好后，开始 TLS 握手（以 TLS 1.2 为例）：
1. 客户端："我支持这些加密算法，这是我的随机数"
2. 服务端："用这个算法吧，这是我的证书和随机数"
3. 客户端验证证书是否合法（是不是权威 CA 签发的、域名对不对、过没过期），然后生成一个"预主密钥"
4. 双方用三个随机数算出会话密钥，之后所有数据都用这个密钥加密

**TLS 1.3** 把这个过程从 2 个往返缩短到了 1 个，甚至能做到 0-RTT 恢复。

### 常见状态码——记住这几组就够了

- **2xx**：成功（200 OK、201 Created、204 No Content）
- **3xx**：重定向（301 永久搬家、302 临时搬家）
- **4xx**：你的问题（400 请求有误、401 没登录、403 没权限、404 找不到）
- **5xx**：服务器的问题（500 内部错误、502 网关收到垃圾回复、503 服务器忙、504 网关超时）

## 示例代码

```bash
# 用 openssl 查看网站的证书信息
openssl s_client -connect example.com:443 -servername example.com </dev/null 2>/dev/null \
  | openssl x509 -noout -subject -dates
```

```python
import urllib.request
# 一行代码发 HTTPS 请求，TLS 握手在底层自动完成
req = urllib.request.Request("https://example.com")
with urllib.request.urlopen(req, timeout=10) as r:
    print(r.status, r.getheader("Content-Type"))
```

## 面试追问

- **面试官可能会这样问你**：TLS 1.3 比 1.2 少了几个往返？0-RTT 有什么安全风险？（少了一个 RTT；0-RTT 数据可能被重放攻击）
- **面试官可能会这样问你**：HTTP/2 既然多路复用了，为什么还有队头阻塞？（TCP 层面丢一个包，整条连接都要等重传，所以 HTTP/3 换成了 QUIC）
- **面试官可能会这样问你**：301 和 302 有什么区别？浏览器行为有什么不同？（301 永久重定向，浏览器会缓存；302 临时的，每次都要问服务器）
- **面试官可能会这样问你**：把 HTTPS 理解成"加密就安全了"有什么问题？（如果不验证证书，中间人可以伪造证书拦截——所以 HTTPS 的三大功能是加密、完整性、身份认证）

## 常见误区

- **很多人会搞混的地方**：把 HTTPS 等同于"只是加密了"——忽略了完整性校验和身份认证，不验证证书照样能被中间人攻击。
- **很多人会搞混的地方**：以为 HTTP/2 一定比 HTTP/1.1 快——小网站、配置不当时可能没什么优势。
- **很多人会搞混的地方**：混淆 302 和 307——302 在实践中浏览器可能把 POST 改成 GET，307 严格保留原方法。
- **很多人会搞混的地方**：以为 HTTP/3 不需要可靠传输——QUIC 在 UDP 之上实现了自己的可靠性和拥塞控制，只是不用 TCP 了。

# DNS 解析原理

- **难度**: 基础
- **分类**: 计算机网络 / 应用层
- **标签**: [DNS, 递归, 迭代, 缓存, DNS劫持, HTTPDNS]

## 核心概念

DNS 就是**互联网的电话簿**。你记得朋友的名字（域名 `google.com`）但不记得电话号码（IP 地址 `142.250.xx.xx`），DNS 帮你查号码。整个过程涉及"多级电话簿"和"多层缓存"——先查手机通讯录（本地缓存），查不到再打 114（递归解析器），114 再去翻各级电话簿（根域名服务器→顶级域→权威域名服务器）。

## 详细解析

### 递归查询 vs 迭代查询

- **递归查询**——你打给 114（递归解析器），114 帮你从头查到尾，直接告诉你最终号码。你只要打一个电话。
- **迭代查询**——你打给一个人，他说"我不知道，你去问张三"，你再打给张三，张三说"你去问李四"……你自己一路问下去。

**实际情况**：你的电脑（客户端）对递归解析器是**递归**的——"帮我查到底"；递归解析器去问根域名服务器、顶级域服务器是**迭代**的——每一级只告诉它"下一步该问谁"。

### 完整的查号流程

1. 浏览器/操作系统**查本地缓存**（之前查过的号码还记着呢）
2. 没命中，问**递归解析器**（通常是运营商或公共 DNS 如 8.8.8.8）
3. 递归解析器也没有，去问**根域名服务器**："`.com` 归谁管？"
4. 根说："去问 `.com` 的**顶级域服务器**"
5. 顶级域说："去问 `google.com` 的**权威域名服务器**"
6. 权威服务器给出最终 IP 地址，结果一路缓存回来

**TTL**（Time To Live）决定这个号码在电话簿里保留多久。TTL 太短=每次都要重新查，负载大；TTL 太长=号码换了你还用旧的。

### DNS 劫持

有人在你和 114 之间"窃听改答案"，把你引到假网站。**防护手段**：DNSSEC（给 DNS 记录加数字签名）、DoH/DoT（加密 DNS 查询，防窃听和篡改）、HTTPDNS（App 直接通过 HTTPS API 查 DNS，绕过运营商劫持）。

## 示例代码

```bash
# 追踪 DNS 解析的全过程，看到从根→顶级域→权威的每一跳
dig +trace example.com

# 指定用 Google 的 DNS 来查
dig @8.8.8.8 example.com A
```

```python
import socket
# 最简单的 DNS 查询——操作系统帮你搞定
print(socket.getaddrinfo("example.com", 443, type=socket.SOCK_STREAM))
```

## 面试追问

- **面试官可能会这样问你**：DNS 用 UDP 还是 TCP？什么时候会用 TCP？（默认 UDP，响应太大被截断时改用 TCP，另外区域传输也用 TCP）
- **面试官可能会这样问你**：TTL 设为 0 会怎样？（每次都要重新查，负载大但域名切换生效快，适合故障切换场景）
- **面试官可能会这样问你**：CDN 是怎么利用 DNS 做调度的？（智能 DNS 根据用户的地理位置返回最近的 CDN 节点 IP）
- **面试官可能会这样问你**：HTTPDNS 是什么？有什么局限？（通过 HTTPS API 查 DNS，绕过劫持，但要处理证书域名校验、Cookie 域名等问题）

## 常见误区

- **很多人会搞混的地方**：以为"递归查询是客户端自己从根开始查"——其实是递归解析器帮你查，客户端只是发了一个递归请求。
- **很多人会搞混的地方**：CNAME 和 A 记录的关系——CNAME 是"别名"，指向另一个域名；A 记录直接对应 IP。同一个域名不能同时有 CNAME 和其他记录。
- **很多人会搞混的地方**：以为 `dig +trace` 看到的就是用户真实的解析路径——实际用户走的是配置的递归解析器的缓存路径，可能完全不同。
- **很多人会搞混的地方**：把 HTTPDNS 当万能药——还是要处理证书校验、Cookie 域匹配、合规隐私等问题。

# MySQL 索引原理

- **难度**: 进阶
- **分类**: 数据库 / MySQL
- **标签**: [索引, B+树, 执行计划, InnoDB]

## 核心概念

索引就是**书的目录**。一本 1000 页的书，你要找"动态规划"这个词，没有目录就得从第一页翻到最后一页（全表扫描）。有了目录，你先看目录找到"动态规划——第 342 页"，直接翻过去就行。MySQL InnoDB 的索引用的是 **B+ 树**——一种特别适合磁盘读写的"多层目录"结构。

## 详细解析

### 为什么用 B+ 树？

想象一本字典的目录分三级：第一级按字母分（A-Z），第二级按拼音分，第三级才是具体词条。这就是"多路平衡树"的思想——**树越矮，翻的次数越少**。B+ 树只在最底层（叶子节点）存真实数据，上面的层只存"路标"，所以同样大小的节点能存更多路标，树就更矮。叶子之间还用链表串起来，**范围查询（如查价格 100-200 的商品）可以顺着链表滑过去**，非常高效。

### 聚簇索引 vs 二级索引

- **聚簇索引**就像正文本身按目录排好了顺序——InnoDB 的表数据就按主键排列存储。一张表只有一个聚簇索引。
- **二级索引（非聚簇索引）**就像书后面的"术语索引"——告诉你"这个词在第 X 页"，但你还得翻到第 X 页去看完整内容。这个"翻回去看"的过程就叫**回表**。
- **覆盖索引**：如果你要查的字段恰好就在索引里，那就不用回表了——相当于索引本身就够了。

### 最左前缀原则

联合索引 `(姓, 名)` 就像电话簿先按姓排、再按名排。你可以查"姓张的"（用到索引），也可以查"姓张名三的"（完全用到索引），但不能只查"名三的"——因为电话簿不是按名字排的，你还是得从头翻。

### 索引什么时候"失灵"？

- 对索引列做计算或函数：`WHERE YEAR(create_time) = 2024`——相当于你把目录里的页码都改了，目录就没用了
- 开头模糊查询：`WHERE name LIKE '%张'`——你不知道姓什么，电话簿目录帮不了你
- 类型不匹配：字符串列拿整数去查，MySQL 悄悄帮你转了类型，索引就不走了
- 数据太少或区分度太低：查"性别=男"这种，表里一半都是男的，MySQL 觉得翻目录还不如直接全表扫

### EXPLAIN 怎么看？

- **type**：`ALL` 是全表扫描（最差），`ref`/`range` 是走索引，`const` 是直接命中（最好）
- **key**：实际用了哪个索引
- **rows**：预估要扫多少行
- **Extra**：`Using index` = 覆盖索引（好），`Using filesort` = 额外排序（可能需要优化）

## 示例代码

```sql
-- 建联合索引：先按 user_id 等值匹配，再按 created_at 范围查询
CREATE INDEX idx_user_time ON orders (user_id, created_at);

-- 走索引：最左列 user_id 等值 + 第二列范围
EXPLAIN SELECT * FROM orders
WHERE user_id = 1 AND created_at >= '2024-01-01';

-- 覆盖索引：只查索引里有的列，不用回表
EXPLAIN SELECT user_id, created_at FROM orders
WHERE user_id = 1 AND created_at >= '2024-01-01';
```

## 面试追问

- **面试官可能会这样问你**：主键为什么建议用自增整数？（自增保证插入有序，不会造成页分裂；整数比长字符串主键省空间，二级索引里都存着主键值呢）
- **面试官可能会这样问你**：`count(*)` 会走哪个索引？（InnoDB 可能会选最小的二级索引来扫，因为二级索引比聚簇索引小）
- **面试官可能会这样问你**：建了索引查询就一定快吗？（不一定！写入时要维护索引、区分度低不如全表扫、回表次数多也会很慢）
- **面试官可能会这样问你**：联合索引的列顺序怎么定？（等值条件的列放前面、范围条件放后面，同时考虑排序和分组的需求）

## 常见误区

- **很多人会搞混的地方**：以为"建了索引就一定快"——写入要维护索引有开销，区分度低、回表多时可能更慢。
- **很多人会搞混的地方**：联合索引的列顺序随便写——应该把高选择性的等值条件列放前面。
- **很多人会搞混的地方**：看 EXPLAIN 只看 type，不看 rows 和 Extra——type 是 range 但 rows 几百万，照样慢。
- **很多人会搞混的地方**：以为 `WHERE a=1 OR b=2` 一定会走索引——两个条件在不同列上，MySQL 可能直接放弃索引。

# Redis 数据结构与应用

- **难度**: 进阶
- **分类**: 数据库 / Redis
- **标签**: [Redis, 底层结构, 缓存, 淘汰策略]

## 核心概念

Redis 就像你桌上的**超快便利贴系统**——所有数据都贴在内存里，随手就能查到，比去档案室（磁盘数据库）翻快上百倍。它提供了 5 种"便利贴"类型：字符串、列表、哈希、集合、有序集合。每种类型底层会根据数据大小自动切换存储方式——数据少的时候用紧凑格式省空间，数据多了就升级成更高效的结构。

## 详细解析

### 五种数据类型——用生活场景理解

**String（字符串）**——最基础的一张便利贴，上面可以写数字、文字、甚至二进制数据。用途：计数器（页面浏览量）、短期缓存、分布式锁。

**List（列表）**——一摞便利贴，可以从头或尾塞/取。用途：消息队列、最新 N 条记录。底层用的是 quicklist（链表+压缩块的混合体）。

**Hash（哈希）**——一张分栏便利贴，每栏有名有值。比如用户信息：`name=张三, age=25, city=北京`。比用多个 String key 更省内存和网络往返。

**Set（集合）**——一堆不重复的标签。用途：共同关注、标签系统、去重。

**ZSet（有序集合）**——每个标签带一个分数，自动按分数排序。用途：排行榜、延时队列。底层用**跳表+哈希表**——跳表支持按分数范围查找 O(log N)，哈希表支持按成员查分数 O(1)。

**跳表长什么样？** 想象一本书有很多页，你在第 1、10、20、50 页贴了书签（索引层），找第 35 页时先跳到 20 页的书签，再往后翻，比从头一页页翻快得多。跳表就是多层"书签"的链表。

### 缓存三大坑——穿透、击穿、雪崩

- **穿透**：有人一直查一个根本不存在的数据，缓存永远没有，每次都打到数据库。**解法**：布隆过滤器挡住不存在的 key，或者缓存空值。
- **击穿**：一个超级热门的 key 刚好过期了，瞬间大量请求涌向数据库。**解法**：加互斥锁只让一个请求去数据库重建缓存，或者热点 key 永不过期+后台异步刷新。
- **雪崩**：大量 key 同时过期，或者 Redis 直接挂了。**解法**：过期时间加随机值（不让它们同时过期）、多级缓存、集群高可用。

### 淘汰策略——内存满了怎么办？

当 Redis 内存到达上限（`maxmemory`），需要决定扔掉哪些便利贴。常见策略：LRU（最近最少使用的扔掉）、LFU（用得最少的扔掉）、TTL（快过期的优先扔）、随机扔、不扔（写满就报错）。

## 示例代码

```bash
# String：页面浏览计数器，1 小时过期
SET page:view:1001 1 EX 3600 NX   # NX = 只在 key 不存在时设置
INCR page:view:1001                 # 原子加 1，不怕并发

# Hash：存用户信息，比多个 String key 更省内存
HSET user:1 name alice age 20
HGETALL user:1

# ZSet：游戏排行榜，分数越高排名越前
ZADD rank:game 100.5 user:alice 99.2 user:bob
ZREVRANGE rank:game 0 9 WITHSCORES  # 取 Top 10

# 缓存空值防穿透：查了数据库发现没有，缓存一个空值 60 秒
SET user:notexist:999 "" EX 60 NX
```

## 面试追问

- **面试官可能会这样问你**：Redis 的 ZSet 为什么用跳表不用红黑树？（跳表实现更简单、范围查询更友好、并发改造更容易）
- **面试官可能会这样问你**：删除一个几百万元素的大 key 会怎样？（会阻塞 Redis 主线程很久，应该用 `UNLINK` 异步删除或分批删）
- **面试官可能会这样问你**：Redis 的 LRU 是严格 LRU 吗？（不是，是近似 LRU——随机采样几个 key 淘汰最旧的。LFU 解决了"偶尔被访问一次就续命"的问题）
- **面试官可能会这样问你**：穿透、击穿、雪崩能用一句话分别概括吗？（穿透=查不存在的；击穿=热点过期；雪崩=大面积失效）

## 常见误区

- **很多人会搞混的地方**：把 Redis 当成可靠消息队列——Redis 没有消费组确认机制和强持久化保证，要可靠就用 Kafka/RabbitMQ。
- **很多人会搞混的地方**：以为 ZSet 只有跳表——数据量小的时候底层可能是压缩列表/listpack，更省内存。
- **很多人会搞混的地方**：雪崩只想到"加随机 TTL"——如果是 Redis 集群故障导致的雪崩，加 TTL 没用，要靠集群高可用和降级限流。
- **很多人会搞混的地方**：淘汰策略选了 `volatile-lru` 但 key 都没设 TTL——这等于没有候选淘汰对象，内存满了直接报错。

# 数据库事务与隔离级别

- **难度**: 进阶
- **分类**: 数据库 / 事务
- **标签**: [ACID, 隔离级别, MVCC, 锁, InnoDB]

## 核心概念

**事务就是银行转账的安全保证**。你从 A 账户转 100 块到 B 账户，这包含两步：A 减 100、B 加 100。事务保证这两步**要么都成功，要么都不做**——不会出现 A 扣了钱但 B 没收到的情况。

这就是 **ACID**：**A**tomicity（原子性，全成全败）、**C**onsistency（一致性，钱的总数不变）、**I**solation（隔离性，多个转账同时进行互不干扰）、**D**urability（持久性，转完账就算断电也不会丢）。

## 详细解析

### 并发事务会出什么问题？

想象银行只有一个柜员（数据库），但有很多客户同时办业务。如果不加管理，就会出三种"事故"：

- **脏读**：你看了隔壁正在办理中（还没确认）的转账金额，结果人家撤销了——你读到的是"假数据"
- **不可重复读**：你查了一次余额是 1000，过一会儿再查变成 900 了——因为中间有人转走了钱。同一个事务里，两次读的结果不一样
- **幻读**：你查"今天有几笔转账"，第一次查是 10 笔，第二次查变成 11 笔了——因为有人新增了一笔。重点是"多出来了新行"

### 四种隔离级别——从松到严

| 级别 | 脏读 | 不可重复读 | 幻读 | 好比... |
|------|------|-----------|------|---------|
| 读未提交 | 会 | 会 | 会 | 什么都不管，跟"裸奔"一样 |
| 读已提交 | 不会 | 会 | 会 | 只看最终确认的结果，但每次可能不同 |
| 可重复读 | 不会 | 不会 | InnoDB 大部分场景也不会 | MySQL 默认级别，事务开始时"拍个快照" |
| 串行化 | 不会 | 不会 | 不会 | 所有人排队一个个来，最安全但最慢 |

### MVCC——"时光快照机"

InnoDB 怎么做到"可重复读"的？用 **MVCC（多版本并发控制）**。打个比方：你在事务开始的那一刻"拍了一张照片"，之后不管别人怎么改数据，你看到的都是照片里的样子。

技术上，每行数据都有个版本链（通过 undo log 串起来），每个事务带着一个"可见性规则"（Read View），决定能看到哪个版本。

- **普通 SELECT**（快照读）：看"照片"，不加锁
- **SELECT ... FOR UPDATE**（当前读）：看最新版本，要加锁

### 锁——InnoDB 怎么防幻读？

在"当前读"的场景下，InnoDB 用了一种叫 **Next-Key Lock** 的锁（行锁 + 间隙锁），不仅锁住已有的行，还锁住行之间的"缝隙"，防止别的事务在中间插入新行。

### redo log 和 undo log——怎么保证不丢数据？

- **redo log**：保证持久性。事务提交时先把"改了什么"写到日志里，即使断电，重启后重放日志就能恢复。
- **undo log**：保证原子性和 MVCC。记录"改之前是什么样"，回滚时照着还原，同时给快照读提供历史版本。

## 示例代码

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;

-- 切换到读已提交（会话级别）
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 快照读演示
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1;  -- 读到 1000
-- 此时另一个事务把 balance 改成了 900 并提交了
SELECT balance FROM accounts WHERE id = 1;  -- RR 下仍然读到 1000（快照）
COMMIT;

-- 当前读（会加锁，读最新值）
START TRANSACTION;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;  -- 读到最新的 900，并锁住这行
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

## 面试追问

- **面试官可能会这样问你**：很多公司用读已提交而不是可重复读，为什么？（RR 下间隙锁更多，容易死锁；RC 下锁粒度更细，并发更好）
- **面试官可能会这样问你**：UPDATE 没命中索引会怎样？（可能锁全表！没有索引就没法精确定位行，InnoDB 会退化成表级锁）
- **面试官可能会这样问你**：redo log 和 binlog 的两阶段提交是什么？（为了保证两者逻辑一致——先 prepare redo，再写 binlog，最后 commit redo）
- **面试官可能会这样问你**：快照读就一定不会有幻读吗？（严格来说，先快照读再当前读可能出现"幻觉"，因为当前读会看到新版本）

## 常见误区

- **很多人会搞混的地方**：以为"可重复读"就完全没有幻读——要区分快照读和当前读，当前读路径上是靠间隙锁来防的。
- **很多人会搞混的地方**：认为 undo log 只用来回滚——它同时是 MVCC 版本链的基础，快照读就靠它找历史版本。
- **很多人会搞混的地方**：混淆"不可重复读"和"幻读"——前者是"同一行的内容变了"，后者是"结果集多出了新行"。
- **很多人会搞混的地方**：隔离级别只背表格不结合具体引擎——MySQL InnoDB 和 PostgreSQL 对同一个级别的实现细节是不一样的。
