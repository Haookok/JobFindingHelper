# 技术选型与决策

- **难度**: 进阶
- **分类**: 项目经验 / 系统设计
- **标签**: [技术选型, 架构演进, 对比, 决策记录, ADR]

## 核心概念

面试官问"你们为什么选这个技术"，他不是在考你背参数，而是在考你**做决策的思路**。好的选型叙述像一道证明题：**先列条件（约束和需求）→ 比较几个解法 → 说明为什么选这个 → 承认代价**。

核心话术结构：「当时约束是 **XX**；我对比了 **A 和 B**，在 **性能/运维成本/团队熟悉度** 上各有优劣；结合实际情况选了 **A**；上线后 **指标/效果** 验证了判断，同时我们接受了 **已知局限**，计划 **XX** 时候演进。」

如果你写过 ADR（架构决策记录）或 RFC 文档，面试时提一句"结论写进了文档方便后人复盘"，比炫技更加分。

## 详细解析

**常见选型对比（面试简版）**：

**MySQL vs PostgreSQL**：MySQL 生态广、运维成熟、云厂商支持好，适合简单 OLTP；PostgreSQL SQL 能力强（JSONB、GIS、复杂查询），分析型混合负载更友好。话术："选 MySQL 因为团队熟悉且托管成熟；如果业务强依赖复杂分析或扩展类型，会评估 PG。"

**Kafka vs RabbitMQ**：Kafka 高吞吐日志流、可回放、消费者组扩展好；RabbitMQ 灵活路由、低延迟任务。话术："需要持久化日志和大数据管道选 Kafka；需要复杂路由和低延迟任务选 RabbitMQ。"

**微服务 vs 单体**：这是最容易"装逼翻车"的话题。早期团队小、业务不确定，单体迭代快、调试简单。等到发布互相阻塞、扩展遇瓶颈时再拆微服务。话术："早期单体 + 清晰模块边界；当发布耦合成为痛点时再拆，配合网关、链路追踪和 CI/CD。"**忌为拆而拆**。

**REST vs gRPC**：对外或前后端之间常用 REST（调试方便、生态通用）；内部高频、强契约的服务间调用倾向 gRPC（高效二进制、强类型）。

**架构演进怎么讲**：用**时间线 + 触发条件**。比如：单机 → 读写分离（用户量涨了）→ 加缓存（DB 扛不住）→ 消息异步（峰值太高）→ 拆服务（团队扩大发布冲突）。每一步说明**为什么现在做**（而不是一开始就做），体现成本意识和渐进思维。

## 示例代码

**好的选型叙述**：

「当时日志量每天约 50GB、需要实时消费和回放能力（**约束**）。我对比了 Kafka 和 RabbitMQ（**候选**）：Kafka 的分区日志模型天然支持回放且吞吐更高，RabbitMQ 在路由灵活性上更好但不支持原生回放（**对比**）。结合团队有 Kafka 运维经验，选了 Kafka（**决策**）。上线后日均消费延迟 < 100ms，但分区扩容时需要手动迁移 consumer group（**代价**）。」

**差的选型叙述**：

「我们用了 Kafka 因为它高吞吐。」——没有约束、没有对比、没有代价，面试官追问一步就接不住。

## 面试追问

- **面试官可能会这样问你**：业务规模翻倍的话，你这个选型里哪一块会先扛不住？预案是什么？
- **面试官可能会这样问你**：如果重新选，你还会做同样的决定吗？有什么新信息可能改变判断？
- **面试官可能会这样问你**：选型落地后怎么验证（压测、灰度、故障演练）？失败了怎么回滚？

## 常见误区

- **很多人会搞混的地方**：只会背参数（"Kafka 吞吐百万级"）但说不出自己的业务 QPS 和延迟要求——脱离场景的参数没有意义。
- **很多人会搞混的地方**：对比只列优点不列代价——任何技术选择都有 trade-off，能说出代价才显得真实。
- **很多人会搞混的地方**：把"流行"当理由（"大厂都在用"）——面试官想听的是**为什么适合你的场景**。
- **很多人会搞混的地方**：团队不熟悉的技术硬上——选型要考虑团队学习曲线和维护能力，不然东西做出来没人能改。
- **很多人会搞混的地方**：被问"为什么不用 XX"就全盘否定——正确做法是先承认 XX 的优点，再说明当时约束下为什么不适合，最后给出条件化的回答（"如果团队规模和业务量到 XX，会重新评估"）。
