# JVM 内存模型与垃圾回收

- **难度**: 进阶
- **分类**: JVM / 内存 / GC
- **标签**: [堆, 栈, 方法区, GC Roots, CMS, G1, ZGC, STW]

## 核心概念

把 JVM 想象成一间**办公室**，不同区域有不同用途：

- **堆**（Heap）：公司的大仓库，所有 `new` 出来的对象都堆在这里。垃圾回收（GC）的主战场。
- **虚拟机栈**：每个员工（线程）自己的工位，放着当前干活用的局部变量、方法调用记录等。方法结束，工位就清空。
- **方法区/元空间**：公司的档案室，存类的定义、常量池等"蓝图"信息。JDK8 后叫 Metaspace，用的是操作系统的内存而不是堆。
- **程序计数器**：每个员工手里的待办清单指针——"我执行到哪一行代码了"。

**垃圾回收**就是"保洁阿姨定期检查仓库，没人用的东西扔掉腾空间"。怎么判断"没人用"？从所有**GC Roots**（栈上的引用、静态变量等）出发，能顺着引用链摸到的就是"有人用的"，摸不到的就是垃圾。

## 详细解析

### 分代假设——大部分对象"朝生夕死"

JVM 把堆分成**新生代**和**老年代**。大部分对象创建后很快就没用了（比如一个方法里的临时变量），这些在新生代就被回收了。少数"长寿"对象活过好几轮 GC 后，会被"晋升"到老年代。

新生代 GC（Minor GC）很快、很频繁；老年代 GC（Major/Full GC）慢、代价大，尽量少触发。

### 主流垃圾回收器——选谁？

- **Serial/Parallel**：简单粗暴，GC 时全公司停工（STW），但吞吐量高，适合后台批处理任务
- **CMS**：尽量边干活边清理（并发标记清除），停顿短，但会产生内存碎片。JDK9 后已废弃
- **G1**：把堆切成很多小块（Region），可以预测停顿时间。目前大部分 Java 应用的默认选择
- **ZGC**：目标是亚毫秒级停顿，堆再大停顿也很短。JDK17+ 生产可用

### OOM 常见类型

- `Java heap space`：堆满了——对象太多或内存泄漏
- `Metaspace`：类太多（动态生成大量类）
- `GC overhead limit exceeded`：GC 占了 98% 的时间只回收了 2% 的内存——基本是泄漏了
- `StackOverflowError`：递归太深，栈炸了——这个和堆 OOM 不是一回事，调 `-Xss` 而不是 `-Xmx`

### 引用类型——不是所有引用都一样"硬"

- **强引用**：正常的 `=` 赋值，GC 绝不回收
- **软引用**：内存不够时才回收，适合做缓存
- **弱引用**：下次 GC 就回收，`WeakHashMap` 用的就是这个
- **虚引用**：不能通过它获取对象，只是用来跟踪对象被回收的时机

## 示例代码

```java
public class GcRootDemo {
    public static void main(String[] args) {
        // big 是栈上的局部变量 → 指向堆上 1MB 数组 → 这是一个 GC Root 引用
        byte[] big = new byte[1024 * 1024];
        System.gc();               // 只是"建议" JVM 做 GC，不保证立刻执行
        System.out.println(big.length);  // big 还活着，不会被回收
        // 方法结束后 big 出栈，没有任何 GC Root 指向这个数组，下次 GC 就会回收它
    }
}
```

## 面试追问

- **面试官可能会这样问你**：三色标记法是什么？CMS 和 G1 在处理并发标记时有什么区别？（CMS 用增量更新，G1 用 SATB 原始快照）
- **面试官可能会这样问你**：为什么不推荐用 `finalize` 释放资源？（执行时机不确定，可能很晚才跑甚至不跑；应该用 try-with-resources 或 Cleaner）
- **面试官可能会这样问你**：G1 的 `MaxGCPauseMillis` 能保证停顿不超过这个值吗？（不能保证，只是"尽量目标"，实际还要看堆大小和分配速率）
- **面试官可能会这样问你**：方法区在 JDK8 后"进了堆"吗？（没有！元空间用的是本地内存，和堆是分开管理的）

## 常见误区

- **很多人会搞混的地方**：把"JVM 内存区域"（堆、栈、方法区）和"Java 内存模型 JMM"（happens-before、可见性）搞混——它们是完全不同的概念。
- **很多人会搞混的地方**：以为 `System.gc()` 会立刻触发 Full GC——只是建议，JVM 可以无视。
- **很多人会搞混的地方**：以为堆调得越大越好——堆太大会导致 Full GC 停顿时间更长。
- **很多人会搞混的地方**：StackOverflowError 和 OOM 用同一套调参——栈溢出调 `-Xss`（线程栈大小），堆溢出调 `-Xmx`，完全不同的旋钮。
