# RAG 检索增强生成

- **难度**: 进阶
- **分类**: AI / LLM
- **标签**: [RAG, 向量检索, Embedding, Chunk, 混合检索, GraphRAG, 评估]

## 核心概念

**RAG（Retrieval-Augmented Generation）** 在生成前从外部知识库检索相关片段，将检索结果作为 **上下文** 与用户问题一并送入 LLM，从而减轻幻觉、支持私域与实时知识更新。典型流水线：**文档摄取 → 切分与索引（常含向量化）→ 查询时检索 Top-K → 拼 prompt 生成答案**。与仅依赖参数记忆相比，RAG 将「事实」外置为可版本化的文档与索引。

落地时还要处理 **权限与多租户**：同一向量库需带 `tenant_id`、文档级 ACL，检索阶段 **先过滤再 ANN**，避免「搜到不该看的片段」。日志与审计上应记录 **命中片段 ID**，便于事后追溯。

## 详细解析

**索引阶段**：原始文档经清洗、**Chunk 切分**（固定长度、按段落/标题、滑动窗口重叠等），每块经 **Embedding 模型** 编码为稠密向量，写入 **向量数据库**。**检索阶段**：用户问题同样编码，用近似最近邻（ANN）取 Top-K；可叠加 **重排序（reranker）** 提升精度。

**向量数据库**：**FAISS**（Meta，库级 ANN，需自建存储与元数据）；**Milvus / Zilliz**（分布式、过滤、混合场景）；**Pinecone**（托管 SaaS，运维简单）。选型看规模、延迟 SLA、过滤需求与合规。

**Embedding 选择**：通用中文/英文可用 **bge、text-embedding-3、E5** 等；需与下游 LLM 语言域一致；多语言场景选多语模型。领域极专时可对比微调 embedding 或改用 **混合检索** 兜底。

**Chunk 策略**：过小丢上下文、过大噪声多；常见 **256–512 token** 重叠 **10%–20%**；结构化文档可按章节边界切；表格/代码需保留完整性。可 **父子块**：小块检索、大块喂给模型。

**混合检索**：**稀疏（BM25/关键词）** 擅长精确词与罕见实体；**稠密向量** 擅长语义近义。线性加权、RRF（倒数排名融合）或两阶段（一路召回一路精排）可兼顾。

**GraphRAG**：将实体与关系构图，结合社区摘要或子图检索，改善多跳推理与全局问题；代价是构建与维护图的成本更高。

**RAG vs 微调**：RAG 适合**频繁变更事实**、**长尾知识**、**可解释引用**；微调适合**风格、格式、领域术语习惯**与**任务形态**固化。生产上常 **RAG + 轻量 SFT** 并用。

**评估**：**Faithfulness（忠实度）**：答案是否被上下文支持，可用 NLI、LLM-as-judge。**Relevance（相关性）**：检索块与 query 的匹配；**Answer correctness** 与金标准对比。离线可建 QA 对；在线关注点击率、人工抽检与拒答率。

**进阶范式（了解）**：**Self-RAG** 让模型自判是否需要检索、片段是否有用；**Corrective-RAG** 在检索质量差时触发改写或网络搜索等纠偏路径。面试提到「动态检索决策」即可，不必展开全部训练细节。

**Prompt 侧**：需规定 **仅依据上下文作答**、**引用编号**、**无法回答则拒答**，并与 **温度** 配合：事实问答倾向低温；创意写作可放宽但应区分场景。**上下文窗口** 有限时，可对 Top-K 再 **压缩摘要** 或 **两阶段回答**（先列要点再综合）。

**多路召回**：除向量与 BM25 外，可 **同义词扩展**、**实体链接**、**FAQ 精确匹配** 并行，再在 rerank 阶段统一排序；工业流水线常见「召回宽、排序严」。

**运维**：索引需 **增量更新**（文档变更、删除向量）、**版本回滚** 与 **脏数据监控**（OCR 乱码、HTML 导航栏污染），否则检索质量缓慢劣化。

**Query 侧增强**：**多查询融合**（一个问题拆成多个子问分别检索再合并）、**Step-back prompting**（先抽象再检索）可提升难问召回；代价是 **延迟与成本** 上升，应配置开关。**术语归一**：产品内部代号与对外名称并存时，可维护 **同义词表** 做查询扩展。

**表格与结构化数据**：纯向量切分易丢表头语义；可 **行级+表摘要** 双索引，或保留 Markdown/HTML 结构再切。**图片 PDF**：先 OCR 再入索引，错误文本会直接污染检索，需置信度过滤。

**成本估算（口述）**：embedding 调用次数 ≈ 文档块数（离线）+ QPS（在线）；大 K + 交叉编码器重排会使 **每次查询算力** 接近一次小模型推理，应对 **缓存热门 query**。

**与 Tool/Agent**：复杂问题可「先检索再让模型决定是否再检」，避免一次塞满上下文；Agent 可把检索当工具，按轮次缩小范围。

**Citation 与可追溯**：回答中插入 `[1][2]` 与片段 id，前端可 **hover 查看原文**；利于人工审核与 **纠纷举证**。**多语言文档**：同一实体多语表述需 **统一 canonical id**，否则检索分散。

**冷启动**：新库文档少时，可先 **LLM 生成 FAQ** 再人工校对入库，避免空检索；但生成片段必须 **标记为未审核** 防误用。

## 示例代码

```python
import torch
import torch.nn.functional as F

# 教学：用随机向量模拟文档块与 query 的相似度检索（实际应调用真实 Embedding API）
def mock_embed(texts: list[str], dim: int = 64) -> torch.Tensor:
    # 真实项目请替换为 sentence-transformers 或 OpenAI embeddings
    g = torch.Generator().manual_seed(42)
    return F.normalize(torch.randn(len(texts), dim, generator=g), dim=-1)


def retrieve_top_k(query: str, chunks: list[str], k: int = 3):
    emb_q = mock_embed([query])
    emb_d = mock_embed(chunks)
    scores = (emb_q @ emb_d.T).squeeze(0)  # 余弦相似度（已 L2 normalize）
    topk = torch.topk(scores, k=min(k, len(chunks)))
    return [(chunks[i], float(scores[i])) for i in topk.indices.tolist()]


chunks = [
    "公司年假规则：满一年 5 天。",
    "报销需在 30 日内提交系统。",
    "VPN 使用需申请令牌。",
]
query = "入职多久有年假？"
for text, s in retrieve_top_k(query, chunks):
    print(f"{s:.3f} | {text}")
```

**BM25 思路（口述）**：词频与逆文档频率加权，短文档长度归一；与向量分数融合时常需 **归一化到可比尺度** 再 RRF。

## 面试追问

- **追问 1**：为什么 RAG 仍可能幻觉？可从检索失败、chunk 边界切断语义、模型忽略上下文几方面说明。
- **追问 2**：RRF 与简单分数加权各适用什么场景？BM25 在哪些 query 上明显优于纯向量？
- **追问 3**：GraphRAG 相比扁平向量 RAG 多解决了哪类问题？引入的主要工程与成本是什么？
- **追问 4**：如何设计 **HyDE（假设文档嵌入）** 或 **query 改写** 提升检索召回？有什么副作用？
- **追问 5**：Faithfulness 自动评估有哪些主流做法？各有什么偏差（如 NLI 训练域不匹配）？

## 常见误区

- 认为「上了向量库就不会胡编」——检索不到或模型不服从上下文时仍会幻觉，需拒答策略与引用约束。
- **Chunk 越大越好**——噪声淹没相关句，且占用上下文窗口；需在召回与精读间平衡。
- 忽略 **元数据过滤**（时间、部门、权限）——企业场景常比「纯语义 Top-K」更关键。
- 把 **RAG** 当成 **微调替代品** 做所有事——形态互补，而非二选一。
- 评估只看 **BLEU/ROUGE**——对事实性任务不够，需检索命中率与忠实度指标。
- **Embedding 与生成模型「语言不一致」**（如英文向量 + 中文 LLM）导致召回差——应统一语种或做多语对齐。
- 检索结果 **不做去重与矛盾检测**——Top-K 可能含重复段落或相互冲突条款，模型易被误导。
- 忽略 **延迟预算**：rerank 大交叉编码器、过大 K 会拖慢首 token，需分层服务与缓存 query 向量。
