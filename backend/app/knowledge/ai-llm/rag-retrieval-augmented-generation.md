# RAG 检索增强生成

- **难度**: 进阶
- **分类**: AI / LLM
- **标签**: [RAG, 向量检索, Embedding, Chunk, 混合检索, GraphRAG, 评估]

## 核心概念

**RAG** 的全名是 Retrieval-Augmented Generation（检索增强生成），用大白话说就是——**让 AI 做"开卷考试"**。

你想啊，大模型平时回答问题靠的是训练时"背"下来的知识，就像闭卷考试全凭记忆。问题是它的"记忆"有时候不准确（会编），也不够新（训练数据有截止日期）。RAG 的做法就是：**在模型回答之前，先帮它去知识库里翻资料，把找到的相关内容一起塞给它看，然后再让它基于这些资料回答**。

整个流程就四步：**灌资料 → 建索引 → 查资料 → 生成答案**。就像考试前先把参考书做好标签、建好目录，考场上遇到题先翻书找到相关段落，再组织语言写答案。

## 详细解析

### 第一步：灌资料 & 切块（Chunk）

把原始文档（PDF、网页、数据库记录等）切成小块。为啥要切？因为模型的上下文窗口有限（就像你考试时桌子就那么大，只能放几页纸），塞太长的原文放不下，也会淹没关键信息。

切块有讲究：**太小**会丢上下文（"第二季度"切断了变成"季度"，意义全变了）；**太大**会掺杂无关内容。常见做法是 256-512 个 token 一块，相邻块重叠 10%-20%（就像拍照时两张照片有重叠区域，拼接时不会断裂）。表格和代码要保持完整性，别从中间切断。

### 第二步：向量化 & 建索引

每个小块经过一个 **Embedding 模型** 变成一个向量（一串数字），存进 **向量数据库**。你可以理解为：给每个段落算了个"语义指纹"，意思相近的段落指纹也相近。常见向量库有 FAISS（本地、快但需自己管理）、Milvus（分布式、功能全）、Pinecone（云服务、省心）。

### 第三步：检索（Retrieval）

用户提问时，同样把问题变成向量，然后去向量库里找"指纹最像的"前 K 个段落。还可以叠加 **Reranker（重排序模型）**——就像初选找了 20 个候选人，再用更精细的标准从里面选前 5。

**混合检索** 是实战中的常见策略：**向量检索**擅长理解语义（"年假"和"带薪休假"能匹配上），**BM25 关键词检索**擅长精确匹配（产品编号、人名等）。两路一起上，再用 RRF（倒数排名融合）合并结果，兼顾语义和精确度。

### 第四步：生成答案

把检索到的段落和用户问题一起塞进 Prompt，让模型"看着资料回答"。Prompt 里通常要明确告诉模型：**只能根据给定资料作答、必须标注引用来源、不确定就说不知道**。

### RAG vs 微调：啥时候用哪个？

| 场景 | 用 RAG | 用微调 |
|------|--------|--------|
| 知识频繁更新（公司制度、新闻） | ✅ 换文档就行 | ❌ 要重新训练 |
| 需要引用来源、可追溯 | ✅ 天然带来源 | ❌ 知识融在参数里 |
| 改变模型的说话风格/格式 | ❌ 改不了模型本身 | ✅ 从根上改变行为 |
| 领域专业术语习惯 | 有限帮助 | ✅ 更自然 |

实际项目中，**RAG + 轻量微调** 并用是最常见的做法。

### GraphRAG 和进阶玩法

**GraphRAG** 是把文档里的实体和关系建成知识图谱，适合"多跳推理"（比如"张三的老板的老板是谁"这种需要跳几步才能回答的问题）。代价是建图和维护成本高。

**Self-RAG** 让模型自己判断"这个问题需不需要查资料"，**Corrective-RAG** 在发现检索质量差时自动改写查询重试。面试时提到"动态检索决策"即可。

## 示例代码

```python
import torch
import torch.nn.functional as F


def mock_embed(texts: list[str], dim: int = 64) -> torch.Tensor:
    """模拟 Embedding（实际项目请用 sentence-transformers 或 OpenAI API）"""
    g = torch.Generator().manual_seed(42)
    return F.normalize(torch.randn(len(texts), dim, generator=g), dim=-1)


def retrieve_top_k(query: str, chunks: list[str], k: int = 3):
    """从文档块中检索与 query 最相似的 Top-K 个"""
    emb_q = mock_embed([query])
    emb_d = mock_embed(chunks)
    # 因为已经 L2 归一化了，内积就等于余弦相似度
    scores = (emb_q @ emb_d.T).squeeze(0)
    topk = torch.topk(scores, k=min(k, len(chunks)))
    return [(chunks[i], float(scores[i])) for i in topk.indices.tolist()]


# 模拟一个公司内部知识库
chunks = [
    "公司年假规则：满一年 5 天。",
    "报销需在 30 日内提交系统。",
    "VPN 使用需申请令牌。",
]
query = "入职多久有年假？"
for text, s in retrieve_top_k(query, chunks):
    print(f"{s:.3f} | {text}")
```

**运行说明**：这里用随机向量模拟 Embedding。真实项目中换成 `sentence-transformers` 或 OpenAI Embedding API，相似度就有实际语义意义了。

## 面试追问

- **面试官可能会这样问你**：RAG 了还会胡编吗？会的话是哪些环节出了问题？（提示：检索没找到、切块把关键信息切断了、模型不听话不看上下文）
- **面试官可能会这样问你**：BM25 在哪些查询上明显比纯向量检索好？两者怎么融合？
- **面试官可能会这样问你**：GraphRAG 比普通向量 RAG 多解决了什么问题？付出了什么代价？
- **面试官可能会这样问你**：HyDE（先让模型猜一个假答案再用假答案去检索）这个思路有啥好处和副作用？
- **面试官可能会这样问你**：怎么自动评估 RAG 的回答质量？常用的"忠实度"指标怎么算？

## 常见误区

- **很多人会搞混的地方**：以为"上了向量库就不会胡编"——检索没命中或模型不服从上下文时照样幻觉，必须加拒答策略。
- **很多人会搞混的地方**：觉得 Chunk 越大越好——大块里噪声多，真正相关的句子被淹没了，还白白浪费上下文窗口。
- **很多人会搞混的地方**：只做语义检索不做元数据过滤——企业场景里"这个文档你有没有权限看"往往比"语义最相似"更重要。
- **很多人会搞混的地方**：把 RAG 当成微调的完全替代品——两者是互补关系，不是二选一。
- **很多人会搞混的地方**：Embedding 模型和生成模型语言不一致——用英文 Embedding 配中文大模型，召回效果会大打折扣。
- **很多人会搞混的地方**：只看 BLEU/ROUGE 评估 RAG 效果——这些指标衡量文本相似度，不衡量事实正确性，得用忠实度和检索命中率来评。
