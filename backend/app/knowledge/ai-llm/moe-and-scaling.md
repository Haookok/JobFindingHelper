# MoE 与模型扩展

- **难度**: 深入
- **分类**: AI / LLM
- **标签**: [MoE, 门控, 负载均衡, DeepSeek, Scaling Laws, 涌现, Dense]

## 核心概念

**MoE（Mixture of Experts，混合专家模型）** 的核心思想就像**医院的专家会诊制度**。

一个普通的 Dense 模型就像一个全科医生——不管你什么病，都是同一个医生从头看到尾。模型的每一个参数对每一个输入都参与计算。

MoE 不一样，它更像一家大医院：里面有心内科、骨科、眼科等很多科室（**专家网络**），门口有个**分诊台（Router/门控）**。每来一个病人（token），分诊台先看看这人什么情况，然后只分给最对口的 1-2 个科室处理，其他科室的医生该喝茶喝茶。

这样做的好处巨大：**医院可以雇很多医生（总参数量大 = 知识容量大），但每个病人只需要付 1-2 个医生的挂号费（每个 token 激活的参数少 = 计算成本低）**。用稀疏激活换大容量，这就是 MoE 的核心交易。

## 详细解析

### 门控（Router）：谁来分诊？

Router 就是一个简单的线性层，看了 token 的表示之后输出每个专家的概率分数，然后选 Top-1 或 Top-2 个专家处理这个 token。

但这里有个大问题——**负载均衡**。想象一下，如果所有病人都只想找"最有名的那个专家"，那其他专家就闲着了，这个名医被累死，整个医院的资源利用率极低。这就是**专家坍塌**——少数专家包揽一切，其余形同虚设。

解决方案：训练时加一个**辅助损失（Auxiliary Loss）**，专门鼓励各专家分配均匀。还可以设**容量因子**——每个专家每次最多接待多少 token，满了就分流到其他专家。

### DeepSeek-MoE：面试热门话题

DeepSeek 在 MoE 上做了几个很有意思的改进：
- **细粒度专家划分**：把大专家切成更多小专家，路由更灵活
- **共享专家（Shared Experts）**：一部分专家每个 token 都会过，负责学通用知识（就像医院的基础体检，每个人都做）；路由选择的专家负责学专门知识
- **改进的路由策略**：提升专家利用率，减少浪费

面试时重点说"共享专家 + 细粒度路由 + 均衡策略"这三个关键词就够了。

### Scaling Laws：不是越大越好，是要"搭配得当"

**Chinchilla 缩放定律** 告诉我们一个反直觉的结论：给定固定的算力预算，**模型大小和训练数据量要协调增长**。

以前大家拼命堆参数但训练数据不够，结果就是"大而欠训"——就像你买了一辆法拉利但只加了半箱油，跑不出真正的性能。Chinchilla 的建议是：与其训一个 300B 参数但只吃 300B token 的模型，不如训一个 70B 参数但吃 1.4T token 的模型，效果反而更好。

MoE 让这个问题更有意思：因为总参数量和激活参数量解耦了，你可以有很大的"知识容量"但每步的计算成本不高。不过训练数据量和路由质量仍然是上限。

### 涌现能力（Emergent Abilities）

一个有趣的现象：模型规模到一定程度后，某些能力会"突然出现"——比如小模型完全不会做某类推理题，参数翻几倍后突然会了。

**但注意**：这个说法有争议。有研究指出"涌现"可能是评估指标的非线性导致的假象（换个更连续的指标看就没那么"突然"了）。面试时保持客观："与规模、数据质量和任务评估方式都有关，不宜简单归因为'参数大了就会涌现'"。

### MoE vs Dense：各有千秋

| 维度 | MoE | Dense |
|------|-----|-------|
| 知识容量 | 大（总参数多） | 受限于计算预算 |
| 每 token 计算量 | 小（稀疏激活） | 全部参数都参与 |
| 实现复杂度 | 高（路由、通信、负载均衡） | 低 |
| 推理延迟稳定性 | 取决于路由分配 | 非常稳定 |
| 显存占用 | 虽然激活少但全部专家权重都要放内存 | 与激活成正比 |

**什么时候 Dense 反而更好？** 极小 batch、要求延迟极稳定、边缘设备资源有限、或者做研究需要简单基线的时候。

### 通信开销：多卡分专家的代价

大规模 MoE 需要把不同专家放在不同 GPU 上（专家并行，EP）。每个 token 算完路由后要通过 **All-to-All 通信**发送到对应 GPU——这个通信开销在小 batch 时占比很高，可能把稀疏计算省下来的时间全吃回去。

## 示例代码

```python
import torch
import torch.nn as nn
import torch.nn.functional as F


class MoELayer(nn.Module):
    """Top-1 路由 MoE 层：每个 token 只选一个专家处理"""

    def __init__(self, d_model: int, d_ff: int, num_experts: int):
        super().__init__()
        self.num_experts = num_experts
        # 门控：看了 token 表示后，给每个专家打分
        self.gate = nn.Linear(d_model, num_experts, bias=False)
        # 每个专家就是一个普通的 FFN
        self.experts = nn.ModuleList(
            [
                nn.Sequential(
                    nn.Linear(d_model, d_ff),
                    nn.GELU(),
                    nn.Linear(d_ff, d_model),
                )
                for _ in range(num_experts)
            ]
        )

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        b, s, d = x.shape
        flat = x.view(-1, d)  # 展平成 [总token数, 维度]
        logits = self.gate(flat)  # 门控打分
        probs = F.softmax(logits, dim=-1)
        idx = torch.argmax(probs, dim=-1)  # 每个 token 选分数最高的专家
        out = torch.zeros_like(flat)
        # 把 token 分发给各自的专家处理
        for e in range(self.num_experts):
            mask = idx == e
            if mask.any():
                out[mask] = self.experts[e](flat[mask])
        return out.view(b, s, d)
```

**运行说明**：需安装 `torch`。这是教学版本，真实系统还需要加上辅助损失（负载均衡）、容量因子、批量专家计算优化（按专家分组而不是循环）。

## 常见误区

- **很多人会搞混的地方**：拿 MoE 的总参数量去估算算力——应该看每个 token 激活了多少参数、实际 FLOPs 是多少，总参数量只代表知识容量。
- **很多人会搞混的地方**：忽略 All-to-All 通信和负载倾斜——论文上的理论加速比，到了真实集群上可能被通信和不均衡吃掉大半。
- **很多人会搞混的地方**：把 DeepSeek-MoE 说成"只是专家多了"——共享专家和改进的路由设计才是关键创新。
- **很多人会搞混的地方**：把 Scaling Law 背成"越大越好"——核心结论是"在给定算力下，参数量和数据量要找最优配比"，不是无脑堆参数。
- **很多人会搞混的地方**：把"涌现"描述成 AI 有了"意识"——面试时请保持工程化表述："特定能力随规模和数据量的增长出现非线性跃升"。
- **很多人会搞混的地方**：觉得专家数越多越好——专家太多路由更难学、通信更重、边际收益递减，需要和总预算平衡。
- **很多人会搞混的地方**：混淆"总 FLOPs"和"每 token 激活 FLOPs"——比较 MoE 和 Dense 时必须对齐这个指标再谈性价比。
