# LLM Agent 智能体

- **难度**: 进阶
- **分类**: AI / LLM
- **标签**: [Agent, ReAct, Tool Use, LangGraph, MCP, Multi-Agent, 记忆]

## 核心概念

**LLM Agent** 指以大模型为「大脑」，在环境中 **感知（读入观察/用户输入）→ 规划（分解目标）→ 执行（调用工具或写代码）→ 反思（根据反馈调整）** 的闭环系统。与单次问答不同，Agent 强调 **多步交互**、**外部能力扩展**（API、数据库、浏览器）与 **状态延续**。工业界常通过 **Function Calling / Tool Use** 将结构化动作暴露给模型，由运行时解析 JSON 并执行再回填观察。

评估 Agent 质量时，除最终答案外，可看 **工具调用正确率**、**步数分布**、**失败重试成功率**；线上需 **灰度** 与 **回滚**，避免新工具 schema 引发大面积误调用。

## 详细解析

**核心循环**：每轮读取当前目标与历史（含工具返回），模型输出「思考/计划」或「工具调用」；执行器运行工具并将 **Observation** 追加到上下文；直到给出 **Final Answer** 或达步数上限。需 **停止条件**、**权限沙箱** 与 **超时/重试** 防止死循环与危险操作。

**范式对比**：**ReAct**：交错 Reasoning + Acting，可读轨迹、利于调试。**Reflexion**：失败后生成 **文字反思** 写入下一轮 prompt，强化纠错。**Plan-and-Solve**：先出完整计划再逐步执行，适合依赖顺序强的任务；可与 ReAct 混合（先 plan 再 react）。

**Tool Use / Function Calling**：OpenAI 兼容接口中模型返回 `tool_calls`；框架侧注册 schema（名称、参数 JSON Schema），执行后把结果以 `role=tool` 消息写回。关键是 **描述清晰**、**参数可校验**、**错误信息可消费**。

**LangChain**：链、Agent、工具抽象丰富，上手快；**LangGraph**：基于 **有向图/状态机** 建模多节点与条件边，适合可控流程、人机协同与循环。选型：快速原型用 LangChain；复杂状态与审计用 LangGraph。

**MCP（Model Context Protocol）**：标准化 **上下文与工具** 的客户端-服务器协议，使模型可通过统一方式连接数据源与工具服务，降低「每个应用各写一套适配」的成本；与具体云厂商 API 解耦思路一致。

**Multi-Agent**：角色分工（规划者、执行者、评审者）、**对话协商**或 **黑板结构**；需注意 **循环对话成本**、**责任归属** 与 **全局目标对齐**。简单任务单 Agent + 好工具往往更稳。

**记忆机制**：**短期**：当前会话的滑动窗口、摘要压缩（如达到 token 阈值时对历史做 LLM 摘要）。**长期**：向量库检索相关片段、结构化记忆表、用户画像键值；写入需 **冲突解决** 与 **过期策略**。避免无界堆叠导致噪声与成本爆炸。

**人在回路（HITL）**：高风险操作（支付、删库、对外发信）前 **暂停并请求确认**；LangGraph 等可用 **中断节点** 实现人工审批。**护栏**：输入输出侧 **策略模型、正则、敏感词、PII 脱敏** 与 **工具白名单** 并列，不能单靠「模型自觉」。

**观测与调试**：OpenTelemetry 风格 trace、逐步保存 messages、对工具入参出参 **结构化日志**（注意脱敏），便于复现「哪一步开始跑偏」。**成本**：每多一轮对话多一次 LLM 调用，需 **最大轮数** 与 **预算上限**。

**与 RAG 的关系**：Agent 可把 **检索器** 当作一个 Tool，实现「需要时再查知识库」；与「先 RAG 再一次性回答」相比，更适合多跳取证，但路径更长、失败点更多。

**状态建模**：除 messages 列表外，可维护 **结构化 state**（当前子目标、已确认事实、待办列表），每轮由程序合并进 prompt，比纯自然语言堆叠更 **可控**。**终止条件**：除 Final Answer 外，可设「连续 N 次无新信息」「工具返回重复错误」等熔断。

**安全与提示注入**：工具返回内容可能含 **恶意指令**（「忽略上文」），应对工具输出做 **隔离模板**（明确标注不可信来源），并对 **可执行代码类工具** 强沙箱。**幂等性**：写操作工具应设计 **幂等键** 或确认流，避免模型重复调用造成双写。

**评估基准**：工具使用类可看 **AgentBench、SWE-bench** 等方向（了解即可）；线上更看 **任务完成率、人工接管率、平均步数**。

**与 Workflow**：固定流程（如审批链）用 **确定性状态机** 优于完全自由 Agent；Agent 适合 **探索空间大** 的任务。LangGraph 的价值在于把二者 **嵌套**：主流程固定、子步骤 ReAct。

**OpenAI Assistants vs 自研**：托管 Assistants 提供线程与文件检索；自研需补齐 **持久化、重试、幂等、观测**。**Streaming**：工具调用需 **边生成边解析** partial JSON 时要谨慎，常等完整 tool_calls 再执行以免半残参数。

**版本治理**：系统 prompt、工具列表、模型版本应 **一并版本化**；否则线上难以复现用户 case。

**多模态 Agent（了解）**：工具可返回图片/音频引用，模型再决定后续 **看图 API** 或 **OCR**；核心是 **统一消息 schema** 与 **大上下文预算**。

**工具错误契约**：统一返回 `{ok, data, error_code, message}`，模型易解析；避免纯堆栈文本淹没上下文。**限流**：每用户/每会话工具 QPS 限制，防刷接口与 **成本失控**。

**审计**：金融、医疗等场景需 **完整审计轨迹**（谁、何时、调用了什么工具、返回摘要），满足内控要求。

## 示例代码

```python
from typing import Any, Callable

# 极简 ReAct 风格伪代码：一轮工具调用（无真实 LLM，仅演示控制流）
def run_agent_step(user_msg: str, tools: dict[str, Callable[..., Any]]):
    # 真实场景：调用 chat.completions + tools 参数，解析 tool_calls
    planned = "get_weather"
    args = {"city": "上海"}
    if planned in tools:
        obs = tools[planned](**args)
        return f"Observation: {obs}"
    return "Unknown tool"


def get_weather(city: str) -> str:
    return f"{city} 今日晴（stub）"


print(run_agent_step("上海天气", {"get_weather": get_weather}))
```

生产中还应对工具结果做 **长度截断**、**敏感信息过滤**，并记录 **trace id** 便于排障。

```python
# 参数校验示例：防止模型传入非字符串或超长 city
def get_weather_safe(city: str, max_len: int = 32) -> str:
    if not isinstance(city, str) or not city.strip():
        raise ValueError("invalid city")
    city = city.strip()[:max_len]
    return f"{city} 今日晴（stub）"
```

## 面试追问

- **追问 1**：ReAct 与纯 CoT（不出工具调用）在 **错误恢复** 与 **可观测性** 上差在哪里？
- **追问 2**：Function Calling 若模型 **虚构参数** 或 **调用不存在工具**，框架层应如何防护与反馈？
- **追问 3**：LangGraph 相对「手写 while 循环 + 消息列表」的核心收益是什么？什么情况下反而过度设计？
- **追问 4**：Multi-Agent 何时比单 Agent 更差（例如协调开销、目标漂移）？
- **追问 5**：长期记忆用向量检索时，如何避免 **陈旧事实** 与 **用户偏好矛盾**？可提版本号、时间戳、冲突检测。

## 常见误区

- 把 **Agent** 等同于 **Prompt 很长**——核心是 **闭环与工具接地**，非单纯堆字数。
- **工具越多越好**——schema 噪声大时模型选错工具概率上升，应 **分层与合并**。
- 忽略 **token 与费用**：多轮 ReAct 上下文线性增长，需摘要或结构化状态裁剪。
- **Multi-Agent 默认更智能**——常增加不稳定与调试难度，应证明任务需角色分离。
- 未区分 **MCP** 与某一具体插件格式——MCP 是协议层，不是某个厂商专属 SDK。
- **把「能调用工具」当成安全边界**——工具本身可能泄露密钥或执行危险命令，必须 **最小权限** 与 **沙箱**。
- **无超时与并发上限** 导致工具拖死整个请求——HTTP 客户端要设 timeout，线程池/队列限制并发。
