# 数据库事务与隔离级别

- **难度**: 进阶
- **分类**: 数据库 / 事务
- **标签**: [ACID, 隔离级别, MVCC, 锁, InnoDB]

## 核心概念

**事务就是银行转账的安全保证**。你从 A 账户转 100 块到 B 账户，这包含两步：A 减 100、B 加 100。事务保证这两步**要么都成功，要么都不做**——不会出现 A 扣了钱但 B 没收到的情况。

这就是 **ACID**：**A**tomicity（原子性，全成全败）、**C**onsistency（一致性，钱的总数不变）、**I**solation（隔离性，多个转账同时进行互不干扰）、**D**urability（持久性，转完账就算断电也不会丢）。

## 详细解析

### 并发事务会出什么问题？

想象银行只有一个柜员（数据库），但有很多客户同时办业务。如果不加管理，就会出三种"事故"：

- **脏读**：你看了隔壁正在办理中（还没确认）的转账金额，结果人家撤销了——你读到的是"假数据"
- **不可重复读**：你查了一次余额是 1000，过一会儿再查变成 900 了——因为中间有人转走了钱。同一个事务里，两次读的结果不一样
- **幻读**：你查"今天有几笔转账"，第一次查是 10 笔，第二次查变成 11 笔了——因为有人新增了一笔。重点是"多出来了新行"

### 四种隔离级别——从松到严

| 级别 | 脏读 | 不可重复读 | 幻读 | 好比... |
|------|------|-----------|------|---------|
| 读未提交 | 会 | 会 | 会 | 什么都不管，跟"裸奔"一样 |
| 读已提交 | 不会 | 会 | 会 | 只看最终确认的结果，但每次可能不同 |
| 可重复读 | 不会 | 不会 | InnoDB 大部分场景也不会 | MySQL 默认级别，事务开始时"拍个快照" |
| 串行化 | 不会 | 不会 | 不会 | 所有人排队一个个来，最安全但最慢 |

### MVCC——"时光快照机"

InnoDB 怎么做到"可重复读"的？用 **MVCC（多版本并发控制）**。打个比方：你在事务开始的那一刻"拍了一张照片"，之后不管别人怎么改数据，你看到的都是照片里的样子。

技术上，每行数据都有个版本链（通过 undo log 串起来），每个事务带着一个"可见性规则"（Read View），决定能看到哪个版本。

- **普通 SELECT**（快照读）：看"照片"，不加锁
- **SELECT ... FOR UPDATE**（当前读）：看最新版本，要加锁

### 锁——InnoDB 怎么防幻读？

在"当前读"的场景下，InnoDB 用了一种叫 **Next-Key Lock** 的锁（行锁 + 间隙锁），不仅锁住已有的行，还锁住行之间的"缝隙"，防止别的事务在中间插入新行。

### redo log 和 undo log——怎么保证不丢数据？

- **redo log**：保证持久性。事务提交时先把"改了什么"写到日志里，即使断电，重启后重放日志就能恢复。
- **undo log**：保证原子性和 MVCC。记录"改之前是什么样"，回滚时照着还原，同时给快照读提供历史版本。

## 示例代码

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;

-- 切换到读已提交（会话级别）
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 快照读演示
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1;  -- 读到 1000
-- 此时另一个事务把 balance 改成了 900 并提交了
SELECT balance FROM accounts WHERE id = 1;  -- RR 下仍然读到 1000（快照）
COMMIT;

-- 当前读（会加锁，读最新值）
START TRANSACTION;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;  -- 读到最新的 900，并锁住这行
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

## 面试追问

- **面试官可能会这样问你**：很多公司用读已提交而不是可重复读，为什么？（RR 下间隙锁更多，容易死锁；RC 下锁粒度更细，并发更好）
- **面试官可能会这样问你**：UPDATE 没命中索引会怎样？（可能锁全表！没有索引就没法精确定位行，InnoDB 会退化成表级锁）
- **面试官可能会这样问你**：redo log 和 binlog 的两阶段提交是什么？（为了保证两者逻辑一致——先 prepare redo，再写 binlog，最后 commit redo）
- **面试官可能会这样问你**：快照读就一定不会有幻读吗？（严格来说，先快照读再当前读可能出现"幻觉"，因为当前读会看到新版本）

## 常见误区

- **很多人会搞混的地方**：以为"可重复读"就完全没有幻读——要区分快照读和当前读，当前读路径上是靠间隙锁来防的。
- **很多人会搞混的地方**：认为 undo log 只用来回滚——它同时是 MVCC 版本链的基础，快照读就靠它找历史版本。
- **很多人会搞混的地方**：混淆"不可重复读"和"幻读"——前者是"同一行的内容变了"，后者是"结果集多出了新行"。
- **很多人会搞混的地方**：隔离级别只背表格不结合具体引擎——MySQL InnoDB 和 PostgreSQL 对同一个级别的实现细节是不一样的。
