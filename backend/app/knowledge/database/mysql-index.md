# MySQL 索引原理

- **难度**: 进阶
- **分类**: 数据库 / MySQL
- **标签**: [索引, B+树, 执行计划, InnoDB]

## 核心概念

InnoDB 默认使用 **B+ 树** 作为索引的物理结构：所有数据（或索引键）存放在叶子节点，叶子之间用指针串联成有序链表，便于范围扫描；非叶子节点只存键与指向子页的指针，使树更矮、减少磁盘 I/O 次。

**聚簇索引** 的叶子节点存整行数据（表通常以主键为聚簇索引）；**二级索引（非聚簇）** 叶子存「索引列 + 主键」，回表时再按主键查聚簇索引。**覆盖索引** 指查询所需列都能从某棵索引树上取得，无需回表。

## 详细解析

### B+ 树为何适合磁盘

- 多路平衡树，层高低，一次页读取可比较多个键。
- 叶子有序且链表相连，`ORDER BY`、范围条件可走索引顺序扫描。
- 与 B 树相比，B+ 非叶子不存数据页指针以外的行数据，同样节点大小下扇出更大。

### 聚簇 vs 非聚簇

- 一张 InnoDB 表 **只有一个聚簇索引**（无主键则用唯一非空列，再否则隐式 6 字节 row_id）。
- 二级索引查找：先定位叶子得到主键，再 **回表** 读聚簇叶子；若 `SELECT` 列全在二级索引里，则构成覆盖，避免回表。

### 最左前缀

联合索引 `(a,b,c)` 等价于先按 `a`、再 `a,b`、再 `a,b,c` 有序。条件必须从 **最左列** 起连续匹配才能充分用上索引；中间断档则后面列无法用于 **索引范围**，可能只剩前缀或完全不走该联合索引。

### 索引失效的常见场景（典型）

- 对索引列做函数/隐式类型转换/运算，导致无法使用索引列比较。
- 前导 `%` 的 `LIKE`（如 `'%abc'`）通常无法走 B+ 树前缀匹配。
- `OR` 两侧条件无法合并为同一索引路径时，优化器可能放弃索引。
- 优化器认为 **全表扫描更便宜**（小表、选择性差、统计信息偏差）。

### EXPLAIN 关注点（面试常问列）

- `type`：`system/const/eq_ref/ref/range/index/ALL`，从左到右一般越来越差；`ALL` 全表扫描需警惕。
- `key`：实际使用的索引；`possible_keys` 为候选。
- `rows`：预估扫描行数（估算值）。
- `filtered`：按条件过滤后剩余比例估计，可与 `rows` 一起看是否「扫很多、留很少」。
- `Extra`：`Using index` 常表示覆盖；`Using filesort`、`Using temporary` 可能带来额外成本；`Using where` 表示在存储引擎返回后再过滤。
- `ref` / `rows` 与 **索引选择性** 相关：重复值多的列单独建索引收益小，常需联合索引或调整查询。

### 索引设计小结（面试加分）

- 高频 **等值 + 范围** 条件：等值列放联合索引左侧，范围列放后（其右侧列往往无法再走范围）。
- 排序字段若与 `WHERE` 可共用同一索引，有机会 **避免 filesort**。
- 长字符串可用 **前缀索引**，代价是无法覆盖完整列做 `ORDER BY`/`GROUP BY` 的精确优化，需权衡。

## 示例代码

```sql
-- 联合索引：注意条件顺序与覆盖
CREATE INDEX idx_user_time ON orders (user_id, created_at);

-- 可走索引：最左列 + 范围
EXPLAIN SELECT * FROM orders
WHERE user_id = 1 AND created_at >= '2024-01-01';

-- 覆盖索引示例（无需回表）
EXPLAIN SELECT user_id, created_at FROM orders
WHERE user_id = 1 AND created_at >= '2024-01-01';

-- 查看执行计划（MySQL 8.0+ 也可用 EXPLAIN ANALYZE 看实际耗时）
EXPLAIN FORMAT=TREE SELECT * FROM orders WHERE user_id = 1;
```

## 常见误区

- 把「建了索引」等同于「查询一定快」：写入维护索引、选择性差、回表次数多都会抵消收益。
- 认为 `WHERE a=1 OR b=2` 一定会走索引：两列若无合适合并策略，可能全表扫描。
- 忽略 **统计信息与直方图** 过期导致优化器选错计划；过度依赖单次 `EXPLAIN` 而不结合数据量与数据分布。
- 联合索引列顺序随意：应把 **高选择性、常做等值条件** 的列靠前，兼顾排序与分组需求。
