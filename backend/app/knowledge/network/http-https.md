# HTTP 与 HTTPS

- **难度**: 进阶
- **分类**: 计算机网络 / 应用层
- **标签**: [HTTP, HTTPS, TLS, HTTP2, HTTP3, Keep-Alive, 队头阻塞]

## 核心概念

**HTTP 就像寄明信片**——内容写在外面，谁都能看到（明文传输），简单直接。**HTTPS 就像寄加密信**——信放在锁着的盒子里，只有收件人有钥匙能打开。HTTPS = HTTP + TLS，不仅加密内容，还能验证"这封信真的是那个人寄的"（身份认证）和"路上没被人偷改过"（完整性）。

HTTP 的版本升级，核心都在解决一个问题：**怎么让一根网线上同时传更多东西、更快**。

## 详细解析

### HTTP 版本演进——一句话总结

- **HTTP/1.0**：每次请求都要重新建连接，像每次寄信都要重新去邮局排队
- **HTTP/1.1**：引入 **Keep-Alive**，一个连接可以连续发多封信，不用每次排队。但信必须按顺序处理——前一封卡住了后面都得等（**队头阻塞**）
- **HTTP/2**：一个连接上可以同时发多封信且互不干扰（**多路复用**），还压缩了信封（HPACK 头部压缩）。但底层用的 TCP 丢一个包整条路都会卡
- **HTTP/3**：换了条新路——基于 **QUIC**（UDP 上实现的可靠传输），每封信走自己的"小车道"，一封信丢了不影响其他信

### HTTPS 握手——怎么交换那把"锁的钥匙"？

TCP 连接建好后，开始 TLS 握手（以 TLS 1.2 为例）：
1. 客户端："我支持这些加密算法，这是我的随机数"
2. 服务端："用这个算法吧，这是我的证书和随机数"
3. 客户端验证证书是否合法（是不是权威 CA 签发的、域名对不对、过没过期），然后生成一个"预主密钥"
4. 双方用三个随机数算出会话密钥，之后所有数据都用这个密钥加密

**TLS 1.3** 把这个过程从 2 个往返缩短到了 1 个，甚至能做到 0-RTT 恢复。

### 常见状态码——记住这几组就够了

- **2xx**：成功（200 OK、201 Created、204 No Content）
- **3xx**：重定向（301 永久搬家、302 临时搬家）
- **4xx**：你的问题（400 请求有误、401 没登录、403 没权限、404 找不到）
- **5xx**：服务器的问题（500 内部错误、502 网关收到垃圾回复、503 服务器忙、504 网关超时）

## 示例代码

```bash
# 用 openssl 查看网站的证书信息
openssl s_client -connect example.com:443 -servername example.com </dev/null 2>/dev/null \
  | openssl x509 -noout -subject -dates
```

```python
import urllib.request
# 一行代码发 HTTPS 请求，TLS 握手在底层自动完成
req = urllib.request.Request("https://example.com")
with urllib.request.urlopen(req, timeout=10) as r:
    print(r.status, r.getheader("Content-Type"))
```

## 面试追问

- **面试官可能会这样问你**：TLS 1.3 比 1.2 少了几个往返？0-RTT 有什么安全风险？（少了一个 RTT；0-RTT 数据可能被重放攻击）
- **面试官可能会这样问你**：HTTP/2 既然多路复用了，为什么还有队头阻塞？（TCP 层面丢一个包，整条连接都要等重传，所以 HTTP/3 换成了 QUIC）
- **面试官可能会这样问你**：301 和 302 有什么区别？浏览器行为有什么不同？（301 永久重定向，浏览器会缓存；302 临时的，每次都要问服务器）
- **面试官可能会这样问你**：把 HTTPS 理解成"加密就安全了"有什么问题？（如果不验证证书，中间人可以伪造证书拦截——所以 HTTPS 的三大功能是加密、完整性、身份认证）

## 常见误区

- **很多人会搞混的地方**：把 HTTPS 等同于"只是加密了"——忽略了完整性校验和身份认证，不验证证书照样能被中间人攻击。
- **很多人会搞混的地方**：以为 HTTP/2 一定比 HTTP/1.1 快——小网站、配置不当时可能没什么优势。
- **很多人会搞混的地方**：混淆 302 和 307——302 在实践中浏览器可能把 POST 改成 GET，307 严格保留原方法。
- **很多人会搞混的地方**：以为 HTTP/3 不需要可靠传输——QUIC 在 UDP 之上实现了自己的可靠性和拥塞控制，只是不用 TCP 了。
