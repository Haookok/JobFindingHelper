# 缓存架构设计

- **难度**: 进阶
- **分类**: 系统设计 / 缓存
- **标签**: [缓存穿透, 缓存击穿, 缓存雪崩, CacheAside, 热点Key, 一致性]

## 核心概念

缓存的思路特别朴素：**把常翻的书放在桌上，而不是每次都跑去图书馆借**。数据库是图书馆（靠谱但慢），缓存是你桌上（快但容量有限）。访问量大的时候，让大部分请求在"桌上"就能找到答案，图书馆就不会被挤爆。

但"桌上放书"这件事有三个经典的坑：**穿透**（有人老问你一本根本不存在的书，你每次都得跑图书馆确认）、**击穿**（全班最热门的那本书到期了，所有人同时冲去图书馆借）、**雪崩**（桌上的书同一时间全到期了，全班集体冲向图书馆）。

## 详细解析

**缓存穿透——查不存在的数据**：恶意请求或参数错误，查的 key 在数据库里压根不存在，缓存永远不命中。解决办法：对不存在的 key 也缓存一个短期的"空值"（"这本书确实没有"）；或者用**布隆过滤器**先拦一道——它就像一个快速名单，不在名单上的直接拒绝。

**缓存击穿——热点 key 过期的那一瞬间**：明星同款商品的缓存过期了，几千个请求同时扑向数据库。解决办法：**加锁/单飞**（只让一个请求去数据库取，其他人等着）；**逻辑过期**（缓存不真的过期，后台异步更新）；热点 key 提前预热。

**缓存雪崩——大量 key 同时失效**：比如凌晨统一刷了缓存，早高峰一到全部过期。解决办法：给 TTL **加随机抖动**（每个 key 的过期时间错开）；设置多级缓存（本地 + Redis）；准备熔断降级。

**Cache Aside——最常用的更新策略**：读的时候先查缓存，没有就查数据库再回填缓存。写的时候**先更新数据库，再删除缓存**。为什么是"删"而不是"更新"缓存？因为删更简单，下次读的时候自然会从数据库拉最新的。

**热点 Key——一个 key 被万人追捧**：单个 key QPS 极高，Redis 某个节点扛不住。解决办法：在应用本地也加一层缓存；或者把一个 key 拆成多个副本（加随机后缀），分散到不同节点读。

**缓存和数据库的一致性**：没有完美方案。常用"TTL + 删除缓存"兜底，业务能容忍短暂不一致就行。更严格的场景可以用 Canal 监听数据库 binlog 来主动清缓存。

import random

def cache_ttl_jitter(base_sec: int, spread: float = 0.2) -> int:
    """防雪崩：给基础 TTL 加一个随机偏移，让 key 不要同时过期"""
    return int(base_sec * (1 + random.uniform(-spread, spread)))

## 面试追问

- **面试官可能会这样问你**："先删缓存再写数据库"和"先写数据库再删缓存"各有什么并发问题？哪种更常用？
- **面试官可能会这样问你**：Redis 集群里热点 key 落在同一个 slot 怎么办？
- **面试官可能会这样问你**：多级缓存（本地 + Redis）的一致性怎么保证？本地缓存的 TTL 该设多短？
- **面试官可能会这样问你**：大 Key 对 Redis 集群迁移和持久化有什么影响？怎么拆？

## 常见误区

- **很多人会搞混的地方**：用布隆过滤器解决"击穿"——布隆过滤器主要防"穿透"（拦截不存在的 key），击穿要用加锁或逻辑过期。
- **很多人会搞混的地方**：空值缓存 TTL 设太长——本来不存在的数据后来写入了，结果缓存里还是"空"，造成长期脏读。
- **很多人会搞混的地方**：以为 Cache Aside 删缓存就能保证强一致——并发场景下还是有短暂不一致的窗口，业务要能容忍。
- **很多人会搞混的地方**：把本地缓存（Caffeine/Guava）当成分布式一致的——多个服务实例之间本地缓存是各自为政的，只适合能容忍短暂不一致的读多写少场景。
- **很多人会搞混的地方**：互斥锁实现不当导致死锁——一定要设短超时，否则抢到锁的进程挂了，其他人全卡住。
