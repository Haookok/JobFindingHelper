# 缓存架构设计

- **难度**: 进阶
- **分类**: 系统设计 / 缓存
- **标签**: [缓存穿透, 缓存击穿, 缓存雪崩, CacheAside, 热点Key, 一致性]

## 核心概念

缓存用于**降低延迟、减压数据源**。设计需同时考虑**正确性**（与 DB 一致）、**可用性**（故障与热点）与**成本**。典型问题：**穿透**（查不存在的数据）、**击穿**（热点键过期瞬间压垮 DB）、**雪崩**（大量键同时失效）。

**命中率与成本**：Key 设计要兼顾**粒度**（过细放大键空间与元数据，过粗放大失效粒度）；**预热**与**懒加载**策略取决于读模式与启动风暴。

## 详细解析

**缓存穿透**：请求的数据**根本不存在**，缓存永不命中，流量直达 DB。  
**方案**：对不存在 key 也缓存**短 TTL 空值**（注意布隆误判）；**布隆过滤器**前置拦截；API 层校验与限流。

**缓存击穿**：**单个热点 key** 过期瞬间，大量并发同时未命中 DB。  
**方案**：**互斥锁/单飞**（只允许一个线程回源）；**逻辑过期**（异步刷新）；**永不过期 + 后台更新**；热点 key **预热**。

**缓存雪崩**：**大量 key 同一时刻过期**或缓存集群宕机，请求洪峰打挂 DB。  
**方案**：TTL **加随机抖动**；**多级缓存**（本地 + 远程）；**熔断降级**；**主从/集群**高可用；必要时**限流排队**。

**缓存更新策略**：

- **Cache Aside（旁路）**：读先缓存后 DB 回写；写**先更 DB 再删缓存**（或先删再写需注意并发窗口）。实现简单，最常用。
- **Read Through**：应用只调缓存组件，未命中由组件**代查 DB 并回填**。
- **Write Behind（异步写）**：先写缓存，**异步批量刷盘**，吞吐高但**丢数据风险**与一致窗口大，需 WAL/队列保证。

**分布式缓存一致性**：无完美方案；常用 **TTL + 删除缓存**、**版本号/时间戳**、**Canal/Debezium 订阅 binlog** 失效缓存、**强一致**场景走 DB 或共识，缓存仅作加速。

**热点 Key**：单 key QPS 极高导致单分片/单节点瓶颈。  
**方案**：**本地缓存**多级；**拆分 key**（副本 key + 随机后缀读聚合）；**Redis 热 key 探测与迁移**；读写分离；必要时在网关做**聚合请求**。

**Write Through**：写操作**同时写缓存与 DB**（或由缓存组件代理），读路径简单，但写延迟与失败处理要设计好（双写失败顺序、重试）。

**内存与淘汰**：Redis 常用 **LRU/LFU/TTL** 等策略；要区分**缓存**与**持久化存储**职责，避免把缓存当唯一数据源却无回源能力。

**监控指标**：命中率、大 key、热 key、慢查询、连接数、内存碎片率；与 DB **QPS 突增**联动告警常能提前发现击穿/穿透。

    """雪崩治理：在基础 TTL 上增加随机抖动。"""
    return int(base_sec * (1 + random.uniform(-spread, spread)))

## 常见误区

- 用**布隆过滤器**解决击穿——布隆主要防**穿透**；击穿要配合**互斥**或**逻辑过期**。
- **空值缓存** TTL 过长导致**长期脏读**真实已写入的数据——需**主动失效**或较短 TTL。
- 认为 **Cache Aside 删缓存** 能保证强一致——并发下仍有短暂不一致窗口，需业务容忍或版本控制。
- **互斥锁**实现不当导致**死锁**或**锁等待放大**——需**短超时**、**看门狗**或**单飞线程池**等配套。
- 把 **本地缓存（Caffeine/Guava）** 当成分布式一致源——多实例间**无协调**，只适合**可容忍短暂不一致**的读多写少场景。

**压测建议**：模拟热点键过期、批量 TTL 到期与缓存宕机，观察 DB **连接池**与 **p99** 是否突变。
