# 分布式一致性

- **难度**: 深入
- **分类**: 系统设计 / 分布式
- **标签**: [CAP, BASE, Paxos, Raft, 2PC, 3PC, 最终一致性]

## 核心概念

想象你和三个朋友**同时编辑同一份在线文档**。你在北京改了标题，朋友在上海改了正文，网断了五秒——这五秒里大家看到的文档不一样，这就是"不一致"。**分布式一致性**要解决的就是：多台机器存了同一份数据的副本，怎么让大家对"数据现在长什么样"达成共识。

**强一致**就像所有人实时看到同一个屏幕，体验最好但代价最高（要不停互相确认）。**最终一致**就像微信群消息有人晚收到几秒，但最终都能收齐。工程上要根据业务选择"多强的一致性才够用"。

## 详细解析

**CAP 定理——三选二的无奈**：网络分区（P）是必然会发生的（网线会断、机房会抖），所以在出问题时你只能在"一致性 C"和"可用性 A"之间选一个。打比方：快递站系统断网了，是选择"谁都查不了快递"（保一致性，拒绝服务）还是"能查但可能显示旧状态"（保可用性，容忍不一致）？

**BASE——退一步海阔天空**：很多互联网业务选择"基本可用 + 软状态 + 最终一致"。你下单后余额可能晚几百毫秒才扣，但最终一定会扣对——这就够了。跟银行转账那种"必须立刻一致"的 ACID 相比，BASE 换来了更高的可用性。

**Raft——选个班长来记笔记**：一群服务器选出一个 Leader（班长），所有写操作都交给 Leader，Leader 把"操作日志"复制给大多数 Follower 后才算成功。Leader 挂了，大家重新选举。Raft 比 Paxos 好懂得多，etcd、部分数据库都用它。

**Paxos——学术界的经典**：和 Raft 解决同一个问题（大家对一个值达成共识），但角色划分不同，教科书里必提。面试能说清"跟 Raft 思想相近但实现更复杂"就够了。

**两阶段提交（2PC）——先举手再干活**：协调者问所有参与者"你们准备好了吗？"全说 OK 就提交，有一个说不行就回滚。问题是：如果协调者问完就挂了，参与者就卡在那里不知道该提交还是回滚——这就是"阻塞问题"。

**最终一致性的工程手段**：消息队列 + 幂等处理、定期对账、版本号冲突解决、Outbox 模式（业务写 DB 的同时写一张"发件箱"表，后台异步发消息，避免"DB 成功但消息丢了"）。

**Saga 和 TCC——长流程的拆分**：微服务里跨多个服务的"大事务"很难用 2PC，改成一连串小事务 + 补偿操作。比如订机票和订酒店，酒店订失败了就取消机票——这就是 Saga 的补偿思路。

**时钟问题**：分布式系统里不能信任"墙上的钟"（机器时间会漂移），所以用逻辑时钟（Lamport Clock）来判断事件的先后因果。面试提一句"墙上时钟不可靠"就能加分。

## 示例代码

```python
# 2PC 简化伪代码：先问所有人准备好没，再统一提交或回滚
def two_pc(participants):
    # 第一阶段：所有参与者表态
    if not all(p.prepare() for p in participants):
        for p in participants:
            p.abort()       # 有人说不行，全体回滚
        return False
    # 第二阶段：所有人都 OK，统一提交
    for p in participants:
        p.commit()
    return True
```

## 常见误区

- **很多人会搞混的地方**：背 CAP 说"选 CA 系统"——分区是必然存在的，讨论的是分区发生时选 C 还是 A，不存在"不分区"的选项。
- **很多人会搞混的地方**：以为 3PC 一定比 2PC 安全——在异步网络下 3PC 也不能保证所有情况的一致性，所以生产环境很少用。
- **很多人会搞混的地方**：把 Paxos 和 Raft 画等号——思想相近但角色、选主方式和实现难度不同。
- **很多人会搞混的地方**：把"最终一致"说成"不保证一致"——正确说法是"在有限时间内、没有新写入的情况下，所有副本会收敛到相同状态"。
- **很多人会搞混的地方**：混淆"线性化"和"可串行化"——前者是单个数据的实时顺序，后者是数据库事务的隔离级别，不是一回事。
