# 分布式一致性

- **难度**: 深入
- **分类**: 系统设计 / 分布式
- **标签**: [CAP, BASE, Paxos, Raft, 2PC, 3PC, 最终一致性]

## 核心概念

分布式系统中多副本、网络分区与节点故障并存，**一致性**指多个副本对外呈现的数据状态是否满足约定语义。**强一致**代价高；工程上常在 **CAP/BASE** 框架下选择 **线性化**、**顺序一致**或**最终一致**等不同等级，并用 **共识算法** 或 **事务协议** 实现。

**副本与 quorum**：读写副本数的组合（如 **N、W、R**）决定**可调一致**：`W+R>N` 常可避免读两套旧值，但与**冲突合并**、**提示移交（hinted handoff）** 等机制需一起理解。

## 详细解析

**CAP 定理**：在**分区容错 P**（网络分区不可避免）前提下，**一致性 C** 与**可用性 A** 不能同时满格。实际解读应避免教条：分区是**时间片段内**的事件；系统可**分区时降级**（牺牲部分 A 或放松 C），恢复后再修复。

**BASE**：**Basically Available**（基本可用）、**Soft state**（软状态）、**Eventually consistent**（最终一致）。与 ACID 对照：强调**高可用 + 异步复制 + 可容忍短暂不一致**，通过消息补偿、对账等达到最终一致。

**Paxos**：解决**异步网络**下多节点对单一值（或日志）达成一致的**共识**问题。角色：Proposer、Acceptor、Learner；两阶段（Prepare/Promise 与 Accept）保证**安全性**（不会决议不同值）与**活性**（在多数派可达时可推进）。Multi-Paxos 将首轮 Prepare 优化为选主后只跑 Accept，近似**日志复制**。

**Raft**：更易教学的共识算法，强Leader；**选举**（任期、心跳）+ **日志复制**（匹配索引与任期）；通过**提交索引**定义已提交条目。与 Paxos **等价表达能力**，工程实现（etcd、部分 MQ/DB）更常见。

**两阶段提交（2PC）**：协调者 **Prepare**（参与者写日志、锁资源、投票）→ 全 Yes 则 **Commit**，任一 No 则 **Abort**。缺点：**同步阻塞**、协调者单点、**脑裂**时可能不一致；网络分区下参与者可能长期阻塞。

**三阶段提交（3PC）**：增加 **PreCommit**，减少部分阻塞窗口，但**无法根本解决**网络分区下的不一致问题，实现复杂，生产少用。

**最终一致性方案**：异步主从复制 + **版本向量/时间戳**冲突解决；**CQRS + 事件溯源**；**业务幂等与对账**；**Gossip** 传播；读时**多副本读修复（read repair）**。根据业务选择「读写何种副本」「是否同步副本数 W+R>N」等（类似 Dynamo 可调一致）。

**ZAB / Multi-Paxos 工程形态**：ZooKeeper 的 **ZAB** 与 Raft 同属**主从式日志复制**家族，强调**全序广播**；与「经典论文 Paxos」表述不同但目标一致：**已提交的提案不会被覆盖**。

**Saga 与 TCC**：长事务在微服务中常用 **Saga**（编排/协同 + 补偿）或 **TCC**（Try-Confirm-Cancel）替代全局 2PC，牺牲部分隔离性换**可用性与扩展性**。

**Outbox 模式**：业务写 DB 与发消息**同事务**写 outbox 表，异步进程读 outbox 投递 MQ，避免「DB 成功、消息丢失」或双写不一致，是**最终一致 + 可靠投递**的常见落地。

**线性化 vs 串行化**：**线性化**强调单寄存器式**实时顺序**；**可串行化**是事务隔离级别概念，二者相关但不等同，勿混为一谈。

**时钟问题**：**逻辑时钟（Lamport）**、**向量时钟**用于**因果**推断；**TrueTime** 用**时钟不确定性界**换全局时间序——面试提到「墙上时钟不可靠」即可建立好感。

**拜占庭容错**：Paxos/Raft 默认**崩溃容错**（非恶意）；**PBFT** 等才针对拜占庭节点，复杂度高，链与部分金融场景才会深入考。

**幂等键与去重**：在「至少一次投递」前提下，**业务层幂等**（订单号、去重表）往往是比强求分布式事务更经济的工程解。

**Lease 与 fencing token**：主从切换常用 **lease** 防**双主写入**；存储层用 **fencing** 拒绝过期主的写，可与共识选主组合使用。

```python
# 2PC 语义的简化伪代码：展示协调者与参与者的两阶段
def two_pc(participants):
    if not all(p.prepare() for p in participants):
        for p in participants:
            p.abort()
        return False
    for p in participants:
        p.commit()
    return True
```

## 常见误区

- 背 CAP 选「CA」系统——分区存在时**必须处理 P**，讨论常是**分区期间**选 C 还是 A。
- 认为 **3PC** 比 **2PC** 一定安全——无法证明在异步模型下解决所有不一致。
- 把 **Paxos == Raft** 画等号：思想相近但角色划分、选主与教学结构不同，优化路径也不同。
- 认为 **共识算法** 能解决一切：**成员变更**（如 Raft joint consensus）、**快照与日志压缩**、**时钟漂移** 仍是工程大坑。
- 把 **最终一致** 说成「不保证一致」——应表述为「在有限时间与无新写条件下收敛到相同状态」，并说明**业务可见窗口**。

**对比记忆**：2PC 偏**数据库跨库事务**；Paxos/Raft 偏**复制日志与元数据**；Saga/TCC 偏**长流程微服务**。
