# 消息队列

- **难度**: 进阶
- **分类**: 系统设计 / 消息中间件
- **标签**: [Kafka, RabbitMQ, RocketMQ, 消息丢失, 重复消费, 顺序消息, 积压]

## 核心概念

消息队列就像**快递驿站**：你（生产者）寄了一个包裹放到驿站（Broker），收件人（消费者）什么时候方便什么时候去取。你不用等收件人在家，收件人也不用盯着你什么时候发货——这就是**解耦**和**异步**。

更重要的是**削峰填谷**：双十一零点涌来百万订单，如果直接打到后端数据库，立马崩盘。但如果先把订单消息扔进驿站，后端按自己的处理能力慢慢取，就像驿站帮你缓冲了快递洪峰。

代价是什么？引入了中间层就多了**延迟**（不是实时送达）、**运维复杂度**（驿站也会故障）以及**一致性难题**（包裹丢了怎么办？送重了怎么办？）。

## 详细解析

**三大主流 MQ 对比（面试简版）**：

| | Kafka | RabbitMQ | RocketMQ |
|---|---|---|---|
| 擅长 | 大数据日志流、高吞吐 | 灵活路由、低延迟任务 | 阿里系场景、事务消息 |
| 模型 | Topic + 分区，日志追加 | Exchange + Queue，AMQP 协议 | 借鉴 Kafka，加了队列级有序 |
| 消费 | 消费者组内分区独占 | 多种模式（工作队列、广播等） | 集群消费/广播消费 |

**消息丢失——三个环节都可能丢**：生产者发送时网络抖了（→ 开启发送确认）；Broker 还没落盘就宕机了（→ 开启持久化和副本同步）；消费者拿到消息还没处理就确认了（→ 先处理业务再提交 offset）。

**重复消费——比丢消息更常见**：网络重试、消费者重平衡都可能导致同一条消息被处理两次。解决核心是**业务幂等**——同一个操作执行多次效果跟执行一次一样。比如用订单号做唯一键，重复写入直接忽略。

**顺序消费——没你想的那么简单**：全局有序代价极高。通常的做法是把同一个业务 ID（比如同一个用户的操作）哈希到同一个分区，保证**分区内有序**就够了。

**消息积压——驿站包裹堆满了**：消费者处理太慢或者挂了，消息越堆越多。应急手段：扩容消费者、降级非核心逻辑、临时转发到备用队列用更多实例消费。但根因通常是下游慢或消费者有 bug。

**精确一次（Exactly Once）——理想很丰满**：端到端的精确一次在分布式场景下非常难做到。Kafka 的事务只保证"Kafka 系统内"的语义，跨数据库和 MQ 还是需要幂等 + 去重 + Outbox 模式来兜底。

**延迟队列和死信队列**：想让消息"30 分钟后再处理"？用延迟队列。消息消费失败重试多次还不行？扔进死信队列（DLQ），避免一条"毒消息"卡住整个队列。

## 面试追问

- **面试官可能会这样问你**：Kafka 的 ISR、HW、LEO 是什么意思？它们怎么决定消费者能看到哪些消息？
- **面试官可能会这样问你**：RabbitMQ 镜像队列和 Kafka 副本机制有什么不同？
- **面试官可能会这样问你**：RocketMQ 的事务消息是怎么通过半消息和回查来保证本地事务和发送一致的？
- **面试官可能会这样问你**：消息积压了怎么快速处理？能不能无限加消费者？

## 常见误区

- **很多人会搞混的地方**：追求"全局精确一次"但不谈端到端——跨数据库和 MQ 的精确一次需要 Outbox 等模式，光靠 Kafka 事务不够。
- **很多人会搞混的地方**："先提交 offset 再处理"说成"至少一次"——这其实是"至多一次"（可能丢消息），因为提交完消费者挂了，消息就没处理。
- **很多人会搞混的地方**：以为加分区就能无限扩容消费者——Kafka 里消费者数不能超过分区数，超了就有消费者闲着。
- **很多人会搞混的地方**：消息体塞巨大 JSON——应该只传引用（比如对象存储的 ID），大文件放 OSS，MQ 只传"通知"。
