# 进程与线程

- **难度**: 基础
- **分类**: 操作系统
- **标签**: [进程, 线程, 协程, IPC, 死锁]

## 核心概念

把电脑想象成一个工业园区。**进程就是一个个独立的工厂**——每个工厂有自己的厂房（内存空间）、设备（文件资源）、工人。工厂之间互相隔离，一个工厂着火了不会影响隔壁。

**线程就是工厂里的工人**——同一个工厂里的工人共享厂房和设备，沟通方便（共享内存），但得协调好谁用哪台机器，不然打架（竞争条件）。

**协程就是工人的"分身术"**——一个工人可以同时处理好几件事，做 A 等材料的时候切去做 B，材料到了再切回来。成本极低，但只是"假并行"（一个人干多件事，不是真的多个人）。

## 详细解析

### 进程 vs 线程——选哪个？

- **隔离性**：进程之间互不影响（一个崩了不影响另一个），线程共享地址空间，一个线程挂了整个进程都得完蛋
- **通信**：进程之间要"写信"（IPC），线程之间直接"喊话"（共享内存，但要加锁）
- **创建成本**：开一个新工厂（进程）比招一个新工人（线程）贵得多

### 进程之间怎么通信（IPC）？

就像两个工厂之间传消息，有这几种方式：
- **管道（Pipe）**：像传话筒，单向传，只能在有亲缘关系的进程间用
- **消息队列**：像寄快递，发一条一条的消息，内核帮你管理
- **共享内存**：两个工厂共用一块仓库，速度最快，但得加锁协调
- **信号（Signal）**：像拍肩膀提醒一下，只能传简单的通知
- **Socket**：最通用，甚至可以跨机器通信

### 线程之间怎么协调？

同一个工厂里多个工人共用设备，得有规矩：
- **互斥锁（Mutex）**：一次只能一个人用这台机器
- **读写锁**：读的人可以同时来，写的人必须独占
- **信号量**：最多允许 N 个人同时进入这个区域

### 死锁——谁也别想动

死锁就像两个人面对面过独木桥，谁也不肯退让。发生死锁必须同时满足四个条件：互斥、占有且等待、不可抢占、循环等待。**破解方法**：打破任一条件，比如"按固定顺序加锁"、"加锁超时主动放弃"。

## 示例代码

```c
// 用互斥锁 + 条件变量协调"生产者-消费者"
pthread_mutex_t mu = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t cv = PTHREAD_COND_INITIALIZER;
int ready = 0;

void wait_ready(void) {
    pthread_mutex_lock(&mu);
    while (!ready)                        // 用 while 而不是 if，防止"虚假唤醒"
        pthread_cond_wait(&cv, &mu);      // 等的时候自动释放锁，被唤醒后重新加锁
    pthread_mutex_unlock(&mu);
}

void signal_ready(void) {
    pthread_mutex_lock(&mu);
    ready = 1;
    pthread_cond_broadcast(&cv);          // 叫醒所有等着的人
    pthread_mutex_unlock(&mu);
}
```

## 面试追问

- **面试官可能会这样问你**：`pthread_cond_wait` 为什么必须在 while 循环里？（因为有"虚假唤醒"，被叫醒不代表条件真的满足了）
- **面试官可能会这样问你**：协程和线程的本质区别是什么？Go 的 GMP 模型解决了什么问题？（协程在用户态切换，比内核调度轻得多；GMP 让协程能利用多核）
- **面试官可能会这样问你**：共享内存和消息队列怎么选？（共享内存快但要自己加锁，消息队列用起来简单但有拷贝开销）
- **面试官可能会这样问你**：多线程一定比多进程快吗？（不一定！锁竞争激烈时线程反而更慢，进程隔离性更好也更安全）

## 常见误区

- **很多人会搞混的地方**：以为"多线程一定比多进程快"——竞争锁、伪共享等开销可能让线程方案更慢。
- **很多人会搞混的地方**：把协程当成"免费并行"——协程擅长 I/O 等待，CPU 密集任务还是得靠多线程/多进程。
- **很多人会搞混的地方**：死锁只背四个条件，面试官追问"工程上怎么防"就答不上来了。要记住"按序加锁、超时放弃、try-lock"这几招。
- **很多人会搞混的地方**：忽略僵尸进程——子进程退出但父进程没回收，会一直占着进程表项。
