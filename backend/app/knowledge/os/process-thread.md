# 进程与线程

- **难度**: 基础
- **分类**: 操作系统
- **标签**: [进程, 线程, 协程, IPC, 死锁]

## 核心概念

**进程** 是资源分配的基本单位，拥有独立虚拟地址空间、文件描述符表等；**线程** 是 CPU 调度的基本单位，共享所属进程的地址空间与大部分资源，自带栈、寄存器、程序计数器等执行上下文。

**协程** 是用户态调度的轻量执行单元，由运行时或程序员协作让出执行权，切换成本低，适合高并发 I/O，但多核利用需结合线程池等模型。

**PCB（进程控制块）** 是内核描述进程的数据结构，保存 PID、状态、寄存器现场、内存指针、打开文件、调度信息等，是进程存在与切换的依据。

## 详细解析

### 进程 vs 线程

- **创建与切换**：进程更重（资源多）；线程共享地址空间，创建与上下文切换通常更轻（仍涉及内核态调度）。
- **通信**：进程间需 IPC；线程间可直接读写共享内存，但要同步。
- **健壮性**：一线程崩溃可能影响整进程；进程隔离性更好。

### 进程间通信（IPC）

- **管道（pipe）**：单向字节流，亲缘进程常用；命名管道（FIFO）可有路径名。
- **消息队列**：结构化消息，内核维护队列，适合离散消息。
- **共享内存**：最快路径之一，数据在用户空间映射同一块物理区域，需 **信号量/锁** 等协调生产者与消费者。
- **信号（signal）**：异步通知，语义简单，适合事件通知而非大块数据传输。
- **套接字**：可跨主机；本地也可用 Unix domain socket。

（不同 OS 还有 mmap、eventfd 等扩展，面试答清分类与典型场景即可。）

### 线程同步

- **互斥锁（mutex）**：临界区互斥。
- **读写锁**：读多写少时提高并发度。
- **条件变量**：等待某条件成立，需与互斥锁配合避免丢失唤醒。
- **信号量**：计数资源，可用于限流与同步。
- **自旋锁**：短期等待不休眠，适合极短临界区且在多核上避免频繁切换（误用会浪费 CPU）。

### 死锁的四个必要条件

1. **互斥**：资源独占使用。  
2. **占有且等待**：持有资源同时等待其他资源。  
3. **不可抢占**：已分配资源只能由持有者释放。  
4. **循环等待**：存在进程/线程资源的等待环。

**预防/缓解**：破坏任一条件，例如 **按序加锁**、**超时放弃**、**银行家算法**（经典教材）、减少嵌套锁、使用 **try-lock**、资源分级分配等。

## 示例代码

```c
// POSIX 线程互斥 + 条件变量（示意）
pthread_mutex_t mu = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t cv = PTHREAD_COND_INITIALIZER;
int ready = 0;

void wait_ready(void) {
    pthread_mutex_lock(&mu);
    while (!ready)
        pthread_cond_wait(&cv, &mu);  // 原子释放锁并等待
    pthread_mutex_unlock(&mu);
}

void signal_ready(void) {
    pthread_mutex_lock(&mu);
    ready = 1;
    pthread_cond_broadcast(&cv);
    pthread_mutex_unlock(&mu);
}
```

## 面试追问

- **追问 1**：为什么 `pthread_cond_wait` 必须在循环里检查条件？与虚假唤醒、状态变更有什么关系？
- **追问 2**：协程切换发生在用户态，与线程的「内核栈/用户栈」有何区别？Go 的 G-M-P 模型解决了什么问题？
- **追问 3**：共享内存 + 信号量 与 消息队列 在延迟、拷贝次数、适用场景上如何对比？

## 常见误区

- 认为「多线程一定比多进程快」：竞争锁、伪共享、调度开销可能抵消收益。
- 把 **协程** 当成免费并行：CPU 密集任务仍需多线程/多进程占满核心。
- 死锁只背四条件不举 **按序加锁、避免持有多锁** 等工程手段。
- 忽略 **文件描述符、SIGPIPE、僵尸进程** 等与进程生命周期相关的常见坑（与 IPC/子进程创建相关题）。
