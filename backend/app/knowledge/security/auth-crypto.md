# 认证授权与加密

- **难度**: 进阶
- **分类**: 安全 / 身份与密码学
- **标签**: [Session, JWT, OAuth2, TLS, 对称加密, 非对称加密, 哈希, RBAC, 零信任]

## 核心概念

**认证（Authentication）** 回答「你是谁」，**授权（Authorization）** 回答「你能做什么」。**Session**：服务端存会话状态，客户端常持 **session id**（Cookie）；易撤销、服务端有状态。**JWT**：自包含声明，服务端用密钥校验签名；无状态扩展好，**吊销与刷新**需额外设计（黑名单、短 TTL + refresh、版本号）。

**OAuth 2.0** 是 **授权框架**（委托访问），不是认证协议；OpenID Connect 在 OAuth2 上提供身份层。**HTTPS/TLS** 在传输层提供机密性、完整性与身份验证：**握手**交换密码套件，证书链校验服务端身份，协商对称会话密钥，记录层用 AEAD 等保护应用数据。

**对称加密**（AES-GCM 等）加解密同一密钥，快，适合大量数据；**非对称**（RSA、ECC）公钥加密/私钥签名，解决密钥分发，通常用于握手与签名。**哈希**（SHA-256）用于完整性；**密码存储**用 **bcrypt/Argon2/scrypt** 等慢哈希 + 盐，**禁止明文或可逆**。**MD5/SHA-1** 已不适合安全场景。

**RBAC** 按角色绑定权限；**ABAC** 按属性（用户、资源、环境）策略决策，灵活但复杂。**零信任** 默认不信任网络位置，持续验证身份与设备态势，最小权限与微分段。

## 详细解析

**Session vs JWT**：Session 适合强服务端控制与即时登出；JWT 适合跨服务与移动端，但要注意 **XSS 存 localStorage 窃 token**、**体积过大**、**算法 none/HS256 误用**（公钥当 HMAC secret）等坑。**Refresh Token** Rotation 可降低长期泄露风险。

**TLS 握手（1.2 简述）**：ClientHello（随机数、套件）→ ServerHello + 证书 + ServerKeyExchange → 客户端验链、生成预主密钥 → 双方派生会话密钥 → Finished 校验握手完整性。**TLS 1.3** 减少往返、仅安全套件、0-RTT 有重放权衡。

**对称 vs 非对称**：大数据用对称；非对称用于密钥封装或数字签名。**混合加密** 是常态。选用 **AEAD**（如 AES-GCM、ChaCha20-Poly1305）避免自行拼接 IV/MAC。

**哈希与密码**：SHA-256 用于文件完整性、证书指纹；**不应用 SHA 存密码**。bcrypt 工作因子可调对抗硬件加速；Argon2 获密码哈希竞赛推荐。** pepper**（服务端密钥）与 **per-user salt** 组合使用需注意密钥管理。

**RBAC/ABAC**：RBAC 模型清晰、审计简单；ABAC 适合细粒度动态策略（如「仅工作时间 + 部门匹配」）。Kubernetes、云 IAM 多为 RBAC + 资源级策略的变体。

**零信任实践**：身份（MFA、设备证书）、网络（微分段、服务网格 mTLS）、数据（加密与 DLP）、可观测与持续评估；与传统边界 VPN 对比强调 **身份为新的边界**。

**mTLS 与服务身份**：双向 TLS 在微服务间验证 **服务证书**（SPIFFE/SPIRE 等），结合 **短周期证书** 降低泄露影响。与「仅边缘 TLS」相比，内网横向移动难度提高。

**密钥管理**：**KMS/HSM** 保护主密钥；应用侧使用 **信封加密**（数据密钥本地加解密，KEK 在 KMS）。**轮换** 需兼容解密旧密文与多版本密钥 ID。

**JWE 与 JWS**：**JWS** 防篡改；敏感声明若必须自包含传输用 **JWE** 加密。**JWK** 轮换通过 `kid` 头选择验证密钥；防止 **alg 混淆**（RS256 token 被当作 HS256 用公钥验）。

**证书与 PKI**：链式校验至信任锚、检查 **hostname/SAN**、**OCSP/CRL** 吊销；**Let's Encrypt** 等自动化 ACME。客户端证书在企业 **B2B API** 中仍常见。

**合规与隐私**：PII 最小化、字段级加密、**KMS 审计日志**；跨境传输注意法规与 **DPA**。

**随机数与安全**：密钥、nonce、session id 须用 **CSPRNG**（如 `crypto/rand`），勿用时间戳或可预测种子。**Nonce 与 IV** 在 AEAD 中不可复用同一密钥下组合，否则机密性与完整性崩溃。

**数字签名与证书透明度**：业务层 **JWS/自定义 API 签名** 与 TLS 证书不同；重要公钥基础设施可结合 **CT log** 监控误签发。

**口令策略与 MFA**：长度优先于复杂轮换频率（NIST 倾向 **长口令 + 泄露检测**）；敏感操作用 **TOTP/WebAuthn** 等多因素，降低钓鱼与重放风险。

**API Key 管理**：Key 应 **可轮换、可吊销、最小权限**；勿提交到仓库；网关层做 **作用域与速率** 控制，与终端用户 JWT 分层设计。

**SAML/OIDC 联邦（简述）**：企业 SSO 常用 **SAML 断言** 或 **OIDC 授权码流**；注意 **断言签名、时钟偏移、重放**（`NotOnOrAfter`、一次性 `InResponseTo`）与 **元数据轮换**。

**侧信道（了解）**：比较密码或 MAC 时用 **常数时间** 算法，避免计时泄漏；高安全场景关注 **缓存侧信道** 与实现库选型。

## 常见误区

- 把 **OAuth2 当登录协议** 不明确 **resource owner** 与 **OpenID Connect** 边界。
- **JWT 不设 exp** 或 **密钥过弱**；或 **在 payload 存敏感明文**（仅 Base64 非加密）。
- 用 **MD5/SHA256 直接哈希密码**、**固定盐** 或 **短随机盐**。
- **TLS 终止在边缘后内网明文** 传输敏感数据；忽略 **证书链与 SNI** 配置。
- 零信任 **只买产品不改造身份与分段**，流于口号；或把 **VPN 下掉却无替代控制**。
- **自签名证书关闭校验**（`InsecureSkipVerify`）上线——中间人可读改流量；应使用可信 CA 与正确校验。
- **同一 RSA 密钥既加密又签名** 且无清晰协议边界——增加密码分析面；实践上多用专门协议与套件。
- **Refresh Token 无绑定**（设备/轮替检测）——被盗后可长期续期；应 **rotation + reuse 检测**。
