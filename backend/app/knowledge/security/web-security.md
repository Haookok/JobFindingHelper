# Web 安全基础

- **难度**: 进阶
- **分类**: 安全 / Web
- **标签**: [XSS, CSRF, SQL注入, SSRF, 文件上传, CSP, CORS, OWASP]

## 核心概念

Web 安全就像**给房子装锁和监控**：你不能假设所有来访的人都是善意的。核心原则三条：**输入不可信**（所有用户输入都可能有猫腻）、**最小权限**（只给必要的权限）、**纵深防御**（不能只靠一道锁，要层层设防）。

最常见的几种"入室方式"：**XSS** 是在你的网页里塞了坏人的脚本，就像有人在你家墙上贴了一张假公告，来访的客人信以为真；**CSRF** 是趁你还登着某个网站的时候，骗你点一个链接自动帮坏人操作，就像有人模仿你的签名去银行取钱；**SQL 注入**是在输入框里写 SQL 代码，让数据库把家底都翻出来。

## 详细解析

**XSS（跨站脚本）——往你的页面塞坏代码**：分三种类型。**反射型**：坏人把恶意代码放在 URL 里，你点了链接就中招，像一次性的陷阱。**存储型**：恶意代码存进了数据库（比如评论区），所有看到这条评论的人都中招，杀伤力最大。**DOM 型**：纯前端问题，用 innerHTML 之类的方法直接拼接了用户输入。

防御核心：**对输出做转义**（把 `<script>` 变成无害的文本）、Cookie 设 **HttpOnly**（JS 读不到）、用 **CSP** 策略限制页面能执行什么脚本。

**CSRF（跨站请求伪造）——借你的身份干坏事**：你登着银行网站，点了一个恶意链接，浏览器自动带上你的 Cookie 发了一个转账请求。防御：Cookie 设 **SameSite**（Lax 或 Strict）、关键操作带 **CSRF Token**、敏感操作要求输密码或验证码。

**SQL 注入——在输入框里写代码**：如果后端把用户输入直接拼进 SQL 语句，攻击者可以输入 `' OR 1=1 --` 之类的东西把数据全部捞出来。防御很简单：**用参数化查询/预编译**，永远不要拼接 SQL 字符串。

**SSRF（服务端请求伪造）——骗服务器帮你访问内网**：如果你的服务接受用户传一个 URL 然后去请求，攻击者可能让你请求内网地址或云服务器的元数据接口。防御：URL 白名单、禁止访问内网 IP 段、禁用危险协议。

**文件上传——上传一个"炸弹"**：如果只检查文件后缀不检查内容，攻击者可能上传一个伪装成图片的可执行脚本。防御：白名单检查文件类型和魔数、重命名文件、存到没有执行权限的地方。

**CORS 不是安全机制**：很多人误解 CORS。它只是浏览器的"跨域读限制"，放宽的是"别的网站能不能读你的响应"。攻击者用 curl 等工具不受 CORS 限制，所以**CORS 不能替代鉴权**。

**访问控制漏洞——最容易被忽略的大坑**：改个 URL 里的 ID 就能看到别人的数据（水平越权），普通用户直接调管理接口（垂直越权）。防御：服务端**每次请求**都校验当前用户有没有权限访问这个资源，别只靠前端隐藏按钮。

## 示例代码

```http
Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none'; base-uri 'self'
Set-Cookie: sid=...; HttpOnly; Secure; SameSite=Lax; Path=/
```

说明：CSP 限制只能加载自己域名的脚本；Cookie 设了 HttpOnly（JS 读不到）、Secure（只走 HTTPS）、SameSite=Lax（跨站不自动带）。

## 面试追问

- **面试官可能会这样问你**：存储型 XSS 和反射型 XSS 的利用链有什么不同？CSP 的 Nonce 机制怎么跟 SSR 配合？
- **面试官可能会这样问你**：CSRF Token 有几种实现方式？SPA 应用和传统表单分别怎么防？
- **面试官可能会这样问你**：预编译/参数化查询为什么能防 SQL 注入？ORM 就完全安全吗？
- **面试官可能会这样问你**：SSRF 有哪些绕过手法（DNS 重绑定、302 跳转、IPv6）？怎么防？
- **面试官可能会这样问你**：`Access-Control-Allow-Origin: *` 有什么风险？和 JSONP 比呢？
- **面试官可能会这样问你**：怎么测试水平越权和垂直越权？API 网关能单独搞定吗？

## 常见误区

- **很多人会搞混的地方**：只在前端做输入校验就觉得安全了——前端校验能被绕过，**服务端必须再验一次**。
- **很多人会搞混的地方**：以为关掉 CORS 就安全了——攻击者可以不用浏览器直接发请求，CORS 只约束浏览器行为。
- **很多人会搞混的地方**：用黑名单过滤 XSS（比如替换 `<script>`）——各种编码和变形可以绕过，应该用**上下文编码 + CSP**。
- **很多人会搞混的地方**：以为"用了 JWT 就不用防 CSRF"——JWT 放在 localStorage 虽然不受 CSRF 影响，但更容易被 XSS 偷走。
- **很多人会搞混的地方**：文件上传只检查后缀名——把 .php 文件改名成 .jpg 照样能上传，要检查文件内容（魔数）。
- **很多人会搞混的地方**：只靠前端路由隐藏管理页面——直接请求 API 照样能越权，授权必须在服务端做。
- **很多人会搞混的地方**：CSP 配了 `unsafe-inline` 和 `unsafe-eval`——这基本把 XSS 防御全抵消了。
